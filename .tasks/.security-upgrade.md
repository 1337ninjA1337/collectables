# Security upgrade — everything to implement

> Grounded in the actual code (file:line evidence given). Ordered by
> severity: **C0 critical → C1 high → C2 medium → C3 hardening**. Each item
> states the finding, the risk, the concrete fix, and how to verify it.
> Cross-references `.be-suggestions.md` (BE-n) where the security fix is the
> same work as a backend item — do them once.

---

## C0 — Critical (exploitable, fix before anything else)

### SEC-1 — Cloudinary API secret is shipped in the public bundle
- **Finding:** `lib/cloudinary.ts:52` reads
  `EXPO_PUBLIC_CLOUDINARY_API_SECRET` via `process.env.EXPO_PUBLIC_*`. Metro
  **inlines every `EXPO_PUBLIC_*` value into the JS bundle** (this is stated
  repeatedly in the codebase's own Sentry/analytics env notes). The secret is
  then used at `:58` to compute a SHA-1 `destroy` signature **client-side**.
- **Risk:** anyone who opens devtools / downloads the web bundle extracts the
  Cloudinary **account API secret** and can sign arbitrary Admin API calls —
  delete *every user's* images, rename, exfiltrate, or run up usage on the
  whole Cloudinary account. This is a full account-credential leak.
- **Fix:**
  1. **Rotate the Cloudinary API secret immediately** (it must be considered
     compromised — it has shipped in every deployed build).
  2. Remove `API_SECRET` from the client entirely. Move image *deletion*
     behind an Edge Function (`supabase/functions/delete-image`) that holds
     the secret in Supabase function secrets and verifies the caller's JWT +
     ownership of the asset before calling Cloudinary destroy. Mirror the
     existing, correctly-built `supabase/functions/delete-account` (it does
     `auth.getUser()` on the caller token before acting — use it as the
     template).
  3. Keep uploads as **unsigned preset** but lock the preset down
     (see SEC-5), or move uploads through the same function.
- **Verify:** a `__tests__/no-secrets-in-bundle.test.ts` that builds and
  greps `dist/_expo/static/js/**` for the secret value / `API_SECRET` usage;
  a CI gitleaks scan; assert `lib/cloudinary.ts` no longer references
  `EXPO_PUBLIC_CLOUDINARY_API_SECRET`.

### SEC-2 — RLS is the only real authorization boundary and it is unverified
- **Finding:** `lib/supabase.ts` exposes the Supabase URL + publishable/anon
  key (it must, to make REST calls from the web bundle). The code calls
  `/rest/v1/{profiles,collections,items,friend_requests,chat_messages,
  chat_reads,marketplace_listings}` directly. There are **no versioned
  `CREATE TABLE` / RLS migrations** for `profiles/collections/items/
  friend_requests` (see BE-1).
- **Risk:** the anon key is public by design — `curl https://<project>.supabase
  .co/rest/v1/profiles?select=*` with that key returns data unless RLS denies
  it. If any base table has RLS disabled or a permissive policy, it is a full
  read/write data breach (read every user's email/collections, write/delete
  arbitrary rows).
- **Fix:** = **BE-1, BE-2, BE-11, BE-12**. Ship the base-schema migration
  with `ENABLE ROW LEVEL SECURITY` + explicit per-table policies (owner RW;
  public-collection / friend-visibility `SELECT` via a `SECURITY DEFINER`
  `is_visible_to()`), then prove deny-by-default with the anon leak test.
- **Verify:** `supabase/tests/rls.sql` pgTAP run in CI: `SET ROLE anon;`
  every base table `SELECT`/`INSERT`/`UPDATE`/`DELETE` either errors or
  returns only intentionally-public rows. Generalise
  `__tests__/analytics-events-rls-verification.test.ts`.

### SEC-3 — Authorization decisions live in client JS, not the server
- **Finding (admin):** `lib/social-context.tsx:256` —
  `isAdmin = selfProfile?.username === "1337antoxa" || email === "1337.antoxa@gmail.com"`.
  Today `deleteProfile` (`:482`) only mutates **local state/AsyncStorage**
  (it hides profiles locally — *not* yet a server delete), so the *current*
  blast radius is local. The danger is the **pattern**: the moment any admin
  action is wired to the server (profile delete, moderation), this
  client-side string check becomes the only gate and is trivially bypassed by
  editing the bundle / React state.
- **Finding (premium):** `lib/premium-context.tsx:46,63` — entitlement is
  read/written to AsyncStorage. A user edits one stored value and unlocks all
  paid features.
- **Risk:** privilege escalation (admin) and paywall bypass (premium), both
  with zero server cost to the attacker.
- **Fix:**
  - Admin → server: a `profiles.is_admin boolean` (or `app_admins` table)
    enforced by RLS / an Edge Function; the client `isAdmin` becomes purely
    cosmetic (show/hide UI), never the authorization. = **BE-11**.
  - Premium → server: server-validated `subscriptions` table; an Edge
    Function is the source of truth; gate paid mutations server-side. =
    **BE-22**.
- **Verify:** integration test proving an admin/premium action with a forged
  client flag but a non-privileged JWT is rejected by the server.

## C1 — High

### SEC-4 — Runtime Supabase config can be swapped via localStorage
- **Finding:** `lib/supabase.ts` `RUNTIME_CONFIG_KEY =
  "collectables-supabase-runtime-config"` + `setRuntimeSupabaseConfig()`
  reads the Supabase URL/key from `window.localStorage` and uses it over the
  build-time config.
- **Risk:** any XSS or a malicious browser extension can point the app at an
  **attacker-controlled Supabase project**, silently harvesting every
  credential/OTP/session the user enters (full account takeover funnel).
  Also conflicts with `CLAUDE.md`'s "ALWAYS USE CREDENTIALS FROM GITHUB
  SECRETS" intent.
- **Fix:** gate the runtime-config path behind an explicit `__DEV__` /
  dev-menu check so it cannot be active in a production web build; or remove
  it and rely solely on build-time GitHub-Secret-injected env. If kept for
  QA, require it be set only through the existing dev-menu
  (`lib/dev-menu.ts`), never auto-read on prod boot. (Note: `CLAUDE.md`
  forbids adding a *UI badge/clear button* for this — a code-level dev gate
  does not violate that.)
- **Verify:** test that `parseRuntimeConfig()` returns `null` when not in a
  dev/QA context in a production build.

### SEC-5 — Unsigned Cloudinary upload preset is an open write endpoint
- **Finding:** `lib/cloudinary.ts:31-33` uploads with a public
  `upload_preset` and no auth.
- **Risk:** the preset + cloud name are in the bundle; anyone can upload
  arbitrary content (illegal media, malware-hosting, free CDN abuse) to your
  Cloudinary account, exhausting the free tier and creating liability.
- **Fix:** in the Cloudinary console restrict the unsigned preset
  (allowed formats = images only, max file size, `moderation`/`access mode`,
  folder lock, incoming transformation that strips non-images), **or** route
  uploads through the SEC-1 Edge Function (signed, JWT-checked, per-user
  folder). Decide alongside **BE-24** (Cloudinary vs Supabase Storage).
- **Verify:** attempt an oversized / non-image / unauthenticated upload and
  confirm rejection.

### SEC-6 — No rate limiting on data mutations
- **Finding:** the sliding-window limiter exists only for analytics/Sentry
  (`lib/analytics.ts`, `lib/sentry.ts`). Cloud writes via
  `lib/supabase-*.ts` have none.
- **Risk:** the public anon key + no app-side throttle means a script can
  hammer inserts/updates, exhausting the Supabase free tier (DB, egress) — a
  cheap DoS against your own backend and bill.
- **Fix:** extract `createSlidingWindowLimiter(max, windowMs)` into
  `lib/rate-limit.ts` (already a logged suggestion) and apply it to mutation
  paths; pair with Postgres-side limits / `pg_cron` abuse sweeps. = **BE-29**.
  Real abuse protection must also be server-side (RLS + function checks),
  client throttle is only first-line.
- **Verify:** unit test the limiter; integration test that a burst past the
  cap is dropped client-side.

### SEC-7 — OAuth callback token handling
- **Finding:** `app/auth/callback.tsx:24-25` parses `access_token` etc. from
  `window.location.hash`. It always `router.replace("/")` (no open-redirect —
  good), but the tokens sit in the URL/hash.
- **Risk:** tokens in `location.hash` can leak via browser history, referrer,
  or extensions if not cleared promptly.
- **Fix:** after extracting the session, `history.replaceState(null, "",
  url.pathname)` to scrub the hash from history before any navigation;
  confirm Supabase `detectSessionInUrl` is configured and the hash is wiped.
- **Verify:** test that the hash is cleared before `router.replace`.

### SEC-8 — `npm audit`: 7 known vulnerabilities (1 high, 6 moderate)
- **Finding:** observed during dependency install this session:
  "7 vulnerabilities (6 moderate, 1 high)".
- **Risk:** known-CVE dependencies in the build/runtime.
- **Fix:** run `npm audit`, triage each (especially the high), `npm audit
  fix` where non-breaking, document any accepted/unfixable in a
  `SECURITY.md`, and add `npm audit --audit-level=high` as a non-blocking →
  later blocking CI step. Re-check the Expo SDK upgrade path for transitive
  fixes.
- **Verify:** CI audit step; the high advisory resolved or explicitly waived.

## C2 — Medium

### SEC-9 — Service-role key blast radius in Edge Functions
- **Finding:** `supabase/functions/delete-account/index.ts:24` uses
  `SUPABASE_SERVICE_ROLE_KEY` (RLS-bypassing). It *does* verify the caller
  via `auth.getUser()` first (`:30`) — correct pattern. New functions
  (SEC-1, BE-20/21/22) must not regress this.
- **Fix:** a shared `assertCaller(req)` helper every function calls before
  any service-role operation; a startup assertion that the resolved key
  carries the `service_role` JWT claim (logged suggestion). Never log the
  key; never return it in responses/errors.
- **Verify:** test each function rejects a missing/invalid `Authorization`
  header with 401 before doing any work.

### SEC-10 — CORS is wide open on Edge Functions
- **Finding:** `delete-account/index.ts:5` `Access-Control-Allow-Headers:
  "authorization,..."` (and typically `Allow-Origin: *`).
- **Risk:** `*` lets any site invoke the function with a victim's token if it
  can obtain one; tighten to the known web origin(s).
- **Fix:** set `Access-Control-Allow-Origin` to the GitHub Pages origin (and
  app deep-link origin) instead of `*`; centralise CORS in one shared module.
- **Verify:** preflight from a disallowed origin is rejected.

### SEC-11 — Account deletion completeness (GDPR + orphans)
- **Finding:** `delete-account` predates the cloud tables; unclear it
  cascades to `collections/items/friend_requests/chat_*` and Cloudinary
  assets.
- **Fix:** FK `ON DELETE CASCADE` (= **BE-6, BE-25**) + delete the user's
  Cloudinary folder via the SEC-1 function. Add a data-export endpoint for
  portability (= **BE-26**).
- **Verify:** integration test: after delete, no row in any table references
  the uid; assets gone.

### SEC-12 — Input validation / coercion on cloud read paths
- **Finding:** read paths cast JSON (`as MarketplaceListing[]`, etc.);
  `normalizeListing`/`mergeCollectionsFromCloud` exist but aren't applied
  uniformly.
- **Risk:** a malformed/hostile row (reachable if RLS is weak — SEC-2) can
  crash render or inject unexpected values into UI/links.
- **Fix:** a `coerce*` validator on every cloud read that drops invalid rows
  (= **BE-10**); never trust REST payloads structurally.
- **Verify:** tests feeding malformed rows assert graceful drop, no throw.

### SEC-13 — PII minimisation in telemetry (verify, don't assume)
- **Finding:** `scrubPII` (`lib/sentry.ts`) strips email/IP/cookies/Authz —
  good. PostHog now rate-limited; Clarity DNT-gated. But event props /
  breadcrumbs could still carry PII.
- **Fix:** assert the analytics event taxonomy (`lib/analytics-events.ts`)
  forbids free-text/PII props; add a test that no `addBreadcrumb`/`capture`
  call site passes raw user input. Document retention windows (ties to
  Analytics #19 / BE-27).
- **Verify:** structural test over call sites + the typed prop schema.

## C3 — Hardening / defense-in-depth

- **SEC-14 — Secret scanning in CI:** add gitleaks (or
  `mcp__github__run_secret_scanning` review) on every PR; enforce
  `CLAUDE.md`'s "never commit creds" rule mechanically. Add a `dist/` bundle
  grep for known secret patterns post-build.
- **SEC-15 — Web security headers / CSP:** GitHub Pages can't set headers,
  so add a strict `<meta http-equiv="Content-Security-Policy">` (+
  `referrer`, `X-Content-Type-Options` via meta where supported) injected by
  `scripts/build-spa-fallback.ts` — restrict `connect-src` to Supabase +
  Cloudinary + PostHog, `script-src 'self'`, block inline where feasible.
  Mitigates the XSS that makes SEC-4 catastrophic.
- **SEC-16 — Subresource Integrity / pinned third-party scripts:** the
  Clarity tag (`lib/clarity.ts`) and any CDN script injected at runtime
  should be reviewed; prefer no runtime-injected third-party JS on
  authenticated screens, or pin/version it.
- **SEC-17 — Dependency supply chain:** enable Dependabot/`npm audit signatures`,
  lockfile-only installs in CI (`npm ci`), and review `postinstall` scripts.
- **SEC-18 — Auth session hardening:** confirm Supabase Auth uses PKCE for
  OAuth, short-lived access tokens + rotating refresh, email-OTP rate limits
  (Supabase dashboard), and that `AsyncStorage`/`localStorage` session
  storage is acceptable for the threat model (it's readable by XSS — another
  reason for SEC-15).
- **SEC-19 — Abuse & enumeration:** `profiles.public_id`/`username` lookups
  should not allow unauthenticated enumeration of the full user base; gate
  list/search behind auth + pagination + rate limit (overlaps BE-28).
- **SEC-20 — Logging hygiene:** ensure no tokens/keys/PII are `console.log`'d
  in production (the runtime-config and OAuth paths are the risk spots);
  strip debug logs in the production build.
- **SEC-21 — Security policy & disclosure:** add `SECURITY.md` (supported
  versions, how to report), and a documented incident runbook (rotate
  Cloudinary/Supabase keys, revoke sessions) — you will need it the first
  time SEC-1's leaked secret is abused.
- **SEC-22 — Automated review gate:** wire the repo's `/security-review`
  skill (or `request_copilot_review`) into the PR template so security-
  sensitive diffs (anything under `lib/supabase-*`, `functions/`,
  `cloudinary`, `auth`) get an automatic pass.

## Suggested order (each independently shippable)

1. **Stop the bleeding:** SEC-1 (rotate + remove secret), SEC-4 (gate
   runtime config), SEC-8 (audit high), SEC-14 (secret scanning). Days, not
   weeks — these are live exposures.
2. **Close the authorization gap:** SEC-2 + SEC-3 (= BE-1/2/11/12/22) —
   the real backend security boundary.
3. **Lock the edges:** SEC-5, SEC-7, SEC-9, SEC-10, SEC-15.
4. **Completeness & DoS:** SEC-6, SEC-11, SEC-12, SEC-19.
5. **Ongoing:** SEC-13, SEC-16…SEC-22.
