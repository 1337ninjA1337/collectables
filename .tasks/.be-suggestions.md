# Backend (BE) suggestions — making the Supabase backend "real"

> Scope note: this app is **not greenfield**. It already runs a hybrid
> offline-first Supabase backend wired into every context:
> `profiles`, `collections`, `items`, `friend_requests` (via
> `lib/supabase-profiles.ts` → social + collections contexts),
> `chat_messages`, `chat_reads` (via `lib/supabase-chat.ts`),
> `marketplace_listings` (via `lib/supabase-marketplace.ts`), and
> write-only `analytics_events`. "Creating a real BE part" therefore means
> **completing, versioning, securing, and operationalising** what is
> half-built — not starting from scratch.

---

## 0. The single most important gap (do this first)

The code in `lib/supabase-profiles-shapes.ts` issues REST calls against
`/rest/v1/{profiles,collections,items,friend_requests}`, but there is **no
`CREATE TABLE` migration** for any of them in `supabase/migrations/` and **no
entry in `MANUAL-TASKS.md`**. Only `chat_messages`, `chat_reads`,
`marketplace_listings`, `analytics_events` and a few `ALTER`s are versioned.
The production schema today exists only in someone's Supabase dashboard —
unauditable, unrepeatable, undeployable to a fresh project.

- **BE-1** — Reverse-engineer the live schema for `profiles`, `collections`,
  `items`, `friend_requests` from the Supabase dashboard and the column
  names used in `lib/supabase-profiles-shapes.ts` /
  `lib/supabase-profiles.ts` / `lib/types.ts`. Write
  `supabase/migrations/20260520_base_schema.sql` containing the authoritative
  `CREATE TABLE … IF NOT EXISTS` for all four, every FK, and the existing
  `ALTER`s folded in (e.g. `items.cost_currency`).
- **BE-2** — Add a `MANUAL-TASKS.md` section for the base-schema migration
  (per CLAUDE.md: every DB change must be reproducible from MANUAL-TASKS.md).
- **BE-3** — Add a schema-parity test (`__tests__/base-schema-migration.test.ts`)
  asserting every column referenced by `*-shapes.ts` URL/body builders has a
  matching column in the migration SQL (comment-stripped). This is the same
  defensive pattern already used by `analytics-events-rls-verification.test.ts`.
- **BE-4** — Stand up a throwaway *second* Supabase project, run **only** the
  committed `supabase/migrations/*` in order, point a local build at it, and
  confirm the app works end-to-end. This proves the migrations are complete
  and ordered. Document the bootstrap in `README-DEPLOY.md`.

## 1. Schema, constraints & indexes

- **BE-5** — Make every entity server-keyed: `id uuid DEFAULT gen_random_uuid()`
  (reuse the `lib/uuid.ts` client-side generator as the idempotency key,
  exactly as chat already does). Audit `collections`/`items` for the same
  `msg-<ts>-<rand>` legacy-id class of bug fixed for chat in
  `20260516_chat_id_integrity.sql`.
- **BE-6** — Foreign keys with explicit `ON DELETE`:
  `items.collection_id → collections(id) ON DELETE CASCADE`,
  `collections.owner_user_id → auth.users(id) ON DELETE CASCADE`,
  `friend_requests.{from,to}_user_id → auth.users(id) ON DELETE CASCADE`.
- **BE-7** — Integrity CHECKs mirroring the chat-id pattern: `friend_requests`
  unique on `least(from,to), greatest(from,to)` so A→B and B→A can't
  double-insert; `collections.role` constrained to `('owner','viewer')`;
  `visibility` to its enum.
- **BE-8** — Indexes on every FK + the hot read paths:
  `items(collection_id)`, `collections(owner_user_id)`,
  `friend_requests(to_user_id, status)`, `profiles(public_id)` (the
  user-facing slug), `profiles(username)` (admin lookup). Missing FK indexes
  are the #1 free-tier performance killer.
- **BE-9** — `created_at timestamptz DEFAULT now()` + `updated_at` with a
  `moddatetime` trigger on every table, so the sync layer (BE-13) can do
  delta pulls instead of full-table reloads.
- **BE-10** — `NOT NULL` + sane defaults everywhere the client already
  guarantees a value; add a `coerce*`/`normalize*` runtime validator on every
  read path (the `normalizeListing` / `mergeCollectionsFromCloud` pattern) so
  a malformed row degrades instead of crashing first paint.

## 2. Row-Level Security (the real authorization model)

RLS is the actual backend security boundary — the REST anon key is public in
the web bundle. Every table must be `ENABLE ROW LEVEL SECURITY` with explicit
policies; no table may be readable by `anon` unless intentionally public.

- **BE-11** — Per-table policies:
  - `profiles`: owner full RW on own row; `SELECT` of public fields by any
    authenticated user (drives People/profile screens); admin override for
    the two hardcoded admin identities (move that allowlist into a
    `profiles.is_admin` column or a `app_admins` table — hardcoding
    `username === "1337antoxa"` in `SocialProvider` is not a server boundary).
  - `collections`/`items`: owner RW; `SELECT` when `visibility = 'public'`
    OR the viewer is a confirmed friend/follower (a SECURITY DEFINER
    `is_visible_to(viewer, owner)` SQL function encapsulates the
    follow/friend graph so policies stay one-liners).
  - `friend_requests`: each side may read/insert/delete only rows where they
    are `from` or `to`.
  - `chat_messages`/`chat_reads`: participant-only (verify the existing
    policies actually enforce this — add the leak test below regardless).
  - `marketplace_listings`: public `SELECT` of active rows; owner RW;
    buyer may set `buyer_user_id` exactly once via an Edge Function (BE-20),
    never via a blanket `UPDATE` policy.
- **BE-12** — Add an `anon`/`authenticated` **RLS leak test** for *every*
  table, generalising `__tests__/analytics-events-rls-verification.test.ts`
  and the `MANUAL-TASKS.md` "RLS leak check" block (`SET ROLE anon; SELECT …`
  must fail or return only intended rows). One pgTAP file
  (`supabase/tests/rls.sql`) run by `supabase test db` in CI is the durable
  form (already listed as a suggestion under Analytics #16).

## 3. Offline-first sync engine

The contexts currently cache to AsyncStorage and "merge cloud on top"
(`collections-cloud-merge.ts`). That's a good start but ad hoc per context.
Consolidate it.

- **BE-13** — A single `lib/sync-engine.ts`: per-entity (a) optimistic local
  write, (b) enqueue a pending mutation keyed by the client uuid, (c) flush
  on connectivity with idempotent `Prefer: resolution=ignore-duplicates`
  upserts — exactly the pattern already proven in `lib/chat-context.tsx`'s
  `flushPending`. Promote that logic out of chat so collections/items/social
  reuse it instead of three divergent copies.
- **BE-14** — Delta pulls: store `lastSyncedAt` per entity per user; pull
  `updated_at=gt.<cursor>` (needs BE-9) instead of full reloads — bounds
  free-tier egress and battery.
- **BE-15** — Conflict policy: last-write-wins by `updated_at` for scalar
  fields is fine for this domain; document it. For deletes, use soft-delete
  (`deleted_at`) + a tombstone sweep so an offline edit can't resurrect a
  row deleted on another device (the social graph already tracks
  "deleted profile IDs" — generalise that).
- **BE-16** — A `pending mutations` count surfaced in the UI (a "syncing…"
  pill) reusing the `realtimeOnline` pattern chat already has.
- **BE-17** — One-time **local→cloud import**: on first authenticated load,
  if AsyncStorage holds seed-derived or pre-cloud data with no cloud
  counterpart, upload it under the new `auth.uid()` and mark a
  `collectables-cloud-imported-v1` flag so it runs exactly once. Without this,
  existing installs silently lose their local-only collections.

## 4. Realtime

- **BE-18** — Route collections/items/friend_requests realtime through the
  existing `getSharedRealtimeClient` (`lib/supabase-realtime.ts`) with a
  fan-out registry so N open tabs share one socket (already a logged
  suggestion). Add `closeSharedRealtimeClient()` on sign-out.
- **BE-19** — Subscribe to `UPDATE`/`DELETE`, not just `INSERT`, on
  `collections`/`items`/`marketplace_listings` so a change on device A
  reflects on device B without a manual refresh (already flagged for
  marketplace `buyer_user_id`).

## 5. Server-authoritative logic (Edge Functions)

Anything a malicious client must not be able to forge belongs in a Deno
Edge Function with the service-role key, never in an RLS `UPDATE` policy.
`supabase/functions/{analytics-mirror,delete-account}` already exist — extend
the pattern.

- **BE-20** — `claim-listing`: atomically sets `buyer_user_id` + `sold_at`
  iff the row is still active and the buyer isn't the seller; rejects double
  claims. Replaces any client-side "mark sold" trust.
- **BE-21** — `accept-friend-request`: flips both directions to `friends`
  transactionally so the graph can't end up half-accepted.
- **BE-22** — `validate-premium`: premium is currently AsyncStorage
  (`premium-context.tsx`) — trivially forgeable. Move entitlement to a
  server-validated `subscriptions` table + a function that the client treats
  as the source of truth (gate paid features server-side, not in JS).
- **BE-23** — Self-check in every function that the resolved key carries the
  `service_role` claim (already a logged suggestion for analytics-mirror) so
  a misconfigured secret fails loudly at deploy, not silently per request.

## 6. Storage (photos)

Photos go through Cloudinary today (`lib/cloudinary.ts`). Decide explicitly:

- **BE-24** — Either (a) keep Cloudinary but route uploads through a signed
  Edge Function so the unsigned upload preset isn't abusable, **or**
  (b) migrate to Supabase Storage buckets with RLS mirroring the
  collection/item visibility rules. (b) consolidates the backend and removes
  a third-party sub-processor; (a) is less work. Record the decision and the
  retention/orphan-cleanup story (deleting an item must delete its asset).

## 7. Data lifecycle & compliance

- **BE-25** — `delete-account` exists — verify it cascades through every new
  table (BE-6 FKs make this automatic) and add an integration test.
- **BE-26** — Data export endpoint (GDPR portability): an Edge Function that
  streams the user's profile/collections/items/chat as JSON.
- **BE-27** — Retention sweeps via `pg_cron`: prune `analytics_events`
  (already suggested, ~13 mo), soft-deleted tombstones, and abandoned
  anonymous data. Document each window in the privacy policy (ties into
  Analytics #19).

## 8. Performance, abuse & free-tier safety

- **BE-28** — Cursor pagination on every list endpoint (collections grid,
  items, chat history — chat pagination already suggested). No unbounded
  `select=*`.
- **BE-29** — App-level write rate limiting reusing the sliding-window
  limiter just shipped for analytics/Sentry (extract
  `createSlidingWindowLimiter` per the logged suggestion) on mutation paths
  to bound a runaway client against Supabase quotas.
- **BE-30** — `EXPO_PUBLIC_*` cache-TTL + kill-switch envs (mirror
  `EXPO_PUBLIC_PROFILE_CACHE_TTL_MS`, `EXPO_PUBLIC_REALTIME_DISABLED`) for
  incident response without redeploying.

## 9. CI/CD, secrets & migration discipline

- **BE-31** — A GitHub Actions job that applies `supabase/migrations/*` to a
  scratch DB and runs `supabase test db` (RLS pgTAP) on every PR — migrations
  that don't apply cleanly never reach `main`.
- **BE-32** — Enforce CLAUDE.md's rule mechanically: a CI grep that fails if a
  new `supabase/migrations/*.sql` lands without a matching `MANUAL-TASKS.md`
  section (the doc is the operator's runbook).
- **BE-33** — Keep all keys in GitHub Secrets / EAS secrets (already the
  convention); document every new server var in `.env.example` and
  `README-DEPLOY.md`. Never commit the service-role key — it must live only
  in Edge Function secrets.
- **BE-34** — Migration filenames must sort in apply order; add a test that
  fails on a timestamp collision (two `20260516_*` already exist — fine, but
  fragile; assert ordering is deterministic).

## 10. Testing

- **BE-35** — Integration tests with an injected fake `fetcher`/`tokenProvider`
  (the pattern `supabase-chat`/`supabase-marketplace` tests already use) for
  every cloud wrapper: assert request shape, idempotency headers, retry, and
  error→requeue behaviour. Keep them peer-dep-free like the Sentry suite.
- **BE-36** — pgTAP for RLS (BE-12) and FK/cascade behaviour (BE-25).
- **BE-37** — Schema-parity test (BE-3) extended to every `*-shapes.ts`.

---

## Suggested phasing (each phase independently shippable)

1. **Phase 0 — Make it reproducible:** BE-1…BE-4, BE-31, BE-32. *Exit:* a
   fresh Supabase project built from committed migrations runs the app.
2. **Phase 1 — Lock it down:** BE-11, BE-12, BE-23, BE-36. *Exit:* anon/leak
   tests green for every table.
3. **Phase 2 — Trustworthy sync:** BE-5…BE-10, BE-13…BE-17. *Exit:* two
   devices converge; existing local installs import without data loss.
4. **Phase 3 — Server-authoritative:** BE-20…BE-22, BE-25, BE-26.
5. **Phase 4 — Scale & comply:** BE-18, BE-19, BE-24, BE-27…BE-30, BE-34.

## Open decisions to make before starting

- Cloudinary vs Supabase Storage (BE-24) — affects sub-processor list & privacy doc.
- Is "offline-first with LWW" acceptable, or do any entities need real
  conflict resolution / CRDTs? (Recommendation: LWW is sufficient here.)
- Should premium move to a real payment provider (RevenueCat/Stripe) now, or
  just server-validated entitlements first? (Recommendation: entitlements
  first; payments later.)
- Self-hosted Supabase vs hosted free tier ceiling — at what MAU do egress /
  row limits force a paid plan or a read replica?
