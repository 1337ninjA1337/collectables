# Sentry env setup ŌĆö step-by-step

Concrete walkthrough for provisioning the two Sentry env vars referenced by
`lib/sentry-config.ts`:

| Variable | Purpose | Required? |
| -------- | ------- | --------- |
| `EXPO_PUBLIC_SENTRY_DSN` | Sentry project Data Source Name. When empty, the SDK never initialises and `captureException` becomes a no-op. | Optional, but **recommended** for production |
| `EXPO_PUBLIC_SENTRY_ENV` | One of `development`, `staging`, `production`. `development` disables event reporting even if a DSN is present. | Optional. Defaults to `production`. |
| `EXPO_PUBLIC_APP_VERSION` | Release tag (`collectables@<version>`). Falls back to `app.json`'s `expo.version`. | Optional |

Auxiliary CI-only secrets used by the future Crash #8 / Crash #9 sourcemap
upload tasks:

| Variable | Where used |
| -------- | ---------- |
| `SENTRY_AUTH_TOKEN` | `getsentry/action-release@v1` step in `.github/workflows/deploy.yml` and EAS post-build hook |
| `SENTRY_ORG`        | Same |
| `SENTRY_PROJECT`    | Same |
| `SENTRY_RELEASE`    | Set automatically by CI to `collectables@${{ github.sha }}` |

## 1. Create the Sentry account (one-time)

1. Open https://sentry.io/signup/ and sign up. The free **Developer plan**
   includes 5,000 errors/month, single-user org ŌĆö sufficient for indie use.
2. After confirming your email, you land on the org dashboard. Note the **org
   slug** in the URL ŌĆö `https://<org-slug>.sentry.io/...` ŌĆö you'll need it
   later. You can also rename it under **Settings ŌåÆ General**.

## 2. Create the React Native project

1. **Projects ŌåÆ Create Project**.
2. Pick **React Native** as the platform.
3. Name it `collectables` (the slug becomes `collectables` automatically).
4. Set **Alert frequency** to "Alert me on every new issue" if you want
   immediate inbox pings; "When 10 events" is gentler.
5. Click **Create Project**.

The wizard shows a code snippet with a DSN ŌĆö copy the `https://ŌĆ”@ŌĆ”sentry.io/ŌĆ”`
line. That's the value of `EXPO_PUBLIC_SENTRY_DSN`.

## 3. Generate an auth token (for sourcemap uploads)

Sentry now uses **organization-scoped auth tokens** (the older "User Auth
Tokens" UI is deprecated for CI usage).

**Direct URL**: `https://<your-org-slug>.sentry.io/settings/auth-tokens/`

**Via the UI**:

1. Click **Settings** (gear icon at the bottom of the left sidebar).
2. In the Settings sidebar, under **Developer Settings**, click
   **Auth Tokens** (some orgs label it **Custom Integrations ŌåÆ Auth
   Tokens** depending on plan).
3. Click **Create New Token**.
4. The new UI groups scopes by resource (**Project / Release / Org / Team**)
   with `read` / `write` / `admin` for each. Tick:
   - **Release: write** ŌĆö required to create releases and upload sourcemaps.
     This is the new equivalent of the old `project:releases` scope.
   - **Org: read** ŌĆö required so the Sentry CLI / Webpack plugin can
     resolve your org slug. Equivalent to the old `org:read`.
   - **Project: read** *(optional)* ŌĆö only needed if the CLI errors out
     resolving the project slug, which happens on multi-project orgs.
5. Save the generated token. It's shown **once** ŌĆö copy it immediately. That's
   the value of `SENTRY_AUTH_TOKEN`.

> If you only see "User Auth Tokens" under your avatar (top-right ŌåÆ
> **User settings ŌåÆ API ŌåÆ Auth Tokens**), that page still works as a
> fallback, but the org-scoped token above is the recommended path going
> forward.

## 4. Add the secrets to GitHub Actions

In the GitHub UI for `1337ninja1337/collectables`:

1. **Settings ŌåÆ Secrets and variables ŌåÆ Actions ŌåÆ New repository secret**.
2. Add each of:
   - `EXPO_PUBLIC_SENTRY_DSN` = `<paste your DSN>`
   - `EXPO_PUBLIC_SENTRY_ENV` = `production`
   - `SENTRY_AUTH_TOKEN` = `<paste your token>`
   - `SENTRY_ORG` = `<your org slug>`
   - `SENTRY_PROJECT` = `collectables`

The deploy workflow (`.github/workflows/deploy.yml`) already injects every
`EXPO_PUBLIC_*` secret into the `expo export` step ŌĆö no workflow change is
needed for the runtime DSN. The `SENTRY_*` (non-`EXPO_PUBLIC_*`) secrets are
read by the future Crash #8 sourcemap upload step.

## 5. Add the secrets to EAS (for iOS / native builds)

```bash
eas secret:create --scope project --name EXPO_PUBLIC_SENTRY_DSN  --value "<paste your DSN>"
eas secret:create --scope project --name EXPO_PUBLIC_SENTRY_ENV  --value "production"
eas secret:create --scope project --name SENTRY_AUTH_TOKEN       --value "<paste your token>"
eas secret:create --scope project --name SENTRY_ORG              --value "<your org slug>"
eas secret:create --scope project --name SENTRY_PROJECT          --value "collectables"
```

Verify with `eas secret:list`.

## 6. Local development (.env)

Create a local `.env` (gitignored ŌĆö never commit) with placeholders:

```bash
# .env (gitignored)
EXPO_PUBLIC_SENTRY_DSN=
EXPO_PUBLIC_SENTRY_ENV=development
```

Leave the DSN empty during day-to-day work, or set
`EXPO_PUBLIC_SENTRY_ENV=development` so the SDK never initialises and your
quota stays untouched. The init helper enforces both gates
(`lib/sentry-config.ts:resolveSentryConfig`).

## 7. Verify the wiring

After your secrets are saved, **trigger a fresh deploy** so the bundle
re-builds with the new env. Either push a no-op commit to `main`, or in
the GitHub UI go to **Actions ŌåÆ Deploy to GitHub Pages ŌåÆ Run workflow**.

Once the deploy lands:

1. Open the deployed site at the GitHub Pages URL.
2. Open devtools console (F12 / ŌīźŌīśI) and run:

   ```js
   __sendSentryTestError()
   ```

   Returns `"captured"` when the SDK is enabled and the event was forwarded.
   Returns `"not-ready"` when a gate is blocking ŌĆö see `__sentryStatus()`.
   Returns `"rate-limited"` if you exhausted the 50/min cap with prior calls.

3. If you got `"not-ready"`, run the diagnostic helper:

   ```js
   __sentryStatus()
   ```

   It returns a structured snapshot:

   ```js
   {
     ready: false,
     initialised: true,
     optedOut: false,
     dsnPresent: false,         // ŌåÉ problem: secret not in the bundle
     environment: "production",
     release: "collectables@ŌĆ”",
     reason: "missing-dsn",      // ŌåÆ re-deploy after adding secrets
   }
   ```

   `reason` will be one of:
   - `"ready"` ŌĆö everything is wired, events are flowing.
   - `"not-initialised"` ŌĆö providers haven't booted yet (rare, race
     between the console call and provider mount).
   - `"user-opted-out"` ŌĆö the diagnostics toggle in Settings is off.
   - `"missing-dsn"` ŌĆö `EXPO_PUBLIC_SENTRY_DSN` is empty in the bundle.
     Re-check the GitHub Actions secret name (case-sensitive) and trigger a
     re-deploy.
   - `"development-env"` ŌĆö `EXPO_PUBLIC_SENTRY_ENV` is `development` (set
     it to `production` in GitHub Actions secrets and re-deploy).
   - `"init-failed"` ŌĆö the SDK loader threw. Check the devtools console
     for `[sentry] init failed` warnings.

4. Within ~1 minute of a `"captured"` result, the error appears in
   **Sentry ŌåÆ Issues** under the `collectables` project. The release tag
   should match your latest commit SHA (e.g. `collectables@2f29d99ŌĆ”`)
   since the deploy workflow tags every build with
   `EXPO_PUBLIC_SENTRY_RELEASE=collectables@${{ github.sha }}`.

You can also fire the test with a custom message:

   ```js
   __sendSentryTestError("custom message for issue title")
   ```

If nothing arrives, sanity-check:
- DSN is correct and unredacted (it includes a public key + project ID).
- `EXPO_PUBLIC_SENTRY_ENV` is **not** `development`.
- The deployed bundle was built **after** the secrets were added.
  GitHub Pages serves the last successful build.
- `lib/sentry.ts` was actually mounted in the provider tree (Crash #4).

## 9. Troubleshooting `dsnPresent: false` after secrets are [set]

If the deploy log shows `EXPO_PUBLIC_SENTRY_DSN: [set]` but
`__sentryStatus()` in the deployed bundle still reports
`dsnPresent: false` and `reason: "missing-dsn"`, the secret reached CI but
**Expo did not inline it into the JS bundle**.

### Why this happens

`babel-preset-expo` only replaces `process.env.EXPO_PUBLIC_*` when it sees a
**direct member access expression** literally written in the source. Passing
`process.env` as a generic object to a helper bypasses inlining and the
bundled code reads `undefined` at runtime ŌĆö even though the secret exists in
CI. The fix is to read each variable literally:

```ts
// Ō£ģ inlined ŌĆö Metro replaces this with the secret value at build time
const dsn = process.env.EXPO_PUBLIC_SENTRY_DSN;

// ŌØī NOT inlined ŌĆö Metro doesn't know which keys you'll access
const dsn = (process.env as Record<string, string | undefined>).EXPO_PUBLIC_SENTRY_DSN;
```

`lib/sentry-config.ts` exposes a `readSentryEnvFromProcess()` helper that
reads every supported variable literally. The module-level `sentryConfig`
constant and `initSentry()` both call it ŌĆö never pass `process.env` whole.
A structural test (`__tests__/sentry-env-inlining.test.ts`) enforces this.

### Quick verification

After a fresh deploy, open devtools on the site and run:

```js
JSON.stringify(__sentryStatus())
```

Expected output when wired correctly:

```json
{"ready":true,"dsnPresent":true,"environment":"production",ŌĆ”}
```

If `dsnPresent` is still false, double-check:
- The variable name in the GitHub Actions secret matches **exactly**
  (`EXPO_PUBLIC_SENTRY_DSN`, case-sensitive, no surrounding whitespace).
- The deploy step that calls `npx expo export --platform web` passes
  `EXPO_PUBLIC_SENTRY_DSN` through its `env:` block (it does ŌĆö see
  `.github/workflows/deploy.yml`).
- You're looking at the **fresh** bundle. GitHub Pages caches aggressively;
  hard-refresh (`Cmd/Ctrl+Shift+R`) or open in a private window.

## 10. Auth-token pitfalls

`SENTRY_AUTH_TOKEN` is **only** used by the sourcemap upload step
(`getsentry/action-release@v1`). It is **not** required for runtime crash
capture. Common failure modes:

| Symptom | Likely cause | Fix |
| ------- | ------------ | --- |
| Deploy log says `SENTRY_AUTH_TOKEN: [missing]`, but events arrive in Sentry | Token is unset; runtime DSN works but stack traces are minified | Generate a new org-scoped token (┬¦3) and add it as a GitHub Actions secret |
| `403 Forbidden` from `getsentry/action-release@v1` | Token missing `Release: write` scope (or the deprecated `project:releases`) | Recreate the token with `Release: write` + `Org: read` |
| `404 Not Found` resolving the project | `SENTRY_ORG` or `SENTRY_PROJECT` mismatch ŌĆö the slug is case-sensitive | Confirm both at `https://<org>.sentry.io/settings/projects/<project>/` |
| Token expired silently | Org tokens default to no expiry, but admins can rotate | Recreate the token; consider documenting the rotation date |
| `Authentication credentials were not provided.` in upload step | Token was added as an environment variable instead of a GitHub secret, or the secret name in the workflow yml doesn't match | Re-check the secret name maps to `SENTRY_AUTH_TOKEN` exactly |

Worth noting: the upload step is **gated** by `if: ${{ env.SENTRY_AUTH_TOKEN != '' }}`,
so a missing token never breaks the deploy ŌĆö it only skips sourcemap upload
silently. Look at the deploy log for the line `Skipping the Upload sourcemaps to Sentry step`
to confirm the gate fired.
