# Sentry env setup — step-by-step

Concrete walkthrough for provisioning the two Sentry env vars referenced by
`lib/sentry-config.ts`:

| Variable | Purpose | Required? |
| -------- | ------- | --------- |
| `EXPO_PUBLIC_SENTRY_DSN` | Sentry project Data Source Name. When empty, the SDK never initialises and `captureException` becomes a no-op. | Optional, but **recommended** for production |
| `EXPO_PUBLIC_SENTRY_ENV` | One of `development`, `staging`, `production`. `development` disables event reporting even if a DSN is present. | Optional. Defaults to `production`. |
| `EXPO_PUBLIC_APP_VERSION` | Release tag (`collectables@<version>`). Falls back to `app.json`'s `expo.version`. | Optional |

Auxiliary CI-only secrets used by the future Crash #8 / Crash #9 sourcemap
upload tasks:

| Variable | Where used |
| -------- | ---------- |
| `SENTRY_AUTH_TOKEN` | `getsentry/action-release@v1` step in `.github/workflows/deploy.yml` and EAS post-build hook |
| `SENTRY_ORG`        | Same |
| `SENTRY_PROJECT`    | Same |
| `SENTRY_RELEASE`    | Set automatically by CI to `collectables@${{ github.sha }}` |

---

## 1. Create the Sentry account (one-time)

1. Open https://sentry.io/signup/ and sign up. The free **Developer plan**
   includes 5,000 errors/month, single-user org — sufficient for indie use.
2. After confirming your email, you land on the org dashboard. Note the **org
   slug** in the URL — `https://<org-slug>.sentry.io/...` — you'll need it
   later. You can also rename it under **Settings → General**.

## 2. Create the React Native project

1. **Projects → Create Project**.
2. Pick **React Native** as the platform.
3. Name it `collectables` (the slug becomes `collectables` automatically).
4. Set **Alert frequency** to "Alert me on every new issue" if you want
   immediate inbox pings; "When 10 events" is gentler.
5. Click **Create Project**.

The wizard shows a code snippet with a DSN — copy the `https://…@…sentry.io/…`
line. That's the value of `EXPO_PUBLIC_SENTRY_DSN`.

## 3. Generate an auth token (for sourcemap uploads)

1. **User → User Auth Tokens → Create New Token**.
2. Required scopes:
   - `project:releases` — required to upload sourcemaps and tag releases.
   - `org:read` — required so the CLI can resolve the org slug.
3. Save the generated token. It's shown **once** — copy it immediately. That's
   the value of `SENTRY_AUTH_TOKEN`.

## 4. Add the secrets to GitHub Actions

In the GitHub UI for `1337ninja1337/collectables`:

1. **Settings → Secrets and variables → Actions → New repository secret**.
2. Add each of:
   - `EXPO_PUBLIC_SENTRY_DSN` = `<paste your DSN>`
   - `EXPO_PUBLIC_SENTRY_ENV` = `production`
   - `SENTRY_AUTH_TOKEN` = `<paste your token>`
   - `SENTRY_ORG` = `<your org slug>`
   - `SENTRY_PROJECT` = `collectables`

The deploy workflow (`.github/workflows/deploy.yml`) already injects every
`EXPO_PUBLIC_*` secret into the `expo export` step — no workflow change is
needed for the runtime DSN. The `SENTRY_*` (non-`EXPO_PUBLIC_*`) secrets are
read by the future Crash #8 sourcemap upload step.

## 5. Add the secrets to EAS (for iOS / native builds)

```bash
eas secret:create --scope project --name EXPO_PUBLIC_SENTRY_DSN  --value "<paste your DSN>"
eas secret:create --scope project --name EXPO_PUBLIC_SENTRY_ENV  --value "production"
eas secret:create --scope project --name SENTRY_AUTH_TOKEN       --value "<paste your token>"
eas secret:create --scope project --name SENTRY_ORG              --value "<your org slug>"
eas secret:create --scope project --name SENTRY_PROJECT          --value "collectables"
```

Verify with `eas secret:list`.

## 6. Local development (.env)

Create a local `.env` (gitignored — never commit) with placeholders:

```bash
# .env (gitignored)
EXPO_PUBLIC_SENTRY_DSN=
EXPO_PUBLIC_SENTRY_ENV=development
```

Leave the DSN empty during day-to-day work, or set
`EXPO_PUBLIC_SENTRY_ENV=development` so the SDK never initialises and your
quota stays untouched. The init helper enforces both gates
(`lib/sentry-config.ts:resolveSentryConfig`).

## 7. Verify the wiring

Once a deploy lands and the Crash #4 provider-wiring task ships:

1. Open the deployed site at the GitHub Pages URL.
2. In devtools console run `throw new Error("sentry smoke")`.
3. Within ~1 minute the error appears in **Sentry → Issues** under the
   `collectables` project. The release tag should match your latest commit
   SHA / `app.json` version.

If nothing arrives, sanity-check:
- DSN is correct and unredacted (it includes a public key + project ID).
- `EXPO_PUBLIC_SENTRY_ENV` is **not** `development`.
- The deployed bundle was built **after** the secrets were added.
  GitHub Pages serves the last successful build.
- `lib/sentry.ts` was actually mounted in the provider tree (Crash #4).

## 8. Privacy / GDPR considerations

- The PII scrubber (Crash #12) strips emails outside dev environments.
- A user-facing diagnostics opt-out toggle (Crash #15) persists under
  `collectables-diagnostics-v1` in AsyncStorage.
- Add Sentry to the public privacy policy as a sub-processor with a link to
  https://sentry.io/privacy/. The App Privacy declaration in
  `APPSTORE-SUBMISSION.md` already lists "Crash data / diagnostics" as
  collected once Sentry is enabled.
