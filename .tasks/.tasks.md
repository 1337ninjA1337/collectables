Add Sentry automatically to your app with the [Sentry wizard](https://docs.sentry.io/platforms/react-native/#install) (call this inside your project directory).

```bash
npx @sentry/wizard@latest -i reactNative --saas --org anton-m3 --project collectables
```

 Configure the SDK with your DSN
 Add source maps upload to your build process
 Add debug symbols upload to your build process

## Collection page chunked loading (decomposed from "items should load via chunks to prevent iOS memory leaks")

Context: `app/collection/[id].tsx` is wrapped in a `NestableScrollContainer` (so its outer scroll is a `ScrollView`, not a virtualized list) and has 3 mutually-exclusive render branches for items — (a) owner + drag mode → `NestableDraggableFlatList data={items}` (nested in a ScrollView, so no real virtualization), (b) owner + selection mode → plain `items.map(<SelectableItemRow>)` (unwindowed), (c) viewer/read-only → masonry `items.filter().map(<ItemCard compact>)` across two columns (unwindowed). All three mount every item-card (and its remote image) up-front, which is the iOS memory hot-spot the user flagged. Plan: ship a pure pagination helper first, wire it into all 3 branches behind one "Load more" CTA (minimum viable per the user — pagination is the explicit fallback), then optionally graduate to true `FlatList` windowing once the pagination is proven.
