# Analytics env setup — step-by-step

Concrete walkthrough for provisioning analytics env vars referenced by the
upcoming Analytics #1–#20 tasks. Two SDKs are layered; Power BI Desktop
consumes the resulting Supabase event mirror.

| Variable | Service | Required? |
| -------- | ------- | --------- |
| `EXPO_PUBLIC_POSTHOG_KEY`        | PostHog product analytics | Optional |
| `EXPO_PUBLIC_POSTHOG_HOST`       | `https://eu.posthog.com` (default) or `https://us.posthog.com` | Optional |
| `EXPO_PUBLIC_CLARITY_PROJECT_ID` | Microsoft Clarity (web-only session replay) | Optional |
| `POSTHOG_WEBHOOK_SECRET`         | Shared secret between PostHog and the Supabase Edge Function (server-only) | Required for Analytics #13 |

When **any** key is missing, `lib/analytics-config.ts` flips
`enabled: false` and the wrapper becomes a no-op — same pattern as
`lib/sentry-config.ts`.

---

## A. PostHog (product analytics — funnels, retention, cohorts)

### A.1 Create the PostHog account (one-time)

1. Go to https://posthog.com/signup. Pick the **EU** host
   (`https://eu.posthog.com`) for GDPR-friendly storage.
2. **Free tier**: 1,000,000 events/month, 5,000 session recordings/month. The
   rate limiter (Analytics #17) caps event volume at 200/min/user.
3. Create a project named `collectables`.

### A.2 Copy the project key

1. **Project settings → Project API Key**. The string starts with `phc_…`.
2. That value is `EXPO_PUBLIC_POSTHOG_KEY`.
3. The host (`https://eu.posthog.com` or `https://us.posthog.com`) is
   `EXPO_PUBLIC_POSTHOG_HOST`.

### A.3 Configure the outgoing webhook (Analytics #13)

PostHog can mirror every event to a custom HTTPS endpoint. We use this to
populate the Supabase `analytics_events` table that Power BI reads from.

1. **Project → Apps → Browse Apps → "Webhooks"** install.
2. Webhook URL: `https://<project>.supabase.co/functions/v1/analytics-mirror`
3. Add a custom header `x-mirror-secret: <random 32-byte hex>`. That random
   value is `POSTHOG_WEBHOOK_SECRET` (also stored on the Supabase function).
4. Set "Filter" to "All events" or restrict to the Analytics #5 taxonomy.

### A.4 GitHub Actions secrets

In **Settings → Secrets and variables → Actions**:

- `EXPO_PUBLIC_POSTHOG_KEY` = `phc_…`
- `EXPO_PUBLIC_POSTHOG_HOST` = `https://eu.posthog.com`

### A.5 EAS secrets

```bash
eas secret:create --scope project --name EXPO_PUBLIC_POSTHOG_KEY  --value "phc_…"
eas secret:create --scope project --name EXPO_PUBLIC_POSTHOG_HOST --value "https://eu.posthog.com"
```

## B. Microsoft Clarity (web-only session replay + heatmaps, fully free)

### B.1 Create the Clarity project

1. Open https://clarity.microsoft.com/ and sign in with a Microsoft
   account.
2. **+ New project**, name it `collectables`, paste the GitHub Pages URL
   (`https://1337ninja1337.github.io/collectables`).
3. Clarity is **fully free** — no event quota.

### B.2 Copy the project ID

1. **Settings → Setup → Tracking code**. The 10-character lowercase ID is
   embedded in the snippet (`clarity("set", …)` or `c.id = "<id>"`).
2. That value is `EXPO_PUBLIC_CLARITY_PROJECT_ID`.

### B.3 Inject the script (Analytics #11)

Analytics #11 mounts the Clarity script tag into the web bundle when
`Platform.OS === "web"` and `EXPO_PUBLIC_CLARITY_PROJECT_ID` is non-empty.
On native (iOS/Android) the script is skipped entirely — Clarity is web-only.

### B.4 Privacy

Clarity masks `<input>` values by default. Add an explicit
`data-clarity-mask="true"` attribute to any DOM node displaying user content
(usernames, chat bubbles) before flipping replay on. The Clarity dashboard
also offers a global "mask all text" toggle for stricter cases.

### B.5 GitHub Actions secret

`EXPO_PUBLIC_CLARITY_PROJECT_ID` = `<your 10-char ID>`

### B.6 EAS secret (web bundle still needs it for the EAS-built web target)

```bash
eas secret:create --scope project --name EXPO_PUBLIC_CLARITY_PROJECT_ID --value "<id>"
```

## C. Power BI Desktop (free, BI surface)

Power BI is **not** an in-app SDK — it's a desktop tool that reads from the
Supabase `analytics_events` table populated by PostHog's webhook. There are
no env vars in the app for Power BI; setup is purely operator-side.

### C.1 Install Power BI Desktop

- Windows: https://www.microsoft.com/en-us/download/details.aspx?id=58494
- macOS: not officially supported. Use Power BI Service (web-only) at
  https://app.powerbi.com/ (free tier — 1 GB workspace, 8 daily refreshes).

### C.2 Create the Supabase read-only role

In Supabase SQL Editor:

```sql
CREATE ROLE powerbi_reader LOGIN PASSWORD '<random>';
GRANT CONNECT ON DATABASE postgres TO powerbi_reader;
GRANT USAGE ON SCHEMA public TO powerbi_reader;
GRANT SELECT ON public.analytics_events TO powerbi_reader;
-- Future tables added via:
-- ALTER DEFAULT PRIVILEGES IN SCHEMA public
--   GRANT SELECT ON TABLES TO powerbi_reader;
```

Add the role's `host:port:database:user:password` to the GitHub repo only as
**operator notes**, not as a secret — Power BI Desktop runs on your machine
and connects directly to Supabase.

### C.3 Connect from Power BI

1. **Get Data → PostgreSQL database**.
2. Server: `db.<project-ref>.supabase.co:5432`
3. Database: `postgres`
4. Authentication: `Database` → user `powerbi_reader` + the password from
   step C.2.
5. Pick `public.analytics_events` and load.

### C.4 Starter dashboard (Analytics #15)

A `.pbit` template ships under `docs/powerbi/Collectables-Starter.pbit`
once Analytics #15 lands. Open it and paste your Supabase host when
prompted.

## D. Local development (.env)

```bash
# .env (gitignored)
EXPO_PUBLIC_POSTHOG_KEY=
EXPO_PUBLIC_POSTHOG_HOST=https://eu.posthog.com
EXPO_PUBLIC_CLARITY_PROJECT_ID=
```

Leave keys empty in development so neither SDK initialises and the free-tier
quotas stay untouched. The Analytics #18 GDPR opt-out toggle and the
`navigator.doNotTrack` check both also gate the wrappers at runtime.

## E. Verify the wiring

Once Analytics #4 + Analytics #6 + Analytics #11 land:

1. Deploy with all secrets present.
2. Open the site, click **Sign up**, complete onboarding.
3. Within 30s the `signup_completed` event appears in **PostHog → Events**.
4. **PostHog → Replays** shows a session.
5. **Clarity → Recordings** shows the same session (web-only).
6. Run `SELECT * FROM analytics_events ORDER BY occurred_at DESC LIMIT 10;`
   in Supabase — the webhook should be inserting rows.
7. Refresh the Power BI starter report — DAU should tick to 1.

If nothing arrives:
- PostHog "Live events" tab shows raw events even before processing — check
  there first.
- The Edge Function (Analytics #13) logs failed webhook deliveries; tail it
  via `supabase functions logs analytics-mirror`.
- Clarity has a 2-hour processing delay before recordings render in the UI.

## F. Privacy / GDPR considerations

- All three integrations are listed in `APPSTORE-SUBMISSION.md` under
  "App Privacy" once Analytics #19 lands.
- The diagnostics opt-out toggle (Analytics #18) shares storage with the
  Sentry opt-out (Crash #15) under `collectables-diagnostics-v1`.
- The browser's `navigator.doNotTrack === "1"` is honoured on web.
- PostHog and Microsoft Clarity must be added to the public privacy policy
  as sub-processors. Power BI Desktop is operator-side and does not need
  user-facing disclosure.
