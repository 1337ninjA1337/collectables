# Upgrade Suggestions

- Add a `release` GitHub Actions workflow that runs `npm test`, `npx tsc --noEmit`, and `npm run build` on pull requests so regressions never reach `main`. The tests already run in ~2s locally; gating PRs is essentially free and would prevent broken builds from landing.
- Add `npm run build` (`expo export --platform web`) as a step in `ci.yml`. The web bundle currently only gets exercised in the deploy workflow, so pure-JS build failures can merge to `main` and only surface on deploy. Running the export in CI catches them pre-merge at the cost of ~1min of runner time.
- Cache resolved viewer profiles in `SocialProvider` (not just in `collection/[id].tsx` local state) so that the People-with-access list, profile links, and chat screens share one source of truth. Today opening the share sheet refetches non-friend profiles on every render pass, and the same ID may be fetched again on the profile screen.
- Surface a small "configured by you in this browser" badge plus a "Clear runtime credentials" button next to the new runtime Supabase config form, so a user who pasted the wrong URL/key can reset without digging into devtools `localStorage`. The clear helper (`clearRuntimeSupabaseConfig`) already exists; this is purely UI plumbing.
- Wire `supabase/migrations/` into a `supabase db push` step in the deploy workflow (gated on a `SUPABASE_DB_URL` secret). Right now migrations like the new `20260424_chat_messages.sql` need to be applied by hand against the project, which is easy to forget and creates drift between the SQL in git and what's actually live.
- Promote the shape-helper pattern used by the new `lib/supabase-chat-shapes.ts` (pure URL/body/header builders kept separate from the fetch wrapper) to the older `lib/supabase-profiles.ts`. Today `supabase-profiles.ts` builds URLs inline inside async functions, which means the only way to verify shape is to mock `fetch`. Splitting out shape helpers would let tests assert the URL/body shape for `upsertCollection`, `fetchProfiles`, and the friend-request endpoints without spinning up a fake fetch.
- Add an outbox queue for offline-sent chat messages: when `cloudSendMessage` returns null in `chat-context.tsx`, the locally-generated message stays only on the originating device and is never replayed once the network comes back. A small `pendingByChatId` slice in the existing AsyncStorage cache plus a flush-on-reconnect effect would close that gap without adding a new storage key.
- Surface the realtime channel state from `subscribeToInbox` (a `connected` boolean) and render a tiny "offline / reconnecting" pill in `app/chat/[id].tsx`. Today realtime-js silently auto-reconnects, so users can stare at a stale conversation with no signal that pushed messages are paused. Plumbing the SUBSCRIBED / CHANNEL_ERROR status up through chat-context costs a few lines and dramatically improves debuggability.
- Auto-clear stale typing presence on tab/window blur (web) and `AppState` background (native) by calling `setTyping(false)` from a single hook. Today the 1s debounce in `app/chat/[id].tsx` clears typing on idle, but if the user backgrounds the app mid-keystroke the other side keeps seeing "is typing..." until the realtime presence channel times them out (~30s). Listening for visibility/AppState changes inside `subscribeToTyping`'s caller is a few lines and removes the visual lag.
- Persist `lastReadByChat` to a server-side `chat_reads` table so the unread badge stays consistent across devices. Today the count is read off `lastReadByChat` in AsyncStorage, so opening a chat on phone A still leaves the dot lit on phone B. A small `chat_reads(user_id, chat_id, last_read_at)` table with an UPSERT on `markRead` plus a SELECT on hydrate would close that gap and reuse the existing supabase-chat REST scaffolding.
- Refactor the runtime wrappers in `lib/supabase-chat.ts` to take an injected `fetcher`/`tokenProvider` so they can be exercised under `node --test` without the React Native peers. Today the wiring is covered by the regex/structural tests in `supabase-chat-wiring.test.ts`, which do catch missing helper imports but won't catch e.g. a swapped `body`/`headers` argument. A 10-line dependency-injection refactor would let us assert real `fetch.calls[0].arguments` shapes.
- Move marketplace listings off AsyncStorage and into a Supabase `marketplace_listings` table once the chat-cloud pattern stabilises. The current `lib/marketplace-context.tsx` is a single-device store, which means a listing made on phone A is invisible to phone B and to other users — the entire point of a marketplace. Mirror the shape-helpers + REST wrapper pattern from `lib/supabase-chat.ts` so the cloud migration is a drop-in that keeps the existing pure helpers testable.
- Lift the bottom-nav "balanced layout" trick (a hidden flex spacer cell so the centered plus button keeps its position when the number of tabs grows) into a small `<NavRow>` helper that takes `leftItems` / `rightItems` arrays and pads automatically. Today the spacer in `components/bottom-nav.tsx` is hard-coded and will silently re-skew the next time someone adds another tab; a 30-line helper that always pads the shorter side closes that gap and makes future tabs (e.g. notifications, stats) cheap to add.
- Replace the bare `<TextInput keyboardType="numeric">` price field in the marketplace listing sheet (`app/item/[id].tsx`) with a small `<CurrencyInput>` helper that strips non-numeric characters as you type, normalises commas to dots, and surfaces a `currency` selector matching the user's locale. Today the user can paste arbitrary text into the price field and the only validation is `Number(...)` at submit time, which silently drops to a null price when the input doesn't parse — the listing then publishes as "trade" with no warning. Centralising this also unlocks consistent behaviour for any future cost/price field.
- Hydrate the listing detail screen (`app/listing/[id].tsx`) from a remote `fetchListingById` helper so deep-link visits work for users whose AsyncStorage hasn't seen the listing yet. Today `getListingById` only sees the local cache, so opening a `/listing/{id}` link from another device renders the not-found EmptyState even when the listing exists on the owner's machine. Once the marketplace moves to Supabase (separate suggestion), wiring an additional REST fetch into the existing `useEffect` will close the gap with one new helper in `lib/supabase-profiles.ts`.
- Tighten the `priceHistoryForTitle` similarity heuristic by also stripping common collectible-stopwords (e.g. "holo", "rare", "edition", "limited", "1st") before computing the Dice coefficient. Today "Charizard Holo 1999" and "Charizard 1st Edition 1999" land at ~0.71 similarity and never meet the 0.9 threshold, even though a human would call them comparable comps. A 20-line stopword filter inside `normalizeTitle` (gated by a constant so tests remain deterministic) would dramatically widen the comp pool without false positives because the stopwords are generic.
- Add a CI smoke step that boots `npx expo export --platform web` and parses the resulting `dist/_expo/static/js/web/*.js` for known regression markers (e.g. an `marketplaceListOnMarketplace` translation key that fails to resolve, or a `_layout.tsx` provider that wasn't wrapped). Today regression detection for the marketplace flow lives only in `__tests__/marketplace-*.test.ts`, which exercise pure helpers but cannot catch screen-level wiring breakage (a missing provider, a stale i18n key) — those slip past `tsc --noEmit` because they're string-keyed lookups. A 30-line bundle-grep step in CI catches the long tail of "build green, screen blank" regressions for free.
- Centralise per-user AsyncStorage keys (chat, social, premium) into a single `lib/storage-keys.ts` module that exports `chatCacheKey(userId)`, `socialCacheKey(userId)`, `premiumKey(userId)`. Today each context hardcodes its own template (`collectables-chats-v1-${userId}`, `collectables-social-v1-${userId}`, `collectables-premium-v1-${userId}`) which means renaming a key requires grepping multiple files and risks leaving stale data behind. A 20-line key-builder module also makes it trivial to add a `clearAllUserData(userId)` helper for the existing "Delete account" flow which today only clears auth + cloud and leaves these caches behind.
- Render a small "Premium" badge next to the user's avatar in `components/bottom-nav.tsx` (and on profile chips) by threading `isPremium` from `usePremium()` into the existing avatar rendering. Today the only signal that premium is active lives on the settings screen, so users have no ambient confirmation that their upgrade took effect — a tiny gold dot or star overlay closes that gap and surfaces the social value of the feature without adding a new screen.
- Add a `formatRelativeDate(iso, locale)` helper in `lib/i18n-context.tsx` and reuse it for the new `premiumActiveSince` string instead of the bare `activatedAt.slice(0, 10)`. Today the settings screen renders raw ISO date prefixes ("Premium since 2026-04-25") which look out of place next to the rest of the UI which speaks human ("just now", "yesterday"). A 30-line `Intl.RelativeTimeFormat`-based helper would also unblock similar polish for the chat preview "last seen" line and the listing "added X ago" hint.
