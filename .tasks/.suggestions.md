# Upgrade Suggestions

- Extract the duplicated `defaultLoader` + module-state + `__resetForTests` boilerplate out of `lib/sentry.ts` and `lib/analytics.ts` into a shared `lib/lazy-sdk.ts:createLazySdkWrapper({ loader, gate })` factory. Today both files reimplement the same `sdk` / `initialised` / `activeConfig` triad with the same try/catch swallow shape; a third telemetry SDK (e.g. Microsoft Clarity for native, see Analytics #11) would make this the third copy.

- Add a sliding-window rate limiter to `lib/analytics.ts:trackEvent` mirroring the Sentry pattern (`rateLimitAllow` in `lib/sentry.ts`). Analytics #17 already plans a 200 events/min/user limiter — but the wrapper currently has zero protection, so a runaway useEffect can exhaust PostHog's 1M-events/mo free tier in a single session. Worth landing the guard before #5–#10 begin firing events.

- Promote the wrapper-loader pattern's `loader: async (config) => sdk` shape into a typed `LazyTelemetryLoader<TSdk, TConfig>` generic in `lib/types.ts` so `SentryLoader` and `AnalyticsLoader` derive from one source. Today the two interface types sit in their own modules and a future SDK would need its own ad-hoc shape — a generic closes the drift.

- Wire `__tests__/analytics-init.test.ts` into the existing `__tests__/sentry-tests-peer-dep-free.test.ts` invariant (or write a parallel `analytics-tests-peer-dep-free.test.ts`) so a future contributor adding a top-level `import "posthog-react-native"` to *any* analytics test fails CI before the SDK isolation regresses. Today the guard only covers Sentry tests; analytics has the same fragility.

- Add an `EXPO_PUBLIC_ANALYTICS_ENV` row to README-DEPLOY.md + APPSTORE-SUBMISSION.md documentation. The new resolver supports it but the docs only list `EXPO_PUBLIC_SENTRY_ENV`; engineers reading the env table won't discover the override unless they read the source.

- Extract `normaliseEnvironment` from both `lib/sentry-config.ts` and `lib/analytics-config.ts` into a shared `lib/env.ts:normaliseDeploymentEnv(value)` helper so the dev/staging/production collapse rule has one canonical implementation. Today the function is duplicated; a future "preview" environment added to one would silently fall back to "production" in the other.

- Add an `EXPO_PUBLIC_ANALYTICS_ENV` row to README-DEPLOY.md + APPSTORE-SUBMISSION.md documentation. The new resolver supports it but the docs only list `EXPO_PUBLIC_SENTRY_ENV`; engineers reading the env table won't discover the override unless they read the source.

- Add a `scripts/check-expo-install.ts` post-install hook that re-runs `npx expo install --check` against the locked `posthog-*` versions and fails CI if Expo's registry recommends a different version for the current SDK. Today the sandbox can't reach the Expo registry, so versions were pinned manually; a CI hook running on full network would surface the recommended version drift before it lands in TestFlight.

- Bundle-size budget: add a `scripts/check-bundle-size.ts` step that fails CI when the web bundle exceeds, say, 4.5MB. PostHog adds ~60KB and Clarity adds ~30KB; with future Analytics #11 (Clarity script injection) we'll cross 4.1MB. A budget guardrail catches a regression where someone imports `posthog-js` statically into a screen instead of lazily.

- Translate `docs/analytics-platform.md` "Privacy and opt-out" section into the public privacy policy (the same one that needs the Sentry paragraph from `APPSTORE-SUBMISSION.md` §6) so users see a unified disclosure for analytics + crash reporting. Today the analytics opt-out story only exists in an engineering doc; user-facing transparency requires it live on the privacy page.

- Add a `docs/analytics-cost-projection.md` companion that turns the $0/month free-tier ceiling in `docs/analytics-platform.md` into a small linear-projection table (DAU → events/mo → next-tier price) so the founder gets a one-glance "you'll cross PostHog's free tier when DAU hits ~30" signal. Today the doc says "1M events/mo at ~30 DAU"; surfacing the math lets the team know *when* to budget for the upgrade.

- Wire the new `__tests__/sentry-tests-peer-dep-free.test.ts` into a `lint:peer-dep-free` script in `package.json` that runs *only* that single test, so a pre-push hook (or a pre-merge gate) can verify the SDK isolation invariant in <100ms instead of waiting for the full 7s test suite. Today the guard runs as part of `npm test`; isolating it makes the failure reason obvious at a glance.

- Promote the `SENTRY_TEST_FILES` array in `__tests__/sentry-tests-peer-dep-free.test.ts` from a hand-maintained list to a glob (`fs.readdirSync` + filter on `^sentry-.*\\.test\\.ts$`) so a future Sentry test file is picked up without editing the array. Today missing a file from the list leaves it un-guarded; a glob makes the invariant fail-closed.

- Add a "delete `node_modules/@sentry/react-native` and re-run only the sentry tests" step to CI as a separate matrix job. Today the guard structurally proves no static import — but a smoke-run with the SDK literally absent would catch a *transitive* dependency that imports it (e.g. `@sentry/core`). Cheap to add: `rm -rf node_modules/@sentry/react-native && npx tsx --test __tests__/sentry-*.test.ts`.

- Commit a `PRIVACY.md` to the repo root that consumes the "Suggested public privacy policy paragraph" from `APPSTORE-SUBMISSION.md` §6 plus the equivalent paragraphs for Supabase + Cloudinary, and host it via the existing GitHub Pages workflow at `/privacy`. Today the policy text only lives in submission docs — App Store review requires a public URL, and a tracked file makes the privacy guarantees enforceable via PR review instead of "trust the markdown".

- Generate translated copies of the Sentry privacy paragraph (`PRIVACY.md.ru`, `PRIVACY.md.de`, etc.) using the existing `lib/i18n-context.tsx` language list, then surface a language picker on the privacy page. Today the paragraph is English-only; a non-English EU user is technically owed the disclosure in their language under GDPR Art. 12.

- Add a `lib/privacy-disclosure.ts` constant that lists every sub-processor (Supabase, Cloudinary, Sentry, future PostHog) with `name + url + dataTypes + retention` so the App Privacy table, the public PRIVACY.md, and the in-app Settings → Privacy disclosure all read from one source of truth. Drift between the three is the most common privacy-review finding.

- Surface the diagnostics opt-out state on `app/settings.tsx` with a clear "Crash reports: enabled — last sent 3 hours ago" footer driven by the new `getSentryStatus()` snapshot. Today the toggle exists but doesn't tell the user whether anything was actually sent — useful for users who flip the toggle and want to verify nothing is leaking.

- Promote the `app.json` plugin/extra parity guard (`__tests__/sentry-eas-config.test.ts`) into a generic `__tests__/app-json-parity.test.ts` that walks every `expo.plugins` array entry that has a matching `expo.extra.<key>` block (Sentry today, future Reanimated/Notifications) and asserts the duplicate fields don't drift. Today only Sentry is checked; a generic walker means future config-plugin additions get parity for free.

- Codify the `SENTRY_AUTH_TOKEN`-not-required-for-runtime invariant into a `lib/sentry-config.ts` doc comment + a `getSentryStatus()` field (`sourcemapsExpected: boolean`) so the diagnostics screen can show "events flowing but stack traces minified — add SENTRY_AUTH_TOKEN" when the bundle has a DSN but the deploy log shows no sourcemap upload step. Closes the gap between "runtime works" and "debugging works" without introducing a new secret-validation pass.

- Add a CI grep step that fails the build if `process.env` is passed whole (e.g. `process.env as Record<`, `(process.env)`) into any `lib/*-config.ts` resolver. Today only `lib/sentry-config.ts` is guarded by a structural test; promoting that test into a generic `.github/workflows/ci.yml` `grep -r "process\\.env as Record" lib/ && exit 1` line catches the same Metro/babel inlining footgun for future config helpers (cloudinary, posthog, etc.) without adding a per-module test.

- Promote `readSentryEnvFromProcess` into a shared `lib/expo-public-env.ts` helper exporting `readExpoPublicEnv(names: readonly string[])` that builds a literal-access object generator. Today every config module has to hand-roll the `EXPO_PUBLIC_X: process.env.EXPO_PUBLIC_X` pairs; a typed helper using a `const` tuple of var names + `Record<typeof names[number], string | undefined>` would compress the boilerplate and make adding a new EXPO_PUBLIC_ var a one-line change.

- Surface the bundle-inlining state on the diagnostics screen: render a "Sentry DSN inlined: ✅ / ❌" row sourced from `getSentryStatus()` so non-engineer testers can verify their TestFlight/preview build was produced from a CI run with the secrets — without touching devtools. Hide the row in production for users (gated on `__DEV__` or admin), but keep it on internal builds.

- Add a `scripts/verify-bundle-inlining.ts` post-build check that grep-tests `dist/_expo/static/js/web/*.js` for the literal `EXPO_PUBLIC_SENTRY_DSN:"https://` substring (or any documented secret) and fails the deploy when it's missing. Today CI prints `[set]` from the secret-status step, but a deploy still ships a bundle with `dsnPresent: false`; an automated bundle inspection closes the loop between "secret in CI" and "secret in artifact".

- Auto-resolve the `EXPO_PUBLIC_APP_VERSION` from the package.json `version` field at build time (read in `app.config.ts` and exported into `extra.expoPublicAppVersion`) so the Sentry release tag stays in sync with package.json without each deploy re-setting `EXPO_PUBLIC_SENTRY_RELEASE`. Removes one CI env var and prevents drift between the npm version and Sentry's release record.
