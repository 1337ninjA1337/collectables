# Upgrade Suggestions

- Add an `expo-dev-menu` peer dep wired through `loadDevMenuModule` in `lib/dev-menu.ts` so the new dev-only DevMenu hook actually surfaces a "Clear runtime Supabase config" entry on native dev builds. Today `loadDevMenuModule` returns null because the package isn't installed, so engineers only get the `globalThis.__clearRuntimeSupabaseConfig` web-console fallback. A 1-line `npx expo install expo-dev-menu` plus a custom config plugin entry would unlock the native DevMenu path without changing any application code.

- Move the `__DEV__` global type declaration into a project-wide `types/globals.d.ts` so both `lib/dev-menu.ts` and any future dev-gated module can reference `__DEV__` without the `(globalThis as { __DEV__?: boolean })` cast. The cast pattern works but already appears in two places; a single ambient declaration removes the repetition and lets TypeScript flag misspelled `__dev__` accesses.

- Extend `registerDevMenu` in `lib/dev-menu.ts` to accept a per-action label (instead of using the function key as the DevMenu label) so engineers see "Clear runtime Supabase config" in the menu instead of the camelCase identifier. Today the action map's keys double as labels, which is fine for `globalThis` helpers but reads awkwardly in the user-facing DevMenu list.

- Auto-register a `__resetCollectablesStorage` dev helper alongside `__clearRuntimeSupabaseConfig` that wipes all `collectables-*` AsyncStorage keys via the existing `lib/storage-keys.ts` exports. Today engineers manually clear AsyncStorage to reset onboarding state; a one-line dev escape hatch keyed off the same `registerDevMenu` call closes that gap.

- Add an integration test in `__tests__/dev-menu-wiring.test.ts` that asserts `app/_layout.tsx` calls `registerDevMenu` with `clearRuntimeSupabaseConfig` in its `actions` map. Today the unit test only covers `registerDevMenu` itself; a structural regex test against `_layout.tsx` would catch a future regression where someone removes the `useEffect` block while refactoring providers.

- Promote the new `normalizeListing` helper out of `lib/marketplace-context.tsx` into `lib/marketplace-helpers.ts` so any future code path that hydrates `MarketplaceListing` objects from JSON (push notifications, deep-link previews, server-rendered routes) can reuse the same defensive-default logic without re-implementing the `?? null` coalesce. Keep the original call site untouched.

- Add a `cloud-marketplace` test file (`__tests__/supabase-marketplace.test.ts`) that mocks `fetcher` and `tokenProvider` to assert `cloudMarkSold` actually serialises `buyer_user_id` into the PATCH body. Today only the pure shapes module is unit-tested; threading the new buyer field through the cloud wrapper layer means a future regression (e.g. someone re-introducing the old `{ sold_at }` literal) would go unnoticed until UI tests catch it.

- Surface a small "transferred" badge on `app/listing/[id].tsx` and the marketplace card grid for sold listings whose `buyerUserId !== null`. Today sold listings are simply hidden from the active feed; once the buy/accept flow lands, an explicit "Sold to {buyerName}" pill on `My listings` makes the cycle visible without opening detail. The plumbing now exists — only the rendering is missing.

- Add a `buyerUserId` filter helper `listingsAcquiredByUser(listings, userId)` in `lib/marketplace-helpers.ts` (mirroring the shape of `listingsForUser`) so future profile screens can render a "Trades & Transfers" section without re-implementing the `l.buyerUserId === userId && l.soldAt != null` predicate. A 5-line addition with a paired unit test.

- Replace the loose `JSON.parse(raw) as MarketplaceListing[]` cast in `lib/marketplace-context.tsx` with a runtime validator that drops malformed entries (e.g. missing `id` or `mode`) instead of trusting the AsyncStorage payload. Today a corrupt local cache crashes downstream rendering because `mode` ends up `undefined`; a `coerceListing(raw)` helper that returns `null` on validation failure would harden first-paint without changing the happy path.
