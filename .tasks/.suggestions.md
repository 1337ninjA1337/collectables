# Upgrade Suggestions

- Add an `expo-dev-menu` peer dep wired through `loadDevMenuModule` in `lib/dev-menu.ts` so the new dev-only DevMenu hook actually surfaces a "Clear runtime Supabase config" entry on native dev builds. Today `loadDevMenuModule` returns null because the package isn't installed, so engineers only get the `globalThis.__clearRuntimeSupabaseConfig` web-console fallback. A 1-line `npx expo install expo-dev-menu` plus a custom config plugin entry would unlock the native DevMenu path without changing any application code.

- Move the `__DEV__` global type declaration into a project-wide `types/globals.d.ts` so both `lib/dev-menu.ts` and any future dev-gated module can reference `__DEV__` without the `(globalThis as { __DEV__?: boolean })` cast. The cast pattern works but already appears in two places; a single ambient declaration removes the repetition and lets TypeScript flag misspelled `__dev__` accesses.

- Extend `registerDevMenu` in `lib/dev-menu.ts` to accept a per-action label (instead of using the function key as the DevMenu label) so engineers see "Clear runtime Supabase config" in the menu instead of the camelCase identifier. Today the action map's keys double as labels, which is fine for `globalThis` helpers but reads awkwardly in the user-facing DevMenu list.

- Auto-register a `__resetCollectablesStorage` dev helper alongside `__clearRuntimeSupabaseConfig` that wipes all `collectables-*` AsyncStorage keys via the existing `lib/storage-keys.ts` exports. Today engineers manually clear AsyncStorage to reset onboarding state; a one-line dev escape hatch keyed off the same `registerDevMenu` call closes that gap.

- Add an integration test in `__tests__/dev-menu-wiring.test.ts` that asserts `app/_layout.tsx` calls `registerDevMenu` with `clearRuntimeSupabaseConfig` in its `actions` map. Today the unit test only covers `registerDevMenu` itself; a structural regex test against `_layout.tsx` would catch a future regression where someone removes the `useEffect` block while refactoring providers.

- Surface a "Sold to @buyer" pill on the marketplace `ListingCard` for sold listings (currently filtered out via `activeListings`), keyed off the new `MarketplaceListing.buyerUserId` field. A "recently sold" carousel under the active grid would let users see what's moving without changing the active-only marketplace contract — useful pricing context for sellers drafting their own listings.

- Add a `MarketplaceListingHistory` view on the user profile screen that lists prior `buyerUserId === user.id` listings (purchases) and `ownerUserId === user.id && soldAt !== null` listings (sales). The data is already in `useMarketplace().listings`; it's a single `useMemo` filter and a list render, no new context required.

- Persist the `buyerUserId` change through the `subscribeToListings` realtime channel by switching the filter from `event: INSERT` to also include `UPDATE` events, so when a buyer claims a listing on device A it disappears from device B's marketplace immediately rather than after a manual refresh. The PostgresChanges payload already exposes the updated `buyer_user_id`; threading it through the same `rowToListing` path would close the gap with ~10 LOC.

- Add a `marketplace_transfers` table (or extend `marketplace_listings` with `transferred_at`) so the future item-ownership transfer step (sub-task #3) records *who got what and when* even after the seller deletes their listing, giving us audit history. Today even with `buyer_user_id` we lose the price/mode context if the listing row is dropped.

- Reuse the new `markSoldPayload(soldAt, buyerUserId)` helper from `lib/supabase-marketplace-shapes.ts` to add a strict integration test asserting that `cloudMarkSold` PATCH bodies always include `buyer_user_id` (even when null) — guards against a future refactor accidentally dropping the column from the PATCH and silently leaving the field unset on existing rows.

- Lock the Buy / Trade button in `app/listing/[id].tsx` behind the same `friendsOnly` rule that already gates the Message-owner CTA, so a non-friend viewer cannot claim a listing whose owner has restricted who can interact with them. Today the claim button renders for any non-self viewer; the friends-only branch already returns a placeholder for messaging — we should reuse it for claiming too.

- Send an automatic chat message to the seller via `ensureChatWith` + `appendMessage` immediately after a successful claim, using a templated body like "Bought your listing for {price}". This closes the loop for the seller without forcing the buyer to open a separate chat thread; the existing chat infrastructure already supports it.

- Add a system-toast on successful claim using the existing `ToastProvider` to mirror the `marketplaceListingCreated` / `premiumActivated` toast pattern. Today the only feedback after claiming is the listing flipping to "Sold" — a transient toast keyed on the `markListingSold` callback's success would feel more responsive on slow networks.

- Persist a `claimingListingId` flag in `MarketplaceContext` instead of the local `claiming` state in `app/listing/[id].tsx`, so navigating away and back during a slow PATCH doesn't reset the spinner. The context already centralises listings — it's the natural home for in-flight claims.

- Migrate the `Platform.OS === "web"` branch in `handleClaimPress` to a shared `lib/confirm-dialog.ts` helper so other screens (delete-collection, cancel-premium, leave-friend) can use the same web-vs-native confirmation primitive instead of each duplicating the `window.confirm` / `Alert.alert` pair.

- Have the seller's device react to a "their listing was claimed" realtime event by offering an "Archive item" or "Delete from my collection" prompt. Today the buyer pulls a snapshot into their collection but the seller still keeps the original — useful for collectors who want a sale audit trail, confusing for those who expect the item to "leave" their collection.

- Migrate the `transferItemToBuyer` Acquired-collection ID generator from `${userId}-acquired-marketplace` into a `userScopedCollectionId(userId, "acquired-marketplace")` helper in `lib/collections-helpers.ts` so future system-managed collections (e.g. "Wishlist", "Trash", "Premium-only") share the same naming pattern.

- Persist the `transferItemToBuyer` invocation in a local "transfer history" log keyed by `${listingId}-${createdAt}` so a buyer can see every marketplace acquisition with its origin listing, even after the original listing is deleted upstream — opens the door to a proper "Purchases" tab (sub-task #4).

- Stop hard-coding `acquiredFrom: t("marketplaceTitle")` in `performClaim` and instead store the seller's `@username` (or fallback `displayName`) so the buyer's Acquired collection records *who* the item came from. Provenance is the most useful field for collectors auditing their own catalogue.

- Add a unit test in `__tests__/transfer-item-to-buyer.test.ts` that drives the `transferItemToBuyer` reducer (extract the inner update logic from the closure into a pure helper if needed) so we can assert: (a) idempotent re-claims don't duplicate items, (b) the Acquired collection's `coverPhoto` updates on first transfer, (c) the new item carries `condition` / `tags` from the snapshot.

- Add a "Sales" subsection on the marketplace screen mirroring "My purchases" — it would render `myListings` filtered by `soldAt !== null`. Together they form a complete personal trading audit ("what I bought" / "what I sold") without leaving the marketplace tab.

- Render the buyer's avatar/displayName chip on each My-purchases card by passing `buyer = getProfileById(listing.buyerUserId)` instead of `owner = getProfileById(listing.ownerUserId)`. Today the purchases list reuses `ListingCard` which shows the seller's name — useful, but for purchases the *seller* is the relevant counterparty so reverting to a "From @seller" treatment would be consistent.

- Make `purchasesForUser` accept an optional `since: Date` argument so the future profile screen can show "purchases this month" without re-filtering on the caller side. The helper already sorts newest-first, so the cap-by-date is a single `.filter` extension.

- Replace the inline `{ ...styles.cardWrap, flexBasis: \`${100/columns}%\` }` spread in the new purchases section with a memoised wrapper component (or extract `<ListingGrid>` from the existing active-listings render). Today the markup is duplicated character-for-character which makes future grid tweaks (gap, alignment) error-prone.

- Persist a "Mark as received" button on each purchase card so the buyer can confirm they actually got the item shipped. That bit (an `arrivedAt` timestamp) would unlock dispute handling and let sellers see "delivery confirmed" in their own listings view.

- Extend the new private-collection premium gate to also block *editing* an existing collection's visibility from public→private when the user isn't premium. Today the gate only fires on creation; a non-premium user could conceivably keep an old private collection alive after letting their subscription lapse without the flow re-prompting upgrade.

- Surface a "Locked premium feature" upsell sheet (instead of the toast) when a non-premium user taps the Private chip in `app/create-collection.tsx`. The toast pattern works but is fire-and-forget; a sheet that explains the benefit + offers a one-tap "Activate premium" CTA would be more conversion-friendly.

- Migrate the `isPremium ? "private" : "public"` default-state expression to a `defaultCollectionVisibilityForUser(isPremium)` helper in `lib/premium-helpers.ts` so the rule lives next to other premium-derived defaults (listing cap, premium badge, etc.) and can be unit-tested independently of the React component.

- Add a structural test to `__tests__/i18n-translations.test.ts` that asserts every language map either declares `visibilityPrivatePremiumOnly` directly or inherits it via the `...en` spread — a regression here would silently render the English string for non-English users on the private-locked toast.

- Hide the `visibilityChip` for "private" entirely on free accounts (instead of locking it) once we have analytics confirming nobody clicks the locked chip — a cleaner flow at the cost of a less discoverable upgrade prompt.
