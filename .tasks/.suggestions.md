## Upgrade suggestions

- **SPA fallback regression guard** — Add a CI step that runs `npm run build` then asserts (a) `dist/404.html` contains `location.replace`, (b) `dist/index.html` contains `data-spa-restore`. This catches a future contributor accidentally reverting the post-build script (e.g. swapping back to `cp dist/index.html dist/404.html`) before the change reaches production iOS users.

- **Service worker for instant deep-link load** — Even with the sessionStorage redirect, the user sees a flash of the redirect page on iPhone. Registering a service worker (Workbox `NetworkFirst` + `NavigationFallback` to `/collectables/index.html`) would let cached deep links resolve to 200 directly and remove the redirect entirely on repeat visits.

- **Sentry breadcrumb for SPA URL restore** — Drop an `addBreadcrumb("spa-redirect", { from, to })` inside the restore script (gated on `__sentry_loaded__`) so Sentry sessions show exactly which deep-link URLs hit the 404 fallback. Useful for ranking which routes deserve a real static export.

- **Static export of common collection IDs** — Expo Router could pre-render the top-N most-shared collections at build time (e.g. by reading from Supabase) so their URLs return real 200s without the redirect dance. Reduces flicker and helps SEO for public collections.

- **Migrate baseUrl to a single source** — `app.json` declares `expo.experiments.baseUrl`, `lib/url-helpers.ts` infers it at runtime, and `scripts/build-spa-fallback.ts` re-reads it from app.json. A single `lib/base-url.ts` exporting a const + a helper that all three import would eliminate the drift risk if the deployment path ever changes (e.g. moving off the `1337ninja1337.github.io/collectables` prefix).

## Chat DB follow-ups (from the 2026-05-15 integrity audit)

- **Idempotent message resend** — The offline `flushPending` loop in `chat-context.tsx` re-POSTs queued messages with the same client `id`. If the first attempt actually inserted the row but the network dropped the response, the retry hits the `chat_messages` PK and 409s forever (the entry never leaves the pending queue). Add an UPSERT-on-conflict-do-nothing path (e.g. a `messages_upsert` RPC or `Prefer: resolution=ignore-duplicates` with a representation echo) so at-least-once delivery becomes effectively-once without re-enabling row UPDATE.

- **Older-page pagination ("load earlier")** — `fetchMessagesUrl` now caps the initial load at `DEFAULT_CHAT_PAGE_SIZE` (200) most-recent rows. A new device opening a long-lived chat silently can't scroll past message 200. Add a keyset-paginated `fetchOlderMessages(chatId, beforeCreatedAt)` (`created_at=lt.<cursor>&order=created_at.desc&limit=N`) and a "Load earlier messages" affordance at the top of `app/chat/[id].tsx`.

- **Server-owned timestamps & ordering** — `messageToInsertPayload` still forwards a client `created_at` for pending flushes, so a malicious or clock-skewed client can backdate messages and reorder a conversation. Consider a `received_at timestamptz default now()` server column used for ordering, keeping the client `created_at` only as display metadata.

- **Soft-delete / unsend** — Messages are intentionally immutable (no UPDATE/DELETE RLS). Real chat UX needs an "unsend within N minutes". Add a `deleted_at` column + a narrow UPDATE policy (`auth.uid() = from_user_id AND created_at > now() - interval '5 min'`) and render tombstones client-side instead of deleting rows.

- **`chat_reads` realtime sync** — `chat_reads` is fetched once on mount; a read on device A doesn't clear the unread badge on device B until a reload. Add `chat_reads` to the `supabase_realtime` publication and subscribe (filtered to `user_id=eq.me`) so the unread count is consistent across sessions, mirroring the `subscribeToInbox` pattern.
