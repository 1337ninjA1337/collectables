# Upgrade Suggestions

- Cache the lazy-imported Sentry SDK module under a top-level promise so a second `initSentry()` call (e.g. on auth state change) doesn't risk a second native bridge handshake. Today the second call returns immediately on the `initialised` guard, but a cleaner pattern is `let pending: Promise<void> | null = null` so the caller can `await` the in-flight promise instead of getting a no-op.

- Add an `addBreadcrumb` overload that accepts a `level: "info" | "warning" | "error"` parameter so non-info breadcrumbs (e.g. "user denied photo permission") render with the right severity in the Sentry timeline. Today every breadcrumb is hard-coded to `"info"`.

- Replace the `console.warn` on loader-failure with a one-shot `console.error` plus a stored `lastInitError` so the future `__tests__/sentry-doctor.ts` script (already in suggestions) can surface "init failed" rather than silently masking the cause.

- Promote the `__resetSentryForTests` invocation into the `node:test` global `beforeEach` via a `__tests__/test-globals.ts` setup file, so any future direct test of the Sentry path starts with a clean cached SDK without each suite duplicating the call.

- Type the `captureException(error)` argument as `unknown` (already done) but additionally narrow `context` to a `CaptureContext` union (`{ scope?: string; extra?: Record<string, unknown> }`) so call sites get IntelliSense on the standard fields. Today the `context` parameter is loosely typed and risks drift between call sites.

- Validate DSN format inside `resolveSentryConfig` by asserting it matches the canonical Sentry DSN pattern (`https?://[^@]+@[^/]+/\d+`). Today an obviously-malformed DSN (e.g. a Slack webhook pasted into the secret) silently flips `enabled` to `true` and Sentry init fails at runtime; a regex guard would surface the misconfig before the SDK ever initialises and could log a one-time warning via `console.error`.

- Add a `tracesSampleRate` field to `SentryConfig` driven by `EXPO_PUBLIC_SENTRY_TRACES_SAMPLE_RATE` (default `0.1`, validated to be in `[0, 1]`) so production performance sampling can be tuned without a redeploy. The Crash #3 init module already needs this value; threading it through the config helper now means a single source of truth.

- Co-locate the `sentryConfig` memoisation pattern with `cloudinaryConfig` in a new `lib/runtime-config.ts` aggregator that re-exports both. Today every module that needs *any* config imports its own helper; a single aggregator file keeps the env-validation surface small and makes it obvious where to add the next env-driven knob (e.g. PostHog).

- Add a `__tests__/runtime-config-parity.test.ts` test that scans `lib/*-config.ts` for the env vars they read and asserts every name appears in `README-DEPLOY.md`. Today drift between the two is only caught by humans reading both files; a 30-line scan ties the two together and refuses to add a new env var without documentation.

- Promote `normaliseEnvironment` into a shared `lib/env.ts` helper alongside the future `resolveNumericEnv` (already pending in `.tasks.md`). Both follow the same "parse, validate, fallback" shape â€” extracting them removes the per-config-helper duplication once the third one (PostHog) lands.

- Codify the missing iOS `infoPlist` block from `APPSTORE-SUBMISSION.md` (NSPhotoLibraryUsageDescription, NSCameraUsageDescription, ITSAppUsesNonExemptEncryption, CFBundleLocalizations, associatedDomains) directly into `app.json` so the next `eas build --platform ios` produces a TestFlight-ready binary without a separate config pass. Today the strings live in the markdown checklist; landing them in `app.json` removes the manual step and lets a structural test enforce that every supported language stays in `CFBundleLocalizations`.

- Add a `scripts/check-appstore-config.ts` validator that's wired into `lint:ci` and asserts `app.json` has a non-empty `expo.ios.bundleIdentifier`, an `expo.ios.icon` path that resolves on disk, and the four required `infoPlist` keys. Today a regression that drops `bundleIdentifier` would only be caught at `eas submit` time; a 30-line pre-flight check catches it locally and in PR CI.

- Generate the App Store Connect privacy nutrition labels from a typed `lib/privacy-manifest.ts` module so the `APPSTORE-SUBMISSION.md` "App Privacy" table and the future iOS 17 `PrivacyInfo.xcprivacy` file stay in sync. Today the table is hand-maintained; a single source of truth that emits both Markdown and the Apple-required plist closes the drift gap before iOS 18+ enforcement.

- Add an `eas-secrets-parity.test.ts` structural test that reads `.github/workflows/deploy.yml` + `APPSTORE-SUBMISSION.md` + `eas.json` and asserts every `EXPO_PUBLIC_*` referenced in one is referenced in the other two. Today the GitHub Actions secrets and the EAS secrets are documented separately â€” a parity test would catch a future env var added to web deploys but missed in iOS builds.

- Wire a CI matrix job that runs `eas build --platform ios --profile preview --non-interactive --no-wait` on `main` so we always have a fresh internal-testflight build available without a manual trigger. Today the iOS build is purely operator-initiated; an automated build-on-merge job (paid EAS minutes permitting) keeps the App Store track aligned with the GitHub Pages track and surfaces native-only regressions (Hermes / Reanimated worklet failures) within minutes of merge instead of at submission time.

- Pin `@sentry/react-native` to an exact version with `npm ls @sentry/react-native` lockfile asserted in `lint:ci` so a `npm install` in a fresh clone doesn't bump the SDK and silently break the upcoming Sentry init module. Tilde range (`~7.5.0`) is fine but a CI guard against unexpected major-version drift saves a debug session when Sentry's RN SDK rewrites its config shape (it's done so 3Ă— since 5.0).

- Track-and-document a `scripts/sentry-doctor.ts` self-check script that prints whether the DSN is reachable, whether sourcemaps for the latest deploy exist, and whether the most recent release was tagged. Wire it into a `npm run sentry:check` script so engineers can validate the wiring without poking the Sentry UI. Cheap to write â€” uses `fetch` against the Sentry public API.

- Promote the `@sentry/react-native` peer dep into the existing `lib/cloudinary-config.ts` env-validation pattern by adding a `scripts/check-required-env.ts` that fails CI when a `EXPO_PUBLIC_*` documented in `README-DEPLOY.md` is missing from `eas.json`'s implicit secret references. Avoids adding a new env var to one workflow and forgetting the other.

- Pre-load the Sentry RN SDK lazily via `import("@sentry/react-native")` only when `enabled === true`, so dev/test bundles never pay the ~120 KB native bridge cost. The Crash #2 / Crash #3 tasks already imply lazy import; codify it as a hard requirement in `lib/sentry.ts` with a structural test that fails on any top-level `import * as Sentry from "@sentry/react-native"` outside the lazy path.

- Add a `.env.example` despite it currently being gitignored â€” flip line 35 of `.gitignore` to track it and seed it with placeholder values for every `EXPO_PUBLIC_*` listed in `README-DEPLOY.md`. Today new engineers have to read README-DEPLOY.md to know what to set; a tracked `.env.example` is the canonical Expo / React Native onboarding pattern and pairs naturally with the existing `cloudinaryConfig` resolver tests.
