## Upgrade suggestions

- **SPA fallback regression guard** — Add a CI step that runs `npm run build` then asserts (a) `dist/404.html` contains `location.replace`, (b) `dist/index.html` contains `data-spa-restore`. This catches a future contributor accidentally reverting the post-build script (e.g. swapping back to `cp dist/index.html dist/404.html`) before the change reaches production iOS users.

- **Service worker for instant deep-link load** — Even with the sessionStorage redirect, the user sees a flash of the redirect page on iPhone. Registering a service worker (Workbox `NetworkFirst` + `NavigationFallback` to `/collectables/index.html`) would let cached deep links resolve to 200 directly and remove the redirect entirely on repeat visits.

- **Sentry breadcrumb for SPA URL restore** — Drop an `addBreadcrumb("spa-redirect", { from, to })` inside the restore script (gated on `__sentry_loaded__`) so Sentry sessions show exactly which deep-link URLs hit the 404 fallback. Useful for ranking which routes deserve a real static export.

- **Static export of common collection IDs** — Expo Router could pre-render the top-N most-shared collections at build time (e.g. by reading from Supabase) so their URLs return real 200s without the redirect dance. Reduces flicker and helps SEO for public collections.

- **Migrate baseUrl to a single source** — `app.json` declares `expo.experiments.baseUrl`, `lib/url-helpers.ts` infers it at runtime, and `scripts/build-spa-fallback.ts` re-reads it from app.json. A single `lib/base-url.ts` exporting a const + a helper that all three import would eliminate the drift risk if the deployment path ever changes (e.g. moving off the `1337ninja1337.github.io/collectables` prefix).

## Chat DB hardening (from the chat-persistence audit)

- **Preserve conversation history on account deletion** — `chat_messages.from_user_id`/`to_user_id` are `ON DELETE CASCADE`, so when one participant deletes their account every message they sent vanishes from the *other* user's history too. Switch to `ON DELETE SET NULL` (nullable columns) plus a `deleted_user` sentinel in the UI, or snapshot a denormalised `from_display` so the surviving participant keeps a readable thread. This is a DB migration → must go through `MANUAL-TASKS.md`.

- **Reject client-backdated timestamps** — the offline-flush path lets the client supply `created_at`. That is correct for preserving compose-time, but a crafted client could backdate a message to the top of a thread. Add a Postgres `CHECK (created_at <= now() + interval '5 minutes')` (clock-skew tolerant) or a trigger that clamps `created_at` to `least(created_at, now())` so ordering can't be gamed while still honouring legitimate offline sends.

- **Server-side `edited_at` + soft-delete column** — messages are currently immutable (no UPDATE/DELETE policy), which blocks "delete for everyone" / edit features users expect. Add `edited_at timestamptz` and `deleted_at timestamptz` plus a narrow UPDATE RLS policy (`auth.uid() = from_user_id`, only those two columns mutable via a trigger) so the table can grow into edit/unsend without a breaking migration later.

- **Pagination + retention for `fetchMessagesForChat`** — it does `select=*&order=created_at.asc` with no limit, so a long-lived friendship loads the entire history into memory on every mount. Add keyset pagination (`created_at < cursor` + `limit`) to `fetchMessagesUrl` and a scheduled `delete from chat_messages where created_at < now() - interval '1 year'` retention job to bound table growth and client memory.

- **Outbound send queue with backoff + ordering guarantee** — `flushPending` iterates `Object.entries` and stops a chat on the first failure, with no retry backoff and no cross-chat ordering. Extract a small `lib/chat-outbox.ts` with per-chat FIFO ordering, exponential backoff on transient failures, and a permanent-failure bucket (e.g. RLS reject because the friendship was revoked) so a single un-sendable message can't block every later message in that chat forever.
