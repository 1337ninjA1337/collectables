## Upgrade suggestions

- Add an integration test that boots `scripts/serve-dist.js` against the built `dist/`, fetches `/collectables/collection/test-id` and asserts the response contains the redirect-script marker. The unit tests cover the templating, but a smoke test would catch a future regression where the build pipeline forgets to call `scripts/build-spa-fallback.js`. Pair with a `tsx --test` HTTP helper.

- Mirror the GitHub Pages 404 semantics in `scripts/serve-dist.js` for local previews — when a file is missing, serve `dist/404.html` with **status 404** (instead of the current 200 fallback to `+not-found.html`). The redirect script will then exercise locally too, so engineers can verify the rafgraph round-trip without pushing to GitHub.

- Add a `dist/+not-found.html` redirect-aware wrapper so the expo-router-generated 404 page also runs the path-restore script (currently only `index.html` is patched). Today a user who visits a non-existent route after the redirect lands on `+not-found.html` without the URL restoration, leaving the URL bar at `/?/<path>` instead of the original. Extending `injectRestoreScript` to also patch `+not-found.html` (if present) closes that visual gap.

- Tighten the path-restore decoder to URL-decode each segment with `decodeURIComponent` after splitting on `&` so a Cyrillic collection name (e.g. `привет-1776966433160`) survives the round-trip. The current implementation forwards the raw bytes; if GitHub Pages percent-encodes the path, the bundle sees double-encoded segments. A one-line `decodeURIComponent(segment)` per part fixes it without affecting the ASCII happy path.

- Add a structural test that asserts `dist/index.html` and `dist/404.html` never reference each other's marker (no `__collectables_spa_path_restore__` in 404.html, no `pathSegmentsToKeep` in index.html) so a future contributor can't accidentally cross-wire the two templates. Pairs with the existing `injectRestoreScript`/`buildRedirectScript` purity guards.
