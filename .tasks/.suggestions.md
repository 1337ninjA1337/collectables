## Upgrade suggestions

### Chat DB (2026-05-16)

- **Cursor pagination for `fetchMessagesForChat`** ‚Äî it currently `select=*&order=created_at.asc` with no `limit`, so a long-lived conversation pulls the entire history into memory on every hydrate. Add `&limit=50` + a `created_at` cursor (or PostgREST `Range` header) and load older pages on scroll-up. Bounds memory and free-tier egress.

- **Realtime reconnect backfill** ‚Äî while the WebSocket is down, `subscribeToInbox` misses INSERTs and only `refreshFromCloud` (mount-time) closes the gap. On `SUBSCRIBED` after a drop, fetch messages `created_at=gt.<last-known>` per active chat so no message is silently lost during a flaky connection.

- **First-class conversations table** ‚Äî replace the free-text `chat_id` with a `conversations(id uuid, user_a uuid, user_b uuid, UNIQUE(user_a,user_b))` table and a `chat_messages.conversation_id uuid REFERENCES conversations(id)`. A real FK + unique participant pair is stronger than the derived-string CHECK and unlocks per-conversation metadata (muted, archived, last_message_at) without re-deriving from messages.

- **Functional idempotency test with a mocked fetcher** ‚Äî the new ignore-duplicates path is only covered structurally. Add a `__tests__/supabase-chat-idempotency.test.ts` that injects a fake `fetcher` returning `200 []` and asserts `sendMessage` resolves to the synthesized message (not `null`), plus a fake returning a populated row for the happy path.

- **`chat_messages.client_id` + server uuid split** ‚Äî keep the server `id uuid DEFAULT gen_random_uuid()` authoritative and add a separate indexed `client_id uuid` (the composition-time key) with a UNIQUE constraint for idempotency. Decouples "who owns the primary key" from "idempotency key", which is the canonical chat-storage pattern and avoids ever trusting a client-supplied PK.

- **SPA fallback regression guard** ‚Äî Add a CI step that runs `npm run build` then asserts (a) `dist/404.html` contains `location.replace`, (b) `dist/index.html` contains `data-spa-restore`. This catches a future contributor accidentally reverting the post-build script (e.g. swapping back to `cp dist/index.html dist/404.html`) before the change reaches production iOS users.

- **Service worker for instant deep-link load** ‚Äî Even with the sessionStorage redirect, the user sees a flash of the redirect page on iPhone. Registering a service worker (Workbox `NetworkFirst` + `NavigationFallback` to `/collectables/index.html`) would let cached deep links resolve to 200 directly and remove the redirect entirely on repeat visits.

- **Sentry breadcrumb for SPA URL restore** ‚Äî Drop an `addBreadcrumb("spa-redirect", { from, to })` inside the restore script (gated on `__sentry_loaded__`) so Sentry sessions show exactly which deep-link URLs hit the 404 fallback. Useful for ranking which routes deserve a real static export.

- **Static export of common collection IDs** ‚Äî Expo Router could pre-render the top-N most-shared collections at build time (e.g. by reading from Supabase) so their URLs return real 200s without the redirect dance. Reduces flicker and helps SEO for public collections.

- **Migrate baseUrl to a single source** ‚Äî `app.json` declares `expo.experiments.baseUrl`, `lib/url-helpers.ts` infers it at runtime, and `scripts/build-spa-fallback.ts` re-reads it from app.json. A single `lib/base-url.ts` exporting a const + a helper that all three import would eliminate the drift risk if the deployment path ever changes (e.g. moving off the `1337ninja1337.github.io/collectables` prefix).

### Power BI template (2026-05-17)

- **Regenerate-the-.pbit CI guard** ‚Äî the `powerbi-template` parity test fails if `Collectables-Starter.pbit` drifts from `queries.m`/`measures.dax`, but only if a contributor runs the suite. Add a CI step that runs `npm run build:powerbi` then `git diff --exit-code docs/powerbi/Collectables-Starter.pbit` so a stale binary blocks the PR instead of shipping silently.

- **Deflate the `.pbit` payload** ‚Äî `zipStore` uses STORE (no compression); the `DataModelSchema` JSON is ~11 KB of repetitive text. Switching the OPC entries to DEFLATE (`zlib.deflateRawSync`, method 8, deterministic at a fixed level) would shrink the committed artifact ~4√ó. Keep STORE as a fallback flag in case a Power BI version is picky about compressed OPC parts.

- **`.pbix` companion with a pre-built report** ‚Äî the template ships an empty report page. A sibling `scripts/build-powerbi-report.ts` that injects a `visualContainers` block (DAU line chart + funnel column chart bound to the measures) into `Report/Layout` would give users a populated dashboard on first open, not a blank canvas.

- **Schema-from-migration generator** ‚Äî `buildDataModelSchema` hardcodes the five `analytics_events` columns. Parsing them from `supabase/migrations/20260508_analytics_events.sql` (single source of truth) would keep the model in lockstep if the table gains a column, mirroring how `queries.m`/`measures.dax` already drive the rest of the template.

- **Round-trip validation via a pure unzip** ‚Äî tests assert the OPC structure with a hand-rolled reader. Extracting that reader into `lib/powerbi-template.ts` as an exported `readZipStore()` and asserting `readZipStore(buildPbit(src))` reproduces every part byte-for-byte would lock the writer/reader pair and let other build scripts reuse the deterministic ZIP primitive.

### Analytics RLS hardening (2026-05-17)

- **pgTAP-ify the leakage test** ‚Äî `analytics_events_rls.sql` uses bespoke `RAISE EXCEPTION` assertions. Re-expressing it with `pgtap` (`SELECT throws_ok(...)`, `SELECT is(...)`) and running it under `supabase test db` in CI would turn the out-of-band manual checklist into an automated gate that fails the pipeline on a regression instead of relying on someone remembering to paste SQL.

- **Generalise to every RLS-default-deny table** ‚Äî the same anon/authenticated leak check applies to any future "service-role only" table. A parameterised `supabase/tests/_rls_deny.sql` (table name as a `\set` variable) plus a manifest list would scale the guarantee instead of copy-pasting the probe block per table.

- **Negative-grant assertion** ‚Äî the test proves SELECT is denied today, but a future `GRANT SELECT ‚Ä¶ TO authenticated` would re-open it while the test still passes (the grant check isn't asserted). Add an assertion over `information_schema.role_table_grants` that **no** privilege exists for anon/authenticated on `analytics_events`, catching the grant before a row is even probed.

- **INSERT/UPDATE/DELETE coverage** ‚Äî #16 only checks SELECT. The same `REVOKE ALL` should block writes too; extend the probe to assert anon/authenticated `INSERT`/`UPDATE`/`DELETE` also raise `insufficient_privilege`, so a future write policy can't silently let end users forge analytics events.

- **Edge-Function smoke in the same script** ‚Äî append an optional, commented-out block that calls the `analytics-mirror` insert path via `SECURITY DEFINER` to prove the service-role write path still works end-to-end (not just raw `SELECT count(*)`), tying the RLS posture test to the actual ingestion route.
