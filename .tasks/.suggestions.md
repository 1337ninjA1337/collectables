## Upgrade suggestions

- **Hoist the inline `{ ...styles.x, color: theme.y }` spreads on `app/index.tsx` into a `useMemo(() => StyleSheet.create({...}), [theme])`** — the home redesign (PR5) now sprinkles ~15 inline object spreads in the JSX, each allocating a fresh style object on every render and defeating `StyleSheet`'s registry caching. A memoized per-theme stylesheet (the React Native canonical pattern) would cut the allocations, keep colors co-located with layout, and read cleaner than the spread soup before PRs 6–8 repeat the same pattern on every screen.

- **Delete the dead styles left in `app/index.tsx` (`socialList`, `personCard`, `personName`, `personMeta`, `emptyCard`, `emptyCardText`, `phoneActionsRow`)** — these `StyleSheet` entries have zero `styles.<name>` references (verified by grep) and only survive because they still anchor `CARD_BG`/`BORDER`/`MUTED`/`TEXT_DARK` imports. Removing them and the now-orphaned token imports would shrink the file ~50 LOC; do it alongside a token-presence test rewrite so the design-tokens guard doesn't break.

- **Add a `theme.shadow` slot to `useAppTheme()` instead of spreading raw `SHADOW_SOFT` inline** — PR5 spreads `...SHADOW_SOFT` directly on three home banners/cards, but `SHADOW_SOFT` is tuned for the light page and reads as nothing in dark mode. Exposing `theme.shadow` (soft on light, hairline/none on dark) would let every redesigned screen get scheme-correct elevation from one read, and a node test could assert `DARK.shadow !== LIGHT.shadow`.

- **Snapshot-test the home screen under both color schemes** — the only home coverage is a regex/structural check that the right token names appear in the source; it can't catch a contrast regression (e.g. `theme.text` on `theme.card` going invisible). A lightweight react-test-renderer mount with `useColorScheme` mocked to `"light"` and `"dark"` would assert the resolved color props actually differ, hardening the redesign before PRs 6–8 fan it out.

- **Make `SHADOW_SOFT` opt-in per platform via `Platform.select`** — on web, RN's `shadow*` props map to `box-shadow` but `elevation` is dropped, while on Android only `elevation` paints; the current single object ships both, so web cards get a slightly different shadow than native. Splitting `SHADOW_SOFT` into a `Platform.select({ web, default })` shape would make the new home/banner elevation pixel-consistent across the deployed PWA and the native builds.

- **Add a `__tests__/design-tokens-frozen-map-parity.test.ts` that asserts every top-level numeric/string export in `lib/design-tokens.ts` is registered in the frozen `designTokens` object** — PR1 had to hand-add the five new geometry names to the map, an easy step to forget. A regex scan of `export const FOO = ...` lines vs. the `Object.freeze({...})` keys would fail-closed the next time someone appends a token but skips the map (which silently breaks `DesignToken` autocomplete + the tokens-preview route).

- **Promote `useAppTheme()` selection into a tiny `pickTheme(scheme)` pure helper** — today the LIGHT/DARK ternary lives inside the hook, so it can't be unit-tested without a React renderer (the current test is purely structural/regex). Extracting `export function pickTheme(scheme: ColorSchemeName): AppTheme` would let a node test assert `pickTheme("dark").isDark === true` and `pickTheme(null) === LIGHT` directly, hardening the mapping before PRs 4–8 consume it everywhere.

- **Add a `SHADOW_SOFT_DARK` companion token** — `SHADOW_SOFT` hard-codes `shadowColor: HERO_DARK`, which "disappears against the dark-mode page surface" by design, but on elevated dark cards a subtle lighter-tinted shadow (or a hairline top-border) reads better than no elevation at all. A paired token plus a `theme.isDark ? SHADOW_SOFT_DARK : SHADOW_SOFT` convention would keep depth perception in dark mode without inline values.

- **Wire `FONT_DISPLAY_EDITORIAL` into the `app/_dev/tokens.tsx` preview route** — the existing dev tokens screen renders color/spacing swatches; adding a "type specimen" row that renders a sample headline in each font constant (`FONT_DISPLAY`, `FONT_DISPLAY_BOLD`, `FONT_DISPLAY_EDITORIAL`) would let reviewers eyeball the new serif chain on each platform before the redesign PRs land, closing the "verify on deployed build" loop earlier.

- **Add a structural test that fails if any `app/**` or `components/**` file still imports `FONT_DISPLAY`/`FONT_DISPLAY_BOLD` after PR8 ships** — the visual-upgrade brief migrates display titles off Syne onto `FONT_DISPLAY_EDITORIAL` but keeps the old exports for the transition. A grep-based guard (skippable until PR8) would catch a half-migrated screen and make the "Syne TTFs are now unreferenced, safe to drop" decision a deterministic check rather than a manual audit.
