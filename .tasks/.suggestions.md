## Upgrade suggestions

- **SPA fallback regression guard** — Add a CI step that runs `npm run build` then asserts (a) `dist/404.html` contains `location.replace`, (b) `dist/index.html` contains `data-spa-restore`. This catches a future contributor accidentally reverting the post-build script (e.g. swapping back to `cp dist/index.html dist/404.html`) before the change reaches production iOS users.

- **Service worker for instant deep-link load** — Even with the sessionStorage redirect, the user sees a flash of the redirect page on iPhone. Registering a service worker (Workbox `NetworkFirst` + `NavigationFallback` to `/collectables/index.html`) would let cached deep links resolve to 200 directly and remove the redirect entirely on repeat visits.

- **Sentry breadcrumb for SPA URL restore** — Drop an `addBreadcrumb("spa-redirect", { from, to })` inside the restore script (gated on `__sentry_loaded__`) so Sentry sessions show exactly which deep-link URLs hit the 404 fallback. Useful for ranking which routes deserve a real static export.

- **Static export of common collection IDs** — Expo Router could pre-render the top-N most-shared collections at build time (e.g. by reading from Supabase) so their URLs return real 200s without the redirect dance. Reduces flicker and helps SEO for public collections.

- **Migrate baseUrl to a single source** — `app.json` declares `expo.experiments.baseUrl`, `lib/url-helpers.ts` infers it at runtime, and `scripts/build-spa-fallback.ts` re-reads it from app.json. A single `lib/base-url.ts` exporting a const + a helper that all three import would eliminate the drift risk if the deployment path ever changes (e.g. moving off the `1337ninja1337.github.io/collectables` prefix).

- **Server-side message pagination** — `fetchMessagesForChat` pulls the entire thread (`select=*` ordered by `created_at,id`) on every hydrate. For a long-running friendship this grows unbounded. Add a `&limit=50` keyset window on the new `(chat_id, created_at, id)` index plus an "older messages" pager in `app/chat/[id].tsx` so the first paint is O(1) regardless of history size.

- **Chat-message delivery/read receipts** — `chat_reads` already tracks a per-user last-read timestamp; surface it in the UI as a "seen" tick on the sender's last message (compare the recipient's `chat_reads.last_read_at` against `message.createdAt`). No schema change needed — just a `fetchPeerChatRead(chatId, peerId)` selector gated by an RLS policy allowing a chat participant to read the *other* participant's read row for shared chats.

- **Soft-delete / unsend window** — Messages are immutable (no UPDATE/DELETE policy). Add a `deleted_at timestamptz` column + an UPDATE RLS policy letting only the sender set it within N minutes, and render tombstoned rows as "message deleted". This is the standard chat UX and avoids the only-option-is-clear-whole-thread limitation.

- **Idempotency-key column instead of reusing the PK** — The offline-flush path overloads the message `id` as both PK and idempotency key. A dedicated `client_msg_id text` unique-per-(from_user_id) column would let the server own a real `id`/`created_at` (closing the client-can-backdate-its-own-message gap) while still deduping retries — a cleaner separation than `resolution=ignore-duplicates` on the PK.

- **Realtime fan-out registry** — `subscribeToInbox` opens one channel per mounted screen; the chats list and an open chat both subscribe. A small shared subscriber registry on top of `getSharedRealtimeClient` (one `chat-inbox-<uid>` channel, N in-process listeners) would stop the same INSERT arriving 2–3× when multiple chat surfaces are mounted.
