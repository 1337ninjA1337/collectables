## Upgrade suggestions

### Chat DB (2026-05-16)

- **Cursor pagination for `fetchMessagesForChat`** — it currently `select=*&order=created_at.asc` with no `limit`, so a long-lived conversation pulls the entire history into memory on every hydrate. Add `&limit=50` + a `created_at` cursor (or PostgREST `Range` header) and load older pages on scroll-up. Bounds memory and free-tier egress.

- **Realtime reconnect backfill** — while the WebSocket is down, `subscribeToInbox` misses INSERTs and only `refreshFromCloud` (mount-time) closes the gap. On `SUBSCRIBED` after a drop, fetch messages `created_at=gt.<last-known>` per active chat so no message is silently lost during a flaky connection.

- **First-class conversations table** — replace the free-text `chat_id` with a `conversations(id uuid, user_a uuid, user_b uuid, UNIQUE(user_a,user_b))` table and a `chat_messages.conversation_id uuid REFERENCES conversations(id)`. A real FK + unique participant pair is stronger than the derived-string CHECK and unlocks per-conversation metadata (muted, archived, last_message_at) without re-deriving from messages.

- **Functional idempotency test with a mocked fetcher** — the new ignore-duplicates path is only covered structurally. Add a `__tests__/supabase-chat-idempotency.test.ts` that injects a fake `fetcher` returning `200 []` and asserts `sendMessage` resolves to the synthesized message (not `null`), plus a fake returning a populated row for the happy path.

- **`chat_messages.client_id` + server uuid split** — keep the server `id uuid DEFAULT gen_random_uuid()` authoritative and add a separate indexed `client_id uuid` (the composition-time key) with a UNIQUE constraint for idempotency. Decouples "who owns the primary key" from "idempotency key", which is the canonical chat-storage pattern and avoids ever trusting a client-supplied PK.

- **SPA fallback regression guard** — Add a CI step that runs `npm run build` then asserts (a) `dist/404.html` contains `location.replace`, (b) `dist/index.html` contains `data-spa-restore`. This catches a future contributor accidentally reverting the post-build script (e.g. swapping back to `cp dist/index.html dist/404.html`) before the change reaches production iOS users.

- **Service worker for instant deep-link load** — Even with the sessionStorage redirect, the user sees a flash of the redirect page on iPhone. Registering a service worker (Workbox `NetworkFirst` + `NavigationFallback` to `/collectables/index.html`) would let cached deep links resolve to 200 directly and remove the redirect entirely on repeat visits.

- **Sentry breadcrumb for SPA URL restore** — Drop an `addBreadcrumb("spa-redirect", { from, to })` inside the restore script (gated on `__sentry_loaded__`) so Sentry sessions show exactly which deep-link URLs hit the 404 fallback. Useful for ranking which routes deserve a real static export.

- **Static export of common collection IDs** — Expo Router could pre-render the top-N most-shared collections at build time (e.g. by reading from Supabase) so their URLs return real 200s without the redirect dance. Reduces flicker and helps SEO for public collections.

- **Migrate baseUrl to a single source** — `app.json` declares `expo.experiments.baseUrl`, `lib/url-helpers.ts` infers it at runtime, and `scripts/build-spa-fallback.ts` re-reads it from app.json. A single `lib/base-url.ts` exporting a const + a helper that all three import would eliminate the drift risk if the deployment path ever changes (e.g. moving off the `1337ninja1337.github.io/collectables` prefix).

- **CI guard: regenerate the .pbit and fail on drift** — Add a `build:powerbi` step to the GitHub Actions workflow that runs `npm run build:powerbi` then `git diff --exit-code docs/powerbi/Collectables-Starter.pbit`. The `powerbi-template.test.ts` parity test already catches drift, but a workflow step makes the regeneration command discoverable and stops a stale binary ever reaching `main` if someone edits `measures.dax`/`queries.m` without rebuilding.

- **Reference report layout in the .pbit** — The template currently opens to a blank canvas. Build a real `Report/Layout` with a line chart (`occurred_at` × `DAU`), two KPI cards (`ListingFunnelRate`, `PremiumConversionRate7d`), and a default theme resource package so users get a dashboard on first open. Gate it behind a `--with-report` flag so the minimal (lower open-risk) layout stays the default until it's validated in Power BI Desktop.

- **Generalise the OPC/ZIP writer** — `zipStore`/`readZipEntries`/`crc32` in `lib/powerbi-template.ts` are a generic, dependency-free deterministic ZIP. Promote them to `lib/opc-zip.ts` so future binary artefacts (e.g. an `.xlsx` data export, a bundled theme pack) reuse the same byte-stable packer instead of pulling in `jszip`/`archiver`.

- **Power BI parameter validation in the M** — The four `Supabase*` parameters are free-text. Add a guard in the partition M (`if SupabasePort = "" then error "SupabasePort is required" else ...`) and a regex check on the host so a user who clicks through the prompt with blanks gets a clear Power Query error instead of an opaque PostgreSQL connection failure.

- **`.pbit` schema-version matrix doc** — `compatibilityLevel: 1550` targets a broad Power BI range, but the report/model JSON evolves across Desktop releases. Add a short table in `docs/powerbi/README.md` mapping the Power BI Desktop versions the template was sanity-checked against (and known-incompatible ones) so users hitting an open failure immediately know to fall back to the copy-paste assets.

- **pgTAP automated RLS test** — The Analytics #16 leak check is a manual paste-into-SQL-editor checklist. Add a `supabase/tests/analytics_events_rls.sql` pgTAP test (`SET ROLE anon; SELECT throws_ok('SELECT … analytics_events', '42501')`) so `supabase test db` enforces the default-deny posture in CI instead of relying on an operator remembering to run it.

- **Reusable RLS-deny audit query** — Generalise the leak check into a single query over `pg_policies` + `information_schema.role_table_grants` that lists every `public.*` table reachable by `anon`/`authenticated`, and document it in MANUAL-TASKS.md. One run flags *any* table that regresses to readable, not just `analytics_events` — cheap defence-in-depth as more tables land.

- **CI grep guard against analytics_events policies** — Add a lint/CI step that greps `supabase/migrations/*.sql` for `CREATE POLICY .* analytics_events` (executable SQL, comment-stripped) and fails the build. The structural test already asserts this for the one migration; a directory-wide grep catches a *future* migration file that adds a permissive policy from outside `20260508_*`.

- **Edge Function service-role assertion** — `analytics-mirror` inserts with the service-role key; add a startup self-check (and a unit assertion) that the resolved key is the service-role JWT (`role: "service_role"` claim) and not the anon key, so a misconfigured secret fails loudly at deploy time instead of silently 401/permission-denying every insert against the RLS'd table.

- **Retention / PII sweep for analytics_events** — The store is long-tail by design; add a documented `DELETE FROM analytics_events WHERE occurred_at < now() - interval '13 months'` cron (pg_cron) plus a note in the privacy doc, so the unbounded event history honours a stated retention window for GDPR — pairs naturally with the existing default-deny posture.

### Analytics rate limiter (2026-05-19)

- **Surface a dropped-event counter** — `rateLimitAllow()` silently returns false when the window is full, so an over-firing screen looks identical to a quiet one. Track a module-level `droppedEventCount` and a `getAnalyticsRateLimitStatus()` snapshot (mirrors `getSentryStatus()`); a future admin/devtools probe can then tell "no events" apart from "rate-limited", which is the difference between a dead instrumentation path and a hot loop.

- **Per-event-name sub-limits** — A single global 200/min bucket lets one chatty event (e.g. a mis-placed `item_photo_attached` in a render) starve rare high-value events like `premium_activated`. A small `Map<AnalyticsEventName, number[]>` with a low per-name cap (e.g. 30/min) plus the existing global ceiling would protect conversion-critical events from a noisy neighbour.

- **Coalesce dropped events into one summary capture** — Instead of fully discarding everything past the cap, emit a single `analytics_rate_limited` event once per window carrying `{ dropped, topEventName }`. Keeps the free-tier bounded while preserving the signal that *something* is looping — currently the loop is invisible in PostHog itself.

- **Shared rate-limiter primitive** — `lib/sentry.ts` and `lib/analytics.ts` now carry byte-identical `RATE_LIMIT_WINDOW_MS` / `recentEvents` / `rateLimitAllow` blocks. Extract a `createSlidingWindowLimiter(max, windowMs)` factory into `lib/rate-limit.ts` (returning `{ allow, reset }`) so both wrappers, plus future ones (chat send, marketplace claim), share one tested implementation instead of three copies drifting apart.

- **Make the cap env-tunable** — Add `EXPO_PUBLIC_ANALYTICS_MAX_EVENTS_PER_MIN` resolved via the proposed `resolveNumericEnv` helper so incident response can throttle analytics to near-zero (or raise it for a load test) without a redeploy — same operational lever the `EXPO_PUBLIC_REALTIME_DISABLED` kill-switch suggestion proposes for WebSockets.

### Diagnostics Do-Not-Track (2026-05-19)

- **Surface a "respecting your browser's Do-Not-Track" line in settings** — When `readDoNotTrack()` is true and the user has made no explicit choice, the toggle silently sits OFF with no explanation. Add an i18n hint under the toggle (`t("diagnosticsDntHint")`) shown only in that state so an EU user understands *why* it defaulted off and that flipping it on is an explicit consent override — clearer GDPR UX and removes a "why is this off?" support question.

- **Global Privacy Control (GPC) signal** — GPC (`navigator.globalPrivacyControl === true`) is the modern successor to DNT and is legally recognised under CCPA/CPRA. Extend `readDoNotTrack()` (or add a sibling `readPrivacySignal()`) to also honour GPC so the opt-out default tracks the standard regulators now point to, not just the deprecated DNT header.

- **Re-evaluate DNT default on web focus** — DNT is only read once on provider mount. If a user changes their browser DNT setting mid-session (or in another tab) the app keeps the stale default until reload. A `visibilitychange`/`focus` listener (web-only) that re-runs `resolveDiagnosticsEnabled` when there's still no explicit stored choice would keep the signal honoured live.

- **Persist a `source` alongside the stored choice** — `{ enabled }` doesn't record whether the value came from an explicit user toggle, a DNT default, or a migration. Storing `{ enabled, source: "user" | "dnt-default" }` lets a future migration safely re-derive defaults (e.g. if the DNT policy changes) without clobbering real user consent, and feeds an analytics property on how many installs are DNT-defaulted.

- **`__tests__/diagnostics-provider-wiring.test.ts`** — The pure helpers are well covered but the `DiagnosticsProvider` effect (hydrate → `setSentryOptOut`/`setAnalyticsOptOut`/`setClarityOptOut` + conditional `init*`) is only exercised indirectly. Add a structural test asserting the provider calls `resolveDiagnosticsEnabled(raw, readDoNotTrack())` on hydrate and that `setDiagnosticsEnabled(false)` calls all three `shutdown*` paths — guards the GDPR contract against a refactor that drops a sub-processor from the fan-out.

### Privacy policy & sub-processors (2026-05-19)

- **Generate PRIVACY.md from the App Privacy table** — `APPSTORE-SUBMISSION.md` §6 now holds the canonical table + sub-processor paragraphs, but the actual public `PRIVACY.md` (the URL App Review checks) is described, not generated. Add a `scripts/build-privacy-md.ts` that extracts the table + paragraphs into a standalone `PRIVACY.md` deployed to GitHub Pages, plus a parity test, so the published policy can never drift from the App Store Connect declaration.

- **Localise the public privacy paragraphs** — The app ships 6 languages (`ru/en/be/pl/de/es`) but the privacy paragraphs are English-only. EU (GDPR) and Apple guidelines expect the policy in the app's supported languages; wire the paragraphs through the existing `t()` i18n catalogue (or a `docs/privacy/<lang>.md` set) so a German/Polish user gets a policy they can read.

- **CI guard: SDK ↔ privacy-table parity** — Add a test that greps `package.json` + `lib/clarity.ts` for telemetry SDKs (`@sentry/*`, `posthog-react-native`, `clarity.ms`) and asserts each has a matching row in the `APPSTORE-SUBMISSION.md` App Privacy table. Today adding a new analytics SDK wouldn't fail any check until App Review rejects the build — a parity test shifts that left.

- **Document the PostHog/Clarity data-retention windows** — The Sentry paragraph states a concrete 90-day retention; the new PostHog and Clarity paragraphs don't. GDPR Art. 13(2)(a) requires the storage period. Add the configured/contractual retention for each (PostHog project setting, Clarity's 30-day default) so the disclosure is complete.

- **Surface the active sub-processor list in-app** — Add a read-only "Data & privacy" sub-screen under Settings that renders the same sub-processor list (Sentry/PostHog/Clarity) and a link to the policy. Several App Review rejections stem from the policy being hard to reach; an in-app surface that mirrors `APPSTORE-SUBMISSION.md` closes that and gives users one place to see who processes their data.

### Analytics test-coverage audit (2026-05-19)

- **Functional rate-limiter window-expiry test** — `analytics-rate-limit.test.ts` covers the cap and the reset helper but not the *sliding* behaviour. Inject a `now` clock (or temporarily monkeypatch `Date.now`) and assert that events allowed >60s ago no longer count, so a regression that turns the sliding window into a fixed one (or drops the `recentEvents.filter` cutoff) is caught.

- **CI guard: every test file is actually picked up by the glob** — `npm test` runs `tsx --test __tests__/*.test.ts`; a file mis-named `*.tests.ts` or in a sub-dir is silently skipped (it would still pass CI). Add a meta-test that globs `__tests__/**` and asserts every `*.test.ts` it finds is also matched by the exact `__tests__/*.test.ts` pattern the script uses.

- **Assert AnalyticsProvider mounts under DiagnosticsProvider** — wiring tests check AnalyticsProvider sits under Auth/I18n/Premium, but not that it mounts *inside* `<DiagnosticsProvider>`. If it ever rendered outside, `identifyUser` could fire before the opt-out flag hydrates. A one-line index-order assertion closes that ordering gap.

- **Snapshot the `AnalyticsEventName` ↔ `ANALYTICS_EVENTS` parity at the type level** — `analytics-events.test.ts` checks the runtime keys equal the expected list, but a `satisfies Record<AnalyticsEventName, AnalyticsEventDefinition>` annotation on `ANALYTICS_EVENTS` would make a missing/extra event a *compile* error too, catching drift before the test even runs.

- **Coverage threshold in CI** — There's no coverage gate, so a new uncovered branch can land green. Wire `node --test --experimental-test-coverage` (or `c8`) with a modest line/branch floor on `lib/analytics*.ts` + `lib/diagnostics-context.tsx` so the telemetry/consent core can't silently lose coverage.

- **Make `resolveNumericEnv` behaviourally unit-testable** — it now lives in `lib/env.ts` which imports `react-native` (`Platform`), so it can only be structurally tested. Move the pure parser into the existing react-native-free `lib/url-helpers.ts` (or a new `lib/env-helpers.ts`) and re-export it from `lib/env.ts`, mirroring the `getAppBaseUrl` → `resolveAppBaseUrl` split. Then a real `__tests__/env-helpers.test.ts` can assert the empty/NaN/zero/negative/valid matrix instead of regex-matching source.

- **Add an optional `{ min, max }` clamp to `resolveNumericEnv`** — the `EXPO_PUBLIC_PROFILE_CACHE_TTL_MS < 30s` foot-gun already has its own pending task (a warning toast). A `resolveNumericEnv(raw, def, { min: 30_000 })` overload that floors out-of-range overrides at the source would protect Supabase free-tier limits for *every* future tunable (chat poll, marketplace cache) without each caller re-deriving the guard.

- **Promote `normaliseEnvironment` into `lib/env.ts` next to `resolveNumericEnv`** — `lib/sentry-config.ts` / `lib/analytics-config.ts` both re-implement an env-name normaliser; co-locating it with the new numeric resolver finishes the "one env-parse surface" consolidation the `.tasks` entry started and lets a single `runtime-config-parity` test cover both.

- **Lint guard against computed `process.env[...]` access** — the Metro-inlining foot-gun is currently policed by hand-written structural tests (`sentry-env-inlining`, `env-resolve-numeric`). A 15-line ESLint `no-restricted-syntax` rule banning `MemberExpression[computed=true][object.object.name="process"][object.property.name="env"]` would enforce it codebase-wide and fail the PR instead of one specific file's test.

- **Reuse `resolveNumericEnv` for the next tunable now, not later** — the `.tasks` entry cites "chat poll interval, marketplace cache" as future beneficiaries but ships zero second callers, so the helper's generality is unverified. Migrate one real numeric knob (e.g. a `EXPO_PUBLIC_CHAT_POLL_MS` for `lib/chat-context.tsx`) in the same PR family so the abstraction is proven by a second consumer rather than asserted.

### CURRENCY_CHIPS picker whitelist (2026-05-20)

- **Tighten `<CurrencyInput>`'s currency prop type from `string` to `CurrencyChipCode`** — the new literal-union type is exported but not yet threaded through the component props. Switching `currency: string` → `currency: CurrencyChipCode` makes a typo on any consumer page (`<CurrencyInput currency="USDT" />` → arbitrary unknown chip) fail at compile time instead of silently rendering nothing-selected. - **Reuse `CURRENCY_CHIPS` in `app/create.tsx`'s autocomplete to short-list the most common codes** — today the full 150-entry `lib/currencies.ts` dictionary feeds the autocomplete; surfacing `CURRENCY_CHIPS` as a "Common" header pinned to the top of the suggestions would speed up the 99% case (USD/EUR/GBP/etc.) without removing the long tail. - **Add an `EXPO_PUBLIC_EXTRA_CURRENCY_CHIPS=NZD,SGD` operator override** — `CURRENCY_CHIPS` is now centralised but still hard-coded.
