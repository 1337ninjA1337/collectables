## Upgrade suggestions

### Chat DB (2026-05-16)

- **Cursor pagination for `fetchMessagesForChat`** — it currently `select=*&order=created_at.asc` with no `limit`, so a long-lived conversation pulls the entire history into memory on every hydrate. Add `&limit=50` + a `created_at` cursor (or PostgREST `Range` header) and load older pages on scroll-up. Bounds memory and free-tier egress.

- **Realtime reconnect backfill** — while the WebSocket is down, `subscribeToInbox` misses INSERTs and only `refreshFromCloud` (mount-time) closes the gap. On `SUBSCRIBED` after a drop, fetch messages `created_at=gt.<last-known>` per active chat so no message is silently lost during a flaky connection.

- **First-class conversations table** — replace the free-text `chat_id` with a `conversations(id uuid, user_a uuid, user_b uuid, UNIQUE(user_a,user_b))` table and a `chat_messages.conversation_id uuid REFERENCES conversations(id)`. A real FK + unique participant pair is stronger than the derived-string CHECK and unlocks per-conversation metadata (muted, archived, last_message_at) without re-deriving from messages.

- **Functional idempotency test with a mocked fetcher** — the new ignore-duplicates path is only covered structurally. Add a `__tests__/supabase-chat-idempotency.test.ts` that injects a fake `fetcher` returning `200 []` and asserts `sendMessage` resolves to the synthesized message (not `null`), plus a fake returning a populated row for the happy path.

- **`chat_messages.client_id` + server uuid split** — keep the server `id uuid DEFAULT gen_random_uuid()` authoritative and add a separate indexed `client_id uuid` (the composition-time key) with a UNIQUE constraint for idempotency. Decouples "who owns the primary key" from "idempotency key", which is the canonical chat-storage pattern and avoids ever trusting a client-supplied PK.

- **SPA fallback regression guard** — Add a CI step that runs `npm run build` then asserts (a) `dist/404.html` contains `location.replace`, (b) `dist/index.html` contains `data-spa-restore`. This catches a future contributor accidentally reverting the post-build script (e.g. swapping back to `cp dist/index.html dist/404.html`) before the change reaches production iOS users.

- **Service worker for instant deep-link load** — Even with the sessionStorage redirect, the user sees a flash of the redirect page on iPhone. Registering a service worker (Workbox `NetworkFirst` + `NavigationFallback` to `/collectables/index.html`) would let cached deep links resolve to 200 directly and remove the redirect entirely on repeat visits.

- **Sentry breadcrumb for SPA URL restore** — Drop an `addBreadcrumb("spa-redirect", { from, to })` inside the restore script (gated on `__sentry_loaded__`) so Sentry sessions show exactly which deep-link URLs hit the 404 fallback. Useful for ranking which routes deserve a real static export.

- **Static export of common collection IDs** — Expo Router could pre-render the top-N most-shared collections at build time (e.g. by reading from Supabase) so their URLs return real 200s without the redirect dance. Reduces flicker and helps SEO for public collections.

- **Migrate baseUrl to a single source** — `app.json` declares `expo.experiments.baseUrl`, `lib/url-helpers.ts` infers it at runtime, and `scripts/build-spa-fallback.ts` re-reads it from app.json. A single `lib/base-url.ts` exporting a const + a helper that all three import would eliminate the drift risk if the deployment path ever changes (e.g. moving off the `1337ninja1337.github.io/collectables` prefix).

- **CI guard for `.pbit` drift** — Add a CI step that runs `npm run powerbi:template` then `git diff --exit-code docs/powerbi/Collectables-Starter.pbit`. The unit test already asserts the committed binary matches a fresh build, but a CI step makes a stale-template PR fail loudly at the pipeline level (mirrors the SPA-fallback regression-guard pattern).

- **Embed starter visuals in the `.pbit` Report/Layout** — `buildReportLayout()` currently ships one empty page. Author a clustered-column funnel visual + a DAU line chart + a premium-conversion KPI card directly into `visualContainers` so the template opens with the three §5 measures already charted, not just defined. A `lib/powerbi-visuals.ts` helper keeps the visual JSON testable.

- **`.pbix` (data snapshot) export sibling** — Add a `scripts/build-powerbi-snapshot.ts` that, given a service-role connection, runs the M query and bakes a small sample dataset into a `.pbix` for offline demoing/screenshots without a live Supabase. Reuses `lib/powerbi-template.ts` part assembly; only the partition source + a DataModel data part differ.

- **Round-trip `.pbit` validation in CI via a headless OPC check** — Power BI Desktop can't run in CI, but `System.IO.Packaging`-equivalent validation can: add a test that asserts `[Content_Types].xml` declares every part, every `_rels/.rels` Target resolves to a present part, and the central-directory offsets are internally consistent. Catches malformed packages before a user hits "can't open template".

- **Deflate the ZIP parts to shrink the committed binary** — `zipStore` uses STORED (no compression); the DataModelSchema dominates at ~7 KB. Switching to `zlib.deflateRawSync` (method 8) with a fixed compression level keeps determinism while roughly halving the committed `.pbit` size — relevant since it lives in git history forever.

- **Generate the schema-reference table in `powerbi-connection.md` from `ANALYTICS_COLUMNS`** — the column list now exists in three places (the doc table, `lib/powerbi-template.ts`, and `lib/analytics-events.ts` for props). A small doc-gen test that asserts the markdown table matches the typed `ANALYTICS_COLUMNS` array prevents the schema reference from silently drifting when a column is added.

- **Run the analytics RLS verification in CI when a DB URL is present** — The `supabase db push` workflow runs only if `SUPABASE_DB_URL` is set. Add a gated step right after it that runs `psql "$SUPABASE_DB_URL" -v ON_ERROR_STOP=1 -f supabase/tests/analytics_events_rls_test.sql`, turning the manual checklist into an automatic pipeline gate whenever the secret is configured.
- **Generalise the RLS-deny test into a pgTAP-style harness** — Extract the DO-block assertions into a reusable `supabase/tests/_rls_helpers.sql` (e.g. `assert_deny_all(schema, table)`), then add sibling scripts for every other RLS-default-deny table. One pattern, many tables, zero copy-paste drift.
- **Negative INSERT/UPDATE coverage in the RLS script** — It currently proves anon/authenticated can't SELECT; extend it to also attempt an `INSERT`/`UPDATE`/`DELETE` under each role and assert each is denied, so a future migration that adds a write-only policy (but no read policy) is still caught.
- **Surface the analytics event count in the verification PASS notice** — Have the service_role read step `RAISE NOTICE` the row count + max(occurred_at) so an operator running the checklist also gets a quick "is the mirror actually receiving events" health signal in the same pass.
- **Schema-reference doc-gen from the migration** — `docs/powerbi-connection.md` §4, `lib/powerbi-template.ts` `ANALYTICS_COLUMNS`, and the migration DDL all hand-maintain the column list. A test that parses the `CREATE TABLE` in `20260508_analytics_events.sql` and asserts the doc table + `ANALYTICS_COLUMNS` match it would make the migration the single source of truth.
