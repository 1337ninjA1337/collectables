## Upgrade suggestions

### Chat DB (2026-05-16)

- **Cursor pagination for `fetchMessagesForChat`** — it currently `select=*&order=created_at.asc` with no `limit`, so a long-lived conversation pulls the entire history into memory on every hydrate. Add `&limit=50` + a `created_at` cursor (or PostgREST `Range` header) and load older pages on scroll-up. Bounds memory and free-tier egress.

- **Realtime reconnect backfill** — while the WebSocket is down, `subscribeToInbox` misses INSERTs and only `refreshFromCloud` (mount-time) closes the gap. On `SUBSCRIBED` after a drop, fetch messages `created_at=gt.<last-known>` per active chat so no message is silently lost during a flaky connection.

- **First-class conversations table** — replace the free-text `chat_id` with a `conversations(id uuid, user_a uuid, user_b uuid, UNIQUE(user_a,user_b))` table and a `chat_messages.conversation_id uuid REFERENCES conversations(id)`. A real FK + unique participant pair is stronger than the derived-string CHECK and unlocks per-conversation metadata (muted, archived, last_message_at) without re-deriving from messages.

- **Functional idempotency test with a mocked fetcher** — the new ignore-duplicates path is only covered structurally. Add a `__tests__/supabase-chat-idempotency.test.ts` that injects a fake `fetcher` returning `200 []` and asserts `sendMessage` resolves to the synthesized message (not `null`), plus a fake returning a populated row for the happy path.

- **`chat_messages.client_id` + server uuid split** — keep the server `id uuid DEFAULT gen_random_uuid()` authoritative and add a separate indexed `client_id uuid` (the composition-time key) with a UNIQUE constraint for idempotency. Decouples "who owns the primary key" from "idempotency key", which is the canonical chat-storage pattern and avoids ever trusting a client-supplied PK.

- **SPA fallback regression guard** — Add a CI step that runs `npm run build` then asserts (a) `dist/404.html` contains `location.replace`, (b) `dist/index.html` contains `data-spa-restore`. This catches a future contributor accidentally reverting the post-build script (e.g. swapping back to `cp dist/index.html dist/404.html`) before the change reaches production iOS users.

- **Service worker for instant deep-link load** — Even with the sessionStorage redirect, the user sees a flash of the redirect page on iPhone. Registering a service worker (Workbox `NetworkFirst` + `NavigationFallback` to `/collectables/index.html`) would let cached deep links resolve to 200 directly and remove the redirect entirely on repeat visits.

- **Sentry breadcrumb for SPA URL restore** — Drop an `addBreadcrumb("spa-redirect", { from, to })` inside the restore script (gated on `__sentry_loaded__`) so Sentry sessions show exactly which deep-link URLs hit the 404 fallback. Useful for ranking which routes deserve a real static export.

- **Static export of common collection IDs** — Expo Router could pre-render the top-N most-shared collections at build time (e.g. by reading from Supabase) so their URLs return real 200s without the redirect dance. Reduces flicker and helps SEO for public collections.

- **Migrate baseUrl to a single source** — `app.json` declares `expo.experiments.baseUrl`, `lib/url-helpers.ts` infers it at runtime, and `scripts/build-spa-fallback.ts` re-reads it from app.json. A single `lib/base-url.ts` exporting a const + a helper that all three import would eliminate the drift risk if the deployment path ever changes (e.g. moving off the `1337ninja1337.github.io/collectables` prefix).

### Power BI starter `.pbit` (2026-05-17)

- **CI step to regenerate-and-diff the `.pbit`** — Add a job that runs `npm run build:pbit` then `git diff --exit-code docs/powerbi/Collectables-Starter.pbit`. The generator is deterministic, so a non-empty diff means someone edited `lib/pbit-template.ts` without committing the regenerated binary (or hand-edited the binary). Catches drift the structural test can't (it validates *content*, not *freshness*).

- **Single source of truth for the seven measures** — `lib/pbit-template.ts`, `docs/powerbi/measures.dax`, and `docs/powerbi-connection.md` §5 each carry the DAX independently; the new parity test only checks measure *names*, not bodies. Extract the seven `Name := expression` blocks into one `lib/powerbi-measures.ts` and have the generator + a doc-lint test derive the others from it, so an expression edit can't silently diverge.

- **Generate `.pbit` report visuals, not just the model** — `buildReportLayout()` ships empty pages (Power BI auto-lays-out on open). Emitting real `visualContainers` (a DAU line chart on `occurred_at`, a clustered column for the funnel, a card for `PremiumConversionRate7d`) would deliver a dashboard out-of-the-box instead of just a primed model.

- **Round-trip an exported `.pbix` as a golden fixture** — Once someone opens the template in Power BI Desktop and saves a working `.pbix`, commit a trimmed copy as a test fixture and assert our generated `DataModelSchema` is structurally compatible (same table/columns/measure names, parameter shape). Closes the "can't validate without Power BI" gap with a one-time human-in-the-loop capture.

- **Parameterise the default region in the `.pbit`** — `SupabaseHost` defaults to the literal `aws-0-<region>.pooler.supabase.com` placeholder, which won't connect until edited. Reading an optional `SUPABASE_POOLER_HOST` build env (like the Sentry/Clarity env pattern) and baking it as the default would let a self-hoster ship a ready-to-open template via GitHub secrets with no manual edit.

### analytics_events RLS hardening (2026-05-17)

- **`pgTAP` automated RLS test** — Promote the manual MANUAL-TASKS.md checklist to a `supabase/tests/analytics_events_rls.sql` pgTAP script (`SET ROLE` + `throws_ok`) runnable in `supabase db test` / CI against a local Supabase, so the lock-down is machine-verified on every push instead of relying on an operator remembering to run it.

- **CI grep guard against permissive policies** — Add a cheap test that scans every file under `supabase/migrations/` and fails if any later migration introduces `CREATE POLICY … ON public.analytics_events` or `GRANT … ON public.analytics_events TO anon|authenticated`. Catches a regression at PR time, before it ever reaches the DB the manual checklist runs against.

- **Column-level REVOKE belt-and-braces** — The migration `REVOKE ALL ON … FROM anon, authenticated` at table scope; a future `GRANT SELECT (name, occurred_at)` column grant would still slip through. Add an explicit column-privilege check to the verification snippet (`SELECT name FROM public.analytics_events` under each role) so partial leaks are caught too.

- **`auth.users` join-leak audit** — `analytics_events.user_id` FKs `auth.users`. Document/verify that the typical Power BI service-role report never widens to PII columns of `auth.users` (email, phone); a starter view `analytics_events_safe` exposing only `user_id` hashed would let dashboards run without ever touching raw identifiers.

- **Retention / TTL policy** — The table is an unbounded long-tail store. A `pg_cron` monthly job (or a documented manual `DELETE FROM analytics_events WHERE occurred_at < now() - interval '18 months'`) bounds storage + blast radius if the service-role key is ever leaked, and should be added to MANUAL-TASKS.md next to the RLS checklist.
