# Upgrade Suggestions

- Migrate the remaining 20+ files (`app/index.tsx`, `app/chat/[id].tsx`, `app/listing/[id].tsx`, `components/bottom-nav.tsx` etc.) from inline hex literals to `lib/design-tokens.ts` imports. `app/marketplace.tsx` is the demo migration; replicating it codebase-wide would cap the brand-color rotation cost at one constant change instead of `grep -rn '#261b14'`.

- Add an ESLint rule (or a small CI grep step) that fails on raw `#[0-9a-fA-F]{6}` hex literals inside `app/**/*.tsx` and `components/**/*.tsx`, with an allowlist for `lib/design-tokens.ts` itself. Today nothing prevents a future contributor from re-introducing inline hexes; a lint guardrail makes the design-tokens convention enforceable.

- Extend `lib/design-tokens.ts` with semantic spacing/radius tokens (e.g. `RADIUS_CARD = 22`, `RADIUS_PILL = 999`, `SPACING_TIGHT = 6`) so card geometry is also one-line tunable. Today the same `borderRadius: 22` recurs across 14+ stylesheets â€” same problem, same solution.

- Generate a Storybook-style "tokens preview" route at `app/_dev/tokens.tsx` (gated behind `__DEV__`) that renders every named color/spacing token as a swatch with its key. Designers reviewing PRs could verify visual changes without spelunking through stylesheets.

- Type the `BackgroundColorValue`/`ColorValue` props on shared components (`PremiumGate`, `EmptyState`, etc.) as `DesignToken | (string & {})` so consumers can pass either a token name or a one-off literal but get autocomplete + typo-detection on the common path. The `(string & {})` trick keeps the prop assignable to anything but surfaces the named tokens in IntelliSense.

- Memoise the `Intl.RelativeTimeFormat` instance per language inside `formatRelativeDate` so a screen with N timestamps (chats list, price-history table) doesn't construct N formatters per render. The current implementation news up the formatter on every call â€” fine for one-off renders, wasteful for lists. A `Map<AppLanguage, Intl.RelativeTimeFormat>` cache keyed by locale closes the gap with ~5 LOC and zero behavioural change.

- Auto-tick the "Listed X ago" / chat preview timestamps every 60 seconds via a single shared `useNow()` hook so a chat preview that says "1 minute ago" rolls over to "2 minutes ago" without requiring a screen re-mount. Today the relative timestamp is captured on render and stays frozen until the parent context re-renders. A 30-line hook in `lib/use-now.ts` would close the gap.

- Extract the new `marketplaceListedAt(when)` template into a reusable `relativeDateLabel(prefix, when)` i18n helper so other screens (item detail's "Acquired X ago", profile's "Member since") can use the same prefix-translation pattern without each adding a new key. Today every "X ago" hint requires a new translation entry per language.

- Render the chat preview timestamp in absolute form (HH:mm) for messages received in the last hour and switch to `formatRelativeDate` only for older entries â€” mirrors how mainstream chat apps surface fresh activity. The `formatWhen` helper that just got removed already had the same-day branch; we could keep that logic and only use `formatRelativeDate` for older messages.

- Add a tooltip / accessible label on the new "Listed X ago" `<Text>` that reveals the absolute ISO date on hover (web) / long-press (native), so power users who care about precise timestamps aren't punished by the relative format. React Native's `accessibilityLabel` + a `Pressable`/`onLongPress` toast covers both platforms.

- Add an `expo-dev-menu` peer dep wired through `loadDevMenuModule` in `lib/dev-menu.ts` so the new dev-only DevMenu hook actually surfaces a "Clear runtime Supabase config" entry on native dev builds. Today `loadDevMenuModule` returns null because the package isn't installed, so engineers only get the `globalThis.__clearRuntimeSupabaseConfig` web-console fallback. A 1-line `npx expo install expo-dev-menu` plus a custom config plugin entry would unlock the native DevMenu path without changing any application code.

- Move the `__DEV__` global type declaration into a project-wide `types/globals.d.ts` so both `lib/dev-menu.ts` and any future dev-gated module can reference `__DEV__` without the `(globalThis as { __DEV__?: boolean })` cast. The cast pattern works but already appears in two places; a single ambient declaration removes the repetition and lets TypeScript flag misspelled `__dev__` accesses.

- Extend `registerDevMenu` in `lib/dev-menu.ts` to accept a per-action label (instead of using the function key as the DevMenu label) so engineers see "Clear runtime Supabase config" in the menu instead of the camelCase identifier. Today the action map's keys double as labels, which is fine for `globalThis` helpers but reads awkwardly in the user-facing DevMenu list.

- Auto-register a `__resetCollectablesStorage` dev helper alongside `__clearRuntimeSupabaseConfig` that wipes all `collectables-*` AsyncStorage keys via the existing `lib/storage-keys.ts` exports. Today engineers manually clear AsyncStorage to reset onboarding state; a one-line dev escape hatch keyed off the same `registerDevMenu` call closes that gap.

- Add an integration test in `__tests__/dev-menu-wiring.test.ts` that asserts `app/_layout.tsx` calls `registerDevMenu` with `clearRuntimeSupabaseConfig` in its `actions` map. Today the unit test only covers `registerDevMenu` itself; a structural regex test against `_layout.tsx` would catch a future regression where someone removes the `useEffect` block while refactoring providers.

- Promote the new `normalizeListing` helper out of `lib/marketplace-context.tsx` into `lib/marketplace-helpers.ts` so any future code path that hydrates `MarketplaceListing` objects from JSON (push notifications, deep-link previews, server-rendered routes) can reuse the same defensive-default logic without re-implementing the `?? null` coalesce. Keep the original call site untouched.

- Add a `cloud-marketplace` test file (`__tests__/supabase-marketplace.test.ts`) that mocks `fetcher` and `tokenProvider` to assert `cloudMarkSold` actually serialises `buyer_user_id` into the PATCH body. Today only the pure shapes module is unit-tested; threading the new buyer field through the cloud wrapper layer means a future regression (e.g. someone re-introducing the old `{ sold_at }` literal) would go unnoticed until UI tests catch it.

- Surface a small "transferred" badge on `app/listing/[id].tsx` and the marketplace card grid for sold listings whose `buyerUserId !== null`. Today sold listings are simply hidden from the active feed; once the buy/accept flow lands, an explicit "Sold to {buyerName}" pill on `My listings` makes the cycle visible without opening detail. The plumbing now exists â€” only the rendering is missing.

- Add a `buyerUserId` filter helper `listingsAcquiredByUser(listings, userId)` in `lib/marketplace-helpers.ts` (mirroring the shape of `listingsForUser`) so future profile screens can render a "Trades & Transfers" section without re-implementing the `l.buyerUserId === userId && l.soldAt != null` predicate. A 5-line addition with a paired unit test.

- Replace the loose `JSON.parse(raw) as MarketplaceListing[]` cast in `lib/marketplace-context.tsx` with a runtime validator that drops malformed entries (e.g. missing `id` or `mode`) instead of trusting the AsyncStorage payload. Today a corrupt local cache crashes downstream rendering because `mode` ends up `undefined`; a `coerceListing(raw)` helper that returns `null` on validation failure would harden first-paint without changing the happy path.

- Surface a "Sold to @buyer" pill on the marketplace `ListingCard` for sold listings (currently filtered out via `activeListings`), keyed off the new `MarketplaceListing.buyerUserId` field. A "recently sold" carousel under the active grid would let users see what's moving without changing the active-only marketplace contract â€” useful pricing context for sellers drafting their own listings.

- Add a `MarketplaceListingHistory` view on the user profile screen that lists prior `buyerUserId === user.id` listings (purchases) and `ownerUserId === user.id && soldAt !== null` listings (sales). The data is already in `useMarketplace().listings`; it's a single `useMemo` filter and a list render, no new context required.

- Persist the `buyerUserId` change through the `subscribeToListings` realtime channel by switching the filter from `event: INSERT` to also include `UPDATE` events, so when a buyer claims a listing on device A it disappears from device B's marketplace immediately rather than after a manual refresh. The PostgresChanges payload already exposes the updated `buyer_user_id`; threading it through the same `rowToListing` path would close the gap with ~10 LOC.

- Add a `marketplace_transfers` table (or extend `marketplace_listings` with `transferred_at`) so the future item-ownership transfer step (sub-task #3) records *who got what and when* even after the seller deletes their listing, giving us audit history. Today even with `buyer_user_id` we lose the price/mode context if the listing row is dropped.

- Reuse the new `markSoldPayload(soldAt, buyerUserId)` helper from `lib/supabase-marketplace-shapes.ts` to add a strict integration test asserting that `cloudMarkSold` PATCH bodies always include `buyer_user_id` (even when null) â€” guards against a future refactor accidentally dropping the column from the PATCH and silently leaving the field unset on existing rows.

- Lock the Buy / Trade button in `app/listing/[id].tsx` behind the same `friendsOnly` rule that already gates the Message-owner CTA, so a non-friend viewer cannot claim a listing whose owner has restricted who can interact with them. Today the claim button renders for any non-self viewer; the friends-only branch already returns a placeholder for messaging â€” we should reuse it for claiming too.

- Send an automatic chat message to the seller via `ensureChatWith` + `appendMessage` immediately after a successful claim, using a templated body like "Bought your listing for {price}". This closes the loop for the seller without forcing the buyer to open a separate chat thread; the existing chat infrastructure already supports it.

- Add a system-toast on successful claim using the existing `ToastProvider` to mirror the `marketplaceListingCreated` / `premiumActivated` toast pattern. Today the only feedback after claiming is the listing flipping to "Sold" â€” a transient toast keyed on the `markListingSold` callback's success would feel more responsive on slow networks.

- Persist a `claimingListingId` flag in `MarketplaceContext` instead of the local `claiming` state in `app/listing/[id].tsx`, so navigating away and back during a slow PATCH doesn't reset the spinner. The context already centralises listings â€” it's the natural home for in-flight claims.

- Migrate the `Platform.OS === "web"` branch in `handleClaimPress` to a shared `lib/confirm-dialog.ts` helper so other screens (delete-collection, cancel-premium, leave-friend) can use the same web-vs-native confirmation primitive instead of each duplicating the `window.confirm` / `Alert.alert` pair.

- Have the seller's device react to a "their listing was claimed" realtime event by offering an "Archive item" or "Delete from my collection" prompt. Today the buyer pulls a snapshot into their collection but the seller still keeps the original â€” useful for collectors who want a sale audit trail, confusing for those who expect the item to "leave" their collection.

- Migrate the `transferItemToBuyer` Acquired-collection ID generator from `${userId}-acquired-marketplace` into a `userScopedCollectionId(userId, "acquired-marketplace")` helper in `lib/collections-helpers.ts` so future system-managed collections (e.g. "Wishlist", "Trash", "Premium-only") share the same naming pattern.

- Persist the `transferItemToBuyer` invocation in a local "transfer history" log keyed by `${listingId}-${createdAt}` so a buyer can see every marketplace acquisition with its origin listing, even after the original listing is deleted upstream â€” opens the door to a proper "Purchases" tab (sub-task #4).

- Stop hard-coding `acquiredFrom: t("marketplaceTitle")` in `performClaim` and instead store the seller's `@username` (or fallback `displayName`) so the buyer's Acquired collection records *who* the item came from. Provenance is the most useful field for collectors auditing their own catalogue.

- Add a unit test in `__tests__/transfer-item-to-buyer.test.ts` that drives the `transferItemToBuyer` reducer (extract the inner update logic from the closure into a pure helper if needed) so we can assert: (a) idempotent re-claims don't duplicate items, (b) the Acquired collection's `coverPhoto` updates on first transfer, (c) the new item carries `condition` / `tags` from the snapshot.

- Add a "Sales" subsection on the marketplace screen mirroring "My purchases" â€” it would render `myListings` filtered by `soldAt !== null`. Together they form a complete personal trading audit ("what I bought" / "what I sold") without leaving the marketplace tab.
