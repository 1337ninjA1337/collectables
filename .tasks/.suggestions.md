## Upgrade suggestions

- **SPA fallback regression guard** — Add a CI step that runs `npm run build` then asserts (a) `dist/404.html` contains `location.replace`, (b) `dist/index.html` contains `data-spa-restore`. This catches a future contributor accidentally reverting the post-build script (e.g. swapping back to `cp dist/index.html dist/404.html`) before the change reaches production iOS users.

- **Service worker for instant deep-link load** — Even with the sessionStorage redirect, the user sees a flash of the redirect page on iPhone. Registering a service worker (Workbox `NetworkFirst` + `NavigationFallback` to `/collectables/index.html`) would let cached deep links resolve to 200 directly and remove the redirect entirely on repeat visits.

- **Sentry breadcrumb for SPA URL restore** — Drop an `addBreadcrumb("spa-redirect", { from, to })` inside the restore script (gated on `__sentry_loaded__`) so Sentry sessions show exactly which deep-link URLs hit the 404 fallback. Useful for ranking which routes deserve a real static export.

- **Static export of common collection IDs** — Expo Router could pre-render the top-N most-shared collections at build time (e.g. by reading from Supabase) so their URLs return real 200s without the redirect dance. Reduces flicker and helps SEO for public collections.

- **Migrate baseUrl to a single source** — `app.json` declares `expo.experiments.baseUrl`, `lib/url-helpers.ts` infers it at runtime, and `scripts/build-spa-fallback.ts` re-reads it from app.json. A single `lib/base-url.ts` exporting a const + a helper that all three import would eliminate the drift risk if the deployment path ever changes (e.g. moving off the `1337ninja1337.github.io/collectables` prefix).

- **Chat DB realtime UPDATE channel** — `subscribeToInbox` currently listens for `event: INSERT` only. As Chat DB #2 / #3 land (soft-delete, soft-edit, status), other devices won't see those mutations live. Adding a parallel `*` (or explicit UPDATE) handler — and a corresponding `chatRowToMessage` merge path in chat-context that replaces the existing entry by id — keeps multi-device timelines coherent without a manual refresh.

- **Idempotent send unit test with mocked fetcher** — The new `client_message_id` + 409-refetch path is covered structurally + by shape tests, but a true behavioural test (mocked fetcher returns 409 once, then a 200 representation; assert `sendMessage` returns the canonical row and never POSTs twice for the same `clientMessageId`) would lock down the runtime wrapper's retry semantics. The Sentry/marketplace tests have a similar pattern that could be copied.

- **Drop msg.id from the optimistic-local fallback** — Today, when the cloud send fails, chat-context generates a `msg-${Date.now()}-${suffix}` local id and persists it. After Chat DB #1 the canonical dedupe key is `clientMessageId`, so the optimistic local id is now redundant; using the same uuid as both `id` and `clientMessageId` would let `appendMessage` short-circuit on either match path without the second findIndex pass.

- **Server-side updated_at + edited_at trigger** — Once Chat DB #3 (soft edit) lands, edits should bump `updated_at` automatically via a `BEFORE UPDATE` trigger rather than relying on the client to stamp `edited_at`. That keeps the source-of-truth on the server and prevents a buggy client from rewriting history. Mirror the pattern from `marketplace_listings.updated_at` if/when that exists.

- **Conversations table normalization (Chat DB #4 prep)** — The deterministic `chat-${a}-${b}` id couples chat addressing to user ids forever; once group chats or chat-archive features land, migrating to `conversations(id uuid, kind text)` + `conversation_participants(conversation_id, user_id)` is unavoidable. Sketching the migration + backfill SQL now (even without applying it) front-loads the design and reveals which RLS policies need rewriting.
