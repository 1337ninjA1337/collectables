## Upgrade suggestions

- **Promote the new `types/globals.d.ts` into a typed ambient module for `EXPO_PUBLIC_*` env vars** — today every `lib/*-config.ts` module reads `process.env.EXPO_PUBLIC_X` and falls back to `undefined`. A `declare namespace NodeJS { interface ProcessEnv { EXPO_PUBLIC_SUPABASE_URL?: string; ...; } }` block in the same `globals.d.ts` would give every consumer IntelliSense on the documented env-var list and flag typos at compile time, without changing any runtime behaviour.

- **Declare `globalThis.__sentryStatus` / `globalThis.__sendSentryTestError` / `globalThis.__clearRuntimeSupabaseConfig` ambient signatures in `types/globals.d.ts`** — these dev helpers are attached to `globalThis` at runtime in `app/_layout.tsx`, but their types are erased so a `globalThis.__sentryStatus()` call from the dev console (or a future structural test) sees `unknown` instead of the real function signature. A `declare const __sentryStatus: () => SentryStatusSnapshot;` block (gated under a `// dev-only`-flavoured comment) gives the typed surface without any runtime change.

- **Add a `__tests__/globals-d-ts-coverage.test.ts` that grep-walks every `(globalThis as` cast across `lib/`+`app/`+`components/`** — the new ambient declaration retired one cast in `lib/dev-menu.ts`; the same pattern is likely to creep back in for the next dev-globals helper. A 20-line structural test asserting the call-site count stays at zero would force every new global through the ambient declaration instead of an ad-hoc cast.

- **Add a copy-to-clipboard tap on each `app/_dev/tokens.tsx` swatch** — designers reviewing the palette currently have to manually retype the token name (e.g. `AMBER_ACCENT`) into their editor; a `Pressable` wrapper with `Clipboard.setStringAsync(token.name)` (web) / `navigator.clipboard.writeText` (RN web) plus a one-line "copied" toast via the existing `ToastProvider` would shave 2-3 seconds per swatch and feels right at home in a dev tool.

- **Render a `TYPOGRAPHY` section on `app/_dev/tokens.tsx` showing each `FONT_*` constant as a paragraph in its actual face** — today the route only covers colours + geometry, but the font ladder (`FONT_DISPLAY` / `FONT_BODY` / `FONT_BODY_BOLD` / etc.) is just as much a design token and equally invisible at a glance. A 30-line `<View>` with one `Text` per font + a "The quick brown fox…" sample would close the gap.

- **Add a `__tests__/dev-tokens-route-registered.test.ts` that asserts `app/_dev/tokens.tsx` is reachable via Expo Router's file-system routing** — today the route exists but nothing structurally tests that it default-exports a function with no required params (Expo Router silently ignores broken default exports). A quick `assert.equal(typeof (await import("../app/_dev/tokens")).default, "function")` would catch a future regression where someone wraps the export in a HOC that breaks the routing contract.

- **Surface the dev tokens preview as a settings-screen footer link gated on `__DEV__ || adminFlag`** — today the route is discoverable only by typing `/_dev/tokens` manually; a "Design tokens preview" row under the Diagnostics block (the same surface where `__sentryStatus` already lives) would let designers and reviewers reach the page in one tap during a real session instead of digging into the URL bar.

- **Add a deterministic CSS-snapshot test that hashes the swatches map** — the `groupColorTokens` helper inside `app/_dev/tokens.tsx` partitions the 100+ named tokens into prefix buckets; if someone reorders the COLOR_GROUPS or adds a token to a prefix that no group matches, the orphan bucket silently catches it but the visual ordering changes. A `__tests__/dev-tokens-classifier.test.ts` that asserts the hash of `[group.id, ...entries.map(name)]` matches a recorded snapshot would catch palette reshuffles before they hit a designer's eyes.
