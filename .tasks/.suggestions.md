PostgREST accepts an array body for a single multi-row `ON CONFLICT` upsert, so a `deliverBatch(entities)` path (still uuid-keyed, still idempotent) would cut reconnect latency dramatically.
- **A "pending sync" pill driven off `hasPendingUpserts`** — BE-16 wants a syncing indicator; the queues now expose exactly the signal. Surfacing `pendingCollections`/`pendingItems` counts (plus chat's) through a shared `usePendingSyncCount()` would let the home header show "N changes waiting to sync" and a manual "retry now" that just bumps `refreshTick`.
- **Tombstone queue for offline deletes (pairs with BE-15)** — deletes today are fire-and-forget `deleteRemote*().catch()`, so a delete issued offline never retries and the row reappears on the next cloud pull. A sibling pending-DELETE queue (uuid-keyed, flushed after upserts) over the same engine would make offline deletes as durable as offline upserts; the current dequeue-on-delete guard only prevents local resurrection, not cloud divergence.
- **Reconnect trigger instead of refresh-tick polling** — the flush only fires on `ready`/`user`/`refreshTick`, so a queue parked while offline waits for the next manual refresh or screen re-mount. Wiring the flush to a real connectivity signal (the realtime client's `realtimeOnline` transition, or `NetInfo`) would flush the instant the network returns rather than on the next incidental re-render.
