## Upgrade suggestions

- **SPA fallback regression guard** — Add a CI step that runs `npm run build` then asserts (a) `dist/404.html` contains `location.replace`, (b) `dist/index.html` contains `data-spa-restore`. This catches a future contributor accidentally reverting the post-build script (e.g. swapping back to `cp dist/index.html dist/404.html`) before the change reaches production iOS users.

- **Service worker for instant deep-link load** — Even with the sessionStorage redirect, the user sees a flash of the redirect page on iPhone. Registering a service worker (Workbox `NetworkFirst` + `NavigationFallback` to `/collectables/index.html`) would let cached deep links resolve to 200 directly and remove the redirect entirely on repeat visits.

- **Sentry breadcrumb for SPA URL restore** — Drop an `addBreadcrumb("spa-redirect", { from, to })` inside the restore script (gated on `__sentry_loaded__`) so Sentry sessions show exactly which deep-link URLs hit the 404 fallback. Useful for ranking which routes deserve a real static export.

- **Static export of common collection IDs** — Expo Router could pre-render the top-N most-shared collections at build time (e.g. by reading from Supabase) so their URLs return real 200s without the redirect dance. Reduces flicker and helps SEO for public collections.

- **Migrate baseUrl to a single source** — `app.json` declares `expo.experiments.baseUrl`, `lib/url-helpers.ts` infers it at runtime, and `scripts/build-spa-fallback.ts` re-reads it from app.json. A single `lib/base-url.ts` exporting a const + a helper that all three import would eliminate the drift risk if the deployment path ever changes (e.g. moving off the `1337ninja1337.github.io/collectables` prefix).

## Chat DB upgrade suggestions

- **Server-authoritative timestamps** — `messageToInsertPayload` lets the client send `created_at`/`id`. This is needed for offline pending-flush dedupe, but it also lets any client back/forward-date messages, which corrupts ordering and the `m.createdAt > lastRead` unread cutoff. Add a `client_created_at timestamptz` column for the offline replay key and keep `created_at` strictly server-set (`DEFAULT now()`, drop it from the insert payload, sort/cutoff on the server column).

- **Paginated message fetch** — `fetchMessagesUrl` selects *all* rows for a chat (`order=created_at.asc`, no limit). A long-lived friendship produces an unbounded payload on every hydrate. Add a `&limit=50&order=created_at.desc` window + a "load older" cursor on `created_at` (the `chat_messages_chat_created_idx` already supports it), reversing client-side for display.

- **Soft-delete support** — Messages are immutable (no UPDATE/DELETE policy). Users expect to delete their own messages. Add `deleted_at timestamptz NULL` + an UPDATE RLS policy allowing only `auth.uid() = from_user_id` to set `deleted_at` (and nothing else, via a column-level guard), with the client rendering a "message deleted" tombstone.

- **`conversations` table** — `chat_id` is a synthesised text key. A real `conversations(id uuid, user_a uuid, user_b uuid)` table with `chat_messages.conversation_id` FK would enable per-conversation metadata (muted, archived, last_message_at denormalised for a cheap inbox query that avoids scanning all messages to build previews).

- **Message delivery/read receipts** — `chat_reads` only tracks a per-chat last-read timestamp. A per-message `read_at`/`delivered_at` (or a `message_receipts` table) would enable WhatsApp-style ✓✓ indicators and a precise unread count that survives clock skew between the two participants' devices.

## Power BI / analytics upgrade suggestions

- **Analytics SQL views over raw `analytics_events`** — Power BI users currently expand `properties` jsonb client-side in Power Query, which drifts from `lib/analytics-events.ts`. A `supabase/migrations` set of `SECURITY DEFINER` views (`analytics_dau`, `analytics_funnel`, `analytics_premium_cohort`) would push the shaping server-side, keep DAX trivial, and let the `.pbit` ship without per-event jsonb-expansion steps.

- **`.pbit` golden-file ZIP test** — Once Analytics #15b lands, add a test that unzips the generated `.pbit` and asserts the OPC part set + that `DataModelSchema` embeds the exact M from `queries.m`. Catches a future `queries.m`/`.pbit` drift without needing Power BI Desktop.

- **Parameterise the connection doc region** — `docs/powerbi-connection.md` and `queries.m` hard-code `aws-0-<region>.pooler.supabase.com`. A single documented placeholder table (region → pooler host) plus a `queries.m` comment listing the common AWS regions would cut the most common "wrong host" support issue.

- **Scheduled refresh via Supabase cron → CSV snapshot** — Power BI Pro is required for scheduled cloud refresh. A nightly `pg_cron` job writing a flattened daily-rollup table (`analytics_daily`) that Power BI Import-mode reads would give free-tier users a once-a-day dashboard without Pro.

- **Clarity + PostHog deep-link columns** — Add `posthog_event_url` / `clarity_session_url` computed columns in `queries.m` (string-built from `user_id` + `occurred_at`) so a Power BI row can click straight through to the PostHog event or Clarity replay for qualitative follow-up.
