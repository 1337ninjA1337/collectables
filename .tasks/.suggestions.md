# Upgrade Suggestions

- Extract `normaliseEnvironment` from both `lib/sentry-config.ts` and `lib/analytics-config.ts` into a shared `lib/env.ts:normaliseDeploymentEnv(value)` helper so the dev/staging/production collapse rule has one canonical implementation. Today the function is duplicated; a future "preview" environment added to one would silently fall back to "production" in the other.

- Add an `EXPO_PUBLIC_ANALYTICS_ENV` row to README-DEPLOY.md + APPSTORE-SUBMISSION.md documentation. The new resolver supports it but the docs only list `EXPO_PUBLIC_SENTRY_ENV`; engineers reading the env table won't discover the override unless they read the source.

- Add a `scripts/check-expo-install.ts` post-install hook that re-runs `npx expo install --check` against the locked `posthog-*` versions and fails CI if Expo's registry recommends a different version for the current SDK. Today the sandbox can't reach the Expo registry, so versions were pinned manually; a CI hook running on full network would surface the recommended version drift before it lands in TestFlight.

- Bundle-size budget: add a `scripts/check-bundle-size.ts` step that fails CI when the web bundle exceeds, say, 4.5MB. PostHog adds ~60KB and Clarity adds ~30KB; with future Analytics #11 (Clarity script injection) we'll cross 4.1MB. A budget guardrail catches a regression where someone imports `posthog-js` statically into a screen instead of lazily.

- Translate `docs/analytics-platform.md` "Privacy and opt-out" section into the public privacy policy (the same one that needs the Sentry paragraph from `APPSTORE-SUBMISSION.md` §6) so users see a unified disclosure for analytics + crash reporting. Today the analytics opt-out story only exists in an engineering doc; user-facing transparency requires it live on the privacy page.

- Add a `docs/analytics-cost-projection.md` companion that turns the $0/month free-tier ceiling in `docs/analytics-platform.md` into a small linear-projection table (DAU → events/mo → next-tier price) so the founder gets a one-glance "you'll cross PostHog's free tier when DAU hits ~30" signal. Today the doc says "1M events/mo at ~30 DAU"; surfacing the math lets the team know *when* to budget for the upgrade.

- Wire the new `__tests__/sentry-tests-peer-dep-free.test.ts` into a `lint:peer-dep-free` script in `package.json` that runs *only* that single test, so a pre-push hook (or a pre-merge gate) can verify the SDK isolation invariant in <100ms instead of waiting for the full 7s test suite. Today the guard runs as part of `npm test`; isolating it makes the failure reason obvious at a glance.

- Promote the `SENTRY_TEST_FILES` array in `__tests__/sentry-tests-peer-dep-free.test.ts` from a hand-maintained list to a glob (`fs.readdirSync` + filter on `^sentry-.*\\.test\\.ts$`) so a future Sentry test file is picked up without editing the array. Today missing a file from the list leaves it un-guarded; a glob makes the invariant fail-closed.

- Add a "delete `node_modules/@sentry/react-native` and re-run only the sentry tests" step to CI as a separate matrix job. Today the guard structurally proves no static import — but a smoke-run with the SDK literally absent would catch a *transitive* dependency that imports it (e.g. `@sentry/core`). Cheap to add: `rm -rf node_modules/@sentry/react-native && npx tsx --test __tests__/sentry-*.test.ts`.

- Commit a `PRIVACY.md` to the repo root that consumes the "Suggested public privacy policy paragraph" from `APPSTORE-SUBMISSION.md` §6 plus the equivalent paragraphs for Supabase + Cloudinary, and host it via the existing GitHub Pages workflow at `/privacy`. Today the policy text only lives in submission docs — App Store review requires a public URL, and a tracked file makes the privacy guarantees enforceable via PR review instead of "trust the markdown".

- Generate translated copies of the Sentry privacy paragraph (`PRIVACY.md.ru`, `PRIVACY.md.de`, etc.) using the existing `lib/i18n-context.tsx` language list, then surface a language picker on the privacy page. Today the paragraph is English-only; a non-English EU user is technically owed the disclosure in their language under GDPR Art. 12.

- Add a `lib/privacy-disclosure.ts` constant that lists every sub-processor (Supabase, Cloudinary, Sentry, future PostHog) with `name + url + dataTypes + retention` so the App Privacy table, the public PRIVACY.md, and the in-app Settings → Privacy disclosure all read from one source of truth. Drift between the three is the most common privacy-review finding.

- Surface the diagnostics opt-out state on `app/settings.tsx` with a clear "Crash reports: enabled — last sent 3 hours ago" footer driven by the new `getSentryStatus()` snapshot. Today the toggle exists but doesn't tell the user whether anything was actually sent — useful for users who flip the toggle and want to verify nothing is leaking.

- Promote the `app.json` plugin/extra parity guard (`__tests__/sentry-eas-config.test.ts`) into a generic `__tests__/app-json-parity.test.ts` that walks every `expo.plugins` array entry that has a matching `expo.extra.<key>` block (Sentry today, future Reanimated/Notifications) and asserts the duplicate fields don't drift. Today only Sentry is checked; a generic walker means future config-plugin additions get parity for free.

- Codify the `SENTRY_AUTH_TOKEN`-not-required-for-runtime invariant into a `lib/sentry-config.ts` doc comment + a `getSentryStatus()` field (`sourcemapsExpected: boolean`) so the diagnostics screen can show "events flowing but stack traces minified — add SENTRY_AUTH_TOKEN" when the bundle has a DSN but the deploy log shows no sourcemap upload step. Closes the gap between "runtime works" and "debugging works" without introducing a new secret-validation pass.

- Add a CI grep step that fails the build if `process.env` is passed whole (e.g. `process.env as Record<`, `(process.env)`) into any `lib/*-config.ts` resolver. Today only `lib/sentry-config.ts` is guarded by a structural test; promoting that test into a generic `.github/workflows/ci.yml` `grep -r "process\\.env as Record" lib/ && exit 1` line catches the same Metro/babel inlining footgun for future config helpers (cloudinary, posthog, etc.) without adding a per-module test.

- Promote `readSentryEnvFromProcess` into a shared `lib/expo-public-env.ts` helper exporting `readExpoPublicEnv(names: readonly string[])` that builds a literal-access object generator. Today every config module has to hand-roll the `EXPO_PUBLIC_X: process.env.EXPO_PUBLIC_X` pairs; a typed helper using a `const` tuple of var names + `Record<typeof names[number], string | undefined>` would compress the boilerplate and make adding a new EXPO_PUBLIC_ var a one-line change.

- Surface the bundle-inlining state on the diagnostics screen: render a "Sentry DSN inlined: ✅ / ❌" row sourced from `getSentryStatus()` so non-engineer testers can verify their TestFlight/preview build was produced from a CI run with the secrets — without touching devtools. Hide the row in production for users (gated on `__DEV__` or admin), but keep it on internal builds.

- Add a `scripts/verify-bundle-inlining.ts` post-build check that grep-tests `dist/_expo/static/js/web/*.js` for the literal `EXPO_PUBLIC_SENTRY_DSN:"https://` substring (or any documented secret) and fails the deploy when it's missing. Today CI prints `[set]` from the secret-status step, but a deploy still ships a bundle with `dsnPresent: false`; an automated bundle inspection closes the loop between "secret in CI" and "secret in artifact".

- Auto-resolve the `EXPO_PUBLIC_APP_VERSION` from the package.json `version` field at build time (read in `app.config.ts` and exported into `extra.expoPublicAppVersion`) so the Sentry release tag stays in sync with package.json without each deploy re-setting `EXPO_PUBLIC_SENTRY_RELEASE`. Removes one CI env var and prevents drift between the npm version and Sentry's release record.

- Mirror the Sentry rate-limiter (`MAX_EVENTS_PER_WINDOW`/`recentEvents`) onto `lib/analytics.ts:trackEvent` so a runaway loop firing `chat_opened` cannot exhaust PostHog's free 1M-events/mo cap in a few hours. Analytics #17 plans this; surfacing the same `__resetAnalyticsRateLimitForTests` shape now would let downstream tasks reuse the same testing pattern. Today every `trackEvent` delegates 1:1 to the SDK.

- Add a `getAnalyticsStatus()` snapshot mirroring `getSentryStatus()` (returning `{ ready, initialised, optedOut, keyPresent, environment, host, reason }`) so the future `app/settings.tsx` Diagnostics screen can render the same "events flowing / blocked because X" diagnostic without reimplementing the gate-walk. Today `lib/analytics.ts` only exposes `isAnalyticsReady()` which collapses every failure into `false`.

- Promote the SDK-loader pattern (`defaultLoader`, `InitOptions.loader`) shared by `lib/sentry.ts` and `lib/analytics.ts` into a generic `lib/lazy-sdk.ts:makeLazyLoader(modulePath, pickExport)` factory. Today both modules hand-roll an identical 5-line `defaultLoader` + override-via-options shape; centralising it would shrink each wrapper by ~10 lines and make the next telemetry SDK (Clarity, Crashlytics) a one-liner.

- Add a structural test that walks every call site of `trackEvent(` across `app/**/*.tsx` and `lib/**/*.ts` and asserts each first-arg literal appears in the `AnalyticsEventName` union in `lib/analytics.ts`. Today the type system already enforces this at compile time, but Analytics #5 will move event metadata to `lib/analytics-events.ts` — a regex-based parity test catches the case where someone renames the union but forgets to update the events file.

- Memoise the dynamic `import("posthog-react-native")` so a future `re-initAnalytics()` after a runtime config refresh (Settings → "Switch project") doesn't re-resolve the module from disk. Today `defaultLoader` issues a fresh dynamic import each call; one cached module promise at module scope mirrors how `getSharedRealtimeClient` memoises the WebSocket client.

- Generate `docs/powerbi/analytics-schema.md` directly from `lib/analytics-events.ts` via a `scripts/generate-analytics-schema.ts` step. Today `ANALYTICS_EVENTS` carries description+props but the Power BI doc (Analytics #14) will hand-copy the same data into a markdown table; auto-generating it in pre-commit (or as part of `npm run build`) closes the drift loop between the runtime taxonomy and the schema doc. Roughly 30 lines using `marked` + `fs.writeFileSync`.

- Tighten `AnalyticsEventDefinition.props` from `readonly string[]` to a per-event union literal type using `[...] as const` so consumers get autocomplete on the prop names too. Today the array is plain `string[]`, which means `ANALYTICS_EVENTS.collection_created.props[0]` is typed as `string` instead of `"visibility"`. Switching each entry to `props: ["visibility", "isPremium"] as const` keeps the runtime shape identical but lifts every prop name into the type system — useful for the future `validateEventProps(name, payload)` runtime guard.

- Add a `lib/analytics-validate.ts:assertValidProps(name, payload)` helper that walks `ANALYTICS_EVENTS[name].props` and asserts the runtime payload has only the allowed keys (in non-production environments — log a warning + drop unknown keys in prod). Wires `lib/analytics.ts:trackEvent` through this guard so a typo in a payload key fails loudly during development without polluting the production event stream. Mirrors how `scrubPII` sits between `captureException` and the SDK in `lib/sentry.ts`.

- Surface the taxonomy on the `app/settings.tsx` diagnostics screen as a read-only "Events captured by this app" list (description + sample props), gated on admin or `__DEV__`. Power users + reviewers asking "what does this app track?" get a one-tap answer that matches the privacy disclosure word-for-word, removing the need to grep the source. The list can be derived from `ANALYTICS_EVENTS` with zero hardcoded copy.

- Add a `__tests__/analytics-events-i18n-parity.test.ts` that asserts every event description has either an English-only entry or a complete set of translations in `lib/i18n-context.tsx`. Today descriptions are EN-only by design; if Analytics #18 (GDPR opt-out) ever surfaces them in the in-app privacy disclosure, a parity test prevents a partial translation from shipping to a non-English user.

- Mirror the `setSentryUser`/`setSentryOptOut` pattern by promoting `<AnalyticsProvider>` into a thinner wrapper that exposes `useAnalytics()` (returning `{ identify, reset, track, optedOut }`) so call-sites that need to fire ad-hoc identifies (e.g. premium upgrade post-payment) don't have to import `@/lib/analytics` directly. Today only `identifyUser`/`resetUser` are wired through the provider; downstream code paths importing the SDK module directly bypass the React lifecycle.

- Debounce the `identifyUser` call in `<AnalyticsProvider>` by 500ms so a rapid premium-flag flip + language-switch within the same render cycle (or React 19 Strict-Mode double-mount) doesn't double-charge the PostHog `identify` quota. Today the `useEffect` fires synchronously on every dep change; debouncing with `setTimeout` + cleanup is ~5 LOC.

- Surface `lastIdentifyAt` + `lastIdentifyTraits` on a `getAnalyticsSnapshot()` helper so the diagnostics screen can render "Last identify: 2 minutes ago — language=ru, isPremium=false" without re-deriving from the React tree. Useful when debugging "why aren't my user traits flowing" complaints from analytics consumers.

- Add a `__tests__/analytics-provider-strict-mode.test.ts` that mounts `<AnalyticsProvider>` under `<React.StrictMode>` (with a stubbed analytics SDK) and asserts `identifyUser` fires exactly once per real user change — not twice from the dev double-render. Today the provider relies on `useEffect`'s default React 19 dedup; an explicit test pins the invariant down before it regresses with a future React upgrade.

- Promote the `<AnalyticsProvider>`-must-be-inside-`<PremiumProvider>` invariant into a runtime check via a typed `useAnalyticsContext()` hook that throws a descriptive error if invoked outside the provider tree. Today the provider relies on the order in `_layout.tsx` and a structural test catches drift in CI; a runtime guard surfaces the mistake during dev with a stack trace pointing at the missing wrapper.

- Extend `signup_completed` to fire from the OAuth path (`signInWithProvider`) too — today only `verifyEmailOtp` triggers it, so a freshly-created Google/Apple user is invisible to the funnel. Cleaner approach: detect inside `onAuthStateChange` when a SIGNED_IN event arrives with a user whose `created_at` is fresh AND whose user ID hasn't been seen this session (track in a `Set` to dedup the OAuth+OTP "first SIGNED_IN" double-emission). Mirrors how the `setSentryUser` block in the same file already differentiates user states.

- Promote the 5-minute freshness window from a hard-coded constant to `EXPO_PUBLIC_SIGNUP_FRESHNESS_WINDOW_MS` (default 300_000) so QA can dial it up to 60min for staging tests where `verifyOtp` race conditions may delay the event past 5 min. Mirrors `EXPO_PUBLIC_PROFILE_CACHE_TTL_MS` from `lib/social-context.tsx`.

- Add a `__tests__/analytics-language-roundtrip.test.ts` that mounts `<I18nProvider>` and toggles `setLanguage("en")` → `setLanguage("ru")` → `setLanguage("ru")` and asserts `trackEvent` fires exactly twice (no double-fire on the no-op repeat). Today the inequality guard handles this; an integration test pins it down before someone "simplifies" the guard away.

- Promote `isFreshlyCreatedUser` into a generic `isWithinDuration(iso, durationMs, now?)` helper in `lib/time-helpers.ts` (or extend `lib/env.ts`) so other "is recent" checks (Sentry release-time staleness, marketplace listing freshness) reuse one tested implementation. Today every screen that needs "X minutes ago" rolls its own `Date.parse + arithmetic`; centralising prevents off-by-ms bugs.

- Surface the `signup_completed` event in `__sentryStatus`/`__sendSentryTestError`-style devtools globals via `globalThis.__simulateSignupEvent()` that calls `trackEvent("signup_completed", { method: "manual_test" })`. Lets a developer verify the PostHog wire is live end-to-end without forging a fresh OTP flow. Gate behind `__DEV__`.

- Mirror the `item_photo_attached` "edge detector" pattern in a generic `useTransitionEvent(prev, next, fire)` hook so future "first X" funnels (e.g. `first_listing_published`, `first_friend_added`) don't each re-implement the `(!hadXBefore && hasXNow)` check. Today the `hadPhotosBefore` snapshot is inline in `app/item/[id].tsx`; lifting it costs ~10 LOC and the analytics taxonomy gets cheaper to extend.

- Add a counterpart `item_photos_replaced` event when an item already has photos and the user picks a different set — captures gallery-rotation behaviour separately from first-photo onboarding. Today only the no→yes edge is tracked; replacements are invisible. Useful for the "are users curating their items?" question in Power BI.

- Move the `collection_created` payload into a shared `buildCollectionAnalyticsProps(collection)` helper in `lib/analytics-helpers.ts` so future events (`collection_shared`, `collection_archived`) automatically carry the same canonical shape (visibility/isPremium/hasCover/templateId). Today each call site hand-rolls the payload literal; centralising prevents drift like a future event omitting `isPremium`.

- Add an `analytics_events_bundle_size_check` step to CI that fails when an `app/**.tsx` file imports `@/lib/analytics-events` directly (only `lib/analytics.ts` should). The taxonomy module is light today, but as it grows with `description` strings the bundle would balloon; gating imports keeps the runtime module slim and forces docs/Power BI tooling to consume it server-side.

- Auto-derive `hasPhoto` / `hasCover` / `hasDescription` boolean props in a single `summarisePayload(item)` so future analytics events on items don't each re-derive the same booleans. Today `item_added.hasPhoto` is hand-rolled from `uploadedPhotos.length > 0`; one helper means new events get the same logic for free, and the implementation can be unit-tested in isolation.
