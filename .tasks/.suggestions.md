## Upgrade suggestions

### Chat DB (2026-05-16)

- **Cursor pagination for `fetchMessagesForChat`** — it currently `select=*&order=created_at.asc` with no `limit`, so a long-lived conversation pulls the entire history into memory on every hydrate. Add `&limit=50` + a `created_at` cursor (or PostgREST `Range` header) and load older pages on scroll-up. Bounds memory and free-tier egress.

- **Realtime reconnect backfill** — while the WebSocket is down, `subscribeToInbox` misses INSERTs and only `refreshFromCloud` (mount-time) closes the gap. On `SUBSCRIBED` after a drop, fetch messages `created_at=gt.<last-known>` per active chat so no message is silently lost during a flaky connection.

- **First-class conversations table** — replace the free-text `chat_id` with a `conversations(id uuid, user_a uuid, user_b uuid, UNIQUE(user_a,user_b))` table and a `chat_messages.conversation_id uuid REFERENCES conversations(id)`. A real FK + unique participant pair is stronger than the derived-string CHECK and unlocks per-conversation metadata (muted, archived, last_message_at) without re-deriving from messages.

- **Functional idempotency test with a mocked fetcher** — the new ignore-duplicates path is only covered structurally. Add a `__tests__/supabase-chat-idempotency.test.ts` that injects a fake `fetcher` returning `200 []` and asserts `sendMessage` resolves to the synthesized message (not `null`), plus a fake returning a populated row for the happy path.

- **`chat_messages.client_id` + server uuid split** — keep the server `id uuid DEFAULT gen_random_uuid()` authoritative and add a separate indexed `client_id uuid` (the composition-time key) with a UNIQUE constraint for idempotency. Decouples "who owns the primary key" from "idempotency key", which is the canonical chat-storage pattern and avoids ever trusting a client-supplied PK.

- **SPA fallback regression guard** — Add a CI step that runs `npm run build` then asserts (a) `dist/404.html` contains `location.replace`, (b) `dist/index.html` contains `data-spa-restore`. This catches a future contributor accidentally reverting the post-build script (e.g. swapping back to `cp dist/index.html dist/404.html`) before the change reaches production iOS users.

- **Service worker for instant deep-link load** — Even with the sessionStorage redirect, the user sees a flash of the redirect page on iPhone. Registering a service worker (Workbox `NetworkFirst` + `NavigationFallback` to `/collectables/index.html`) would let cached deep links resolve to 200 directly and remove the redirect entirely on repeat visits.

- **Sentry breadcrumb for SPA URL restore** — Drop an `addBreadcrumb("spa-redirect", { from, to })` inside the restore script (gated on `__sentry_loaded__`) so Sentry sessions show exactly which deep-link URLs hit the 404 fallback. Useful for ranking which routes deserve a real static export.

- **Static export of common collection IDs** — Expo Router could pre-render the top-N most-shared collections at build time (e.g. by reading from Supabase) so their URLs return real 200s without the redirect dance. Reduces flicker and helps SEO for public collections.

- **Migrate baseUrl to a single source** — `app.json` declares `expo.experiments.baseUrl`, `lib/url-helpers.ts` infers it at runtime, and `scripts/build-spa-fallback.ts` re-reads it from app.json. A single `lib/base-url.ts` exporting a const + a helper that all three import would eliminate the drift risk if the deployment path ever changes (e.g. moving off the `1337ninja1337.github.io/collectables` prefix).

### Power BI starter `.pbit` (2026-05-19)

- **CI regenerate-and-diff guard** — Add a CI step `npm run build:powerbi && git diff --exit-code docs/powerbi/Collectables-Starter.pbit` so a PR that edits `queries.m`/`measures.dax` without regenerating the binary fails loudly. The test already asserts byte-parity, but a CI diff makes the *fix* (`npm run build:powerbi`) obvious in the failure message and prevents a stale committed binary slipping through if the test file is skipped.

- **Add a line-chart "DAU over time" visual** — the template ships five `card` visuals; the highest-value Power BI view is DAU on a date axis. Add one `lineChart` visualContainer with `occurred_at` on Category and `DAU` on Y to `buildReportLayout()` so the funnel story is visible the moment the model loads, not just scalar cards.

- **Property-key expansion sample query** — `properties` lands as raw jsonb text. Ship a second commented M query (`queries-properties.m`) that demonstrates `Json.Document` + `Table.ExpandRecordColumn` for the common keys (`signup_completed.method`, `listing_created.mode`, `premium_activated.source`) so analysts don't have to reverse-engineer the taxonomy from `lib/analytics-events.ts`.

- **`.pbit` schema-version matrix doc** — compatibilityLevel 1550 / `powerBI_V3` is a conservative choice but some very old Power BI builds reject it and some very new ones rewrite it on save. A short table in `docs/powerbi/README.md` mapping Power BI Desktop versions → expected open behaviour (opens / opens-and-upgrades / falls back to copy-paste) would set expectations and reduce "the template won't open" support noise.

- **Generate the connection-doc DAX from `measures.dax`** — `docs/powerbi-connection.md` §5 hand-duplicates the seven measures and a parity test guards drift. Instead, have a docs build step inject the parsed `measures.dax` blocks into the markdown between sentinel comments, making `measures.dax` the single source for the doc, the `.pbit`, and the copy-paste fallback.

### analytics_events RLS test (2026-05-19)

- **CI job that runs the SQL test against an ephemeral Postgres** — spin up `postgres:16` in a CI service container, apply `supabase/migrations/*.sql` + create the `anon`/`authenticated` roles, then run `supabase/tests/analytics_events_rls.sql` and fail the job on any `RAISE EXCEPTION`. Converts the currently-manual RLS proof into an automated gate so a permissive policy can never reach production silently.

- **Generalise into a per-table RLS harness** — `chat_messages`, `chat_reads`, `marketplace_listings` all have their own RLS posture. Extract the anon/authenticated probe loop into a reusable `supabase/tests/_rls_helpers.sql` (a function taking a table name + expected-visible predicate) and add one script per sensitive table so the whole data layer's RLS is regression-tested the same way.

- **Negative test for the `analytics-mirror` service-role path** — the Edge Function inserts with the service-role key. Add a check that a request *without* the `x-posthog-webhook-secret` (or with a wrong one) is rejected 401 and inserts nothing, complementing the read-side RLS test with a write-side authz test.

- **Assert the `properties` jsonb can't be used for RLS bypass** — add a probe that, as `authenticated`, attempts a `SELECT ... WHERE properties @> '{}'` and a function-call form to confirm no SECURITY DEFINER helper or view inadvertently exposes the table; documents that the only read path is the service role.

- **Pin role names via a doc-comment contract test** — the script hard-codes `anon`/`authenticated`. If Supabase ever renames the default roles the test would pass vacuously (SET ROLE errors caught as `insufficient_privilege`). Add an explicit `SELECT 1 FROM pg_roles WHERE rolname IN ('anon','authenticated')` precondition that aborts with a clear message if the roles are missing.
