## Upgrade suggestions

- **Route `activatePremium`/`cancelPremium` through the server (BE-22c follow-up)** — now that load-time `validate` is server-authoritative, a *locally* activated premium (the Activate button only mutates local `PremiumState`) is silently downgraded on the next authed load, because `mergePremiumState` lets the server's `isPremium=false` win. Wiring `activatePremium` to `cloudValidatePremium("activate")` (and a future `cancel` action) before/after the optimistic local flip closes that gap so an activation actually survives a reload.

- **Re-validate premium on app foreground / session refresh, not just initial load** — the BE-22c server pull fires once per `storageKey` mount; a long-lived session whose `current_period_end` lapses mid-use keeps gating paid features until the next cold start. A lightweight `AppState` "active" listener (or hooking the auth `TOKEN_REFRESHED` event) that re-runs `cloudValidatePremium("validate")` would expire a churned subscription promptly without a relaunch.

- **Surface a "couldn't verify premium" soft indicator when the server pull returns null** — `cloudValidatePremium` returning `null` (offline/transient) is intentionally invisible so a payer isn't downgraded, but it also means a *stale* cache is shown with no signal. A subtle "last verified …" timestamp on the settings premium card (set only on a successful validate) would let a user tell a genuinely-active entitlement from a never-server-confirmed local one.

- **De-dupe the premium load + server-pull into a single reducer transition** — the load effect now calls `setState` twice (cache, then merged server truth), so a fast server response can cause a brief `false→true` flicker for a payer whose cache was empty. Folding both into one `setState` (await the validate before the first commit when a token is already present, with a short timeout fallback to the cache) would remove the flicker without sacrificing the offline-keeps-cache guarantee.

- **Integration test for premium-context server-truth merge against a fake `cloudValidatePremium` (BE-35 seed)** — the BE-22c wiring is only structurally guarded; a render test (or a thin extraction of the merge step into a pure `applyServerValidation(prev, validation)` helper) that asserts a server `isPremium=false` downgrades a cached payer while a `null` preserves the cache would lock the LWW semantics that the regex test can't observe at runtime.

- **Idempotency / replay guard on `validate-premium` activate (BE-22b follow-up)** — `action:"activate"` is idempotent only against an *already-active* row; two near-simultaneous activates on a *lapsed* row both pass the `isActive` gate and both upsert, the last-writer's `activated_at` winning. Threading an `Idempotency-Key` header (or a conditional upsert that only resets `activated_at` when the row wasn't active) would make repeated taps on the "Activate premium" button deterministic.

- **A real IAP/receipt verification step before `validate-premium` activates** — today `action:"activate"` trusts any authenticated caller, so the server is authoritative about *storage* but not about *payment*. Wiring the function to verify an App Store / Play / Stripe receipt (passed in the body) before flipping `status='active'` closes the "any signed-in user can mint premium" gap that the BE-22a table was introduced to fix.

- **`pg_cron` nightly expiry sweep for lapsed subscriptions (pairs with BE-27)** — `validate-premium` only expires a row when the owner happens to call "validate"; a churned user who never opens the app keeps `status='active'` (with a past `current_period_end`) forever in the table. A nightly `UPDATE subscriptions SET status='expired' WHERE status='active' AND current_period_end < now()` keeps server-side analytics/entitlement queries honest without waiting for a client round-trip.

- **Emit a `premium_validated` / `premium_expired` analytics event from the function** — the Edge Function is the one place that knows the *server* truth about an entitlement transition (active→expired on lazy expiry, or a fresh activate). Forwarding that to the analytics sink (like `analytics-mirror`) would give a churn/renewal funnel that the client-side `premiumActivated` toast can't, since the client never sees the lazy server expiry.

- **Integration test exercising `cloudValidatePremium` against a fake `fetcher`/`tokenProvider` (BE-35 seed)** — `cloudValidatePremium` is currently only structurally guarded; a node test that mocks `fetcher` to return a `{ isPremium, activatedAt, expiresAt }` body and asserts the wrapper POSTs the right URL/body, coerces via `parseValidation`, and returns `null` on `!res.ok`/no-token would catch a regression the regex tests can't (e.g. a future refactor that drops the token guard). This is the BE-22 slice of the broader BE-35 "fake fetcher for every cloud wrapper" item.

- **Cap `cloudAcceptFriendRequest` retries on a hard server error (BE-21 follow-up)** — the wrapper treats `res.ok || res.status===409` as terminal, but a `500` (e.g. `function misconfigured`, or the function not yet deployed) returns `false` forever, so the `accept-request` mutation re-flushes on every remount until the function is fixed. A bounded retry/backoff (or surfacing a one-time "couldn't accept, try again later" toast after N failures) would stop a silently-undeployed function from pinning the queue.

- **Replace the `accept-friend-request` self-accept guard with a server-side dedup of the "accept your own outgoing request" case** — the function 400s when `fromUserId === toUserId`, but it doesn't catch the subtler "I already sent A→B, now I 'accept' B (who never requested me)" — `accept_friend_request` correctly raises `P0002` (no inbound), so the client should distinguish 409-no-inbound from a true accept and not optimistically flip the local relationship to `friend` in that case (today `addFriend` always inserts the local row first).

- **pgTAP test for `accept_friend_request` concurrency** — add a `supabase/tests/` case that opens two sessions, both accepting the same inbound request, and asserts exactly one reverse row exists + no error leaks; the `FOR UPDATE` lock is the whole point of BE-21 but only the structural SQL shape is currently guarded (the Docker-backed supabase-test CI can actually exercise the lock).

- **Emit a distinct `friend_accepted` analytics event on the accept path** — `addFriend` fires `friend_requested` for both a fresh request and an accept, so the funnel can't tell "sent" from "accepted". Now that `addFriend` already computes `isAccept`, branching the `trackEvent` name (and threading it through `analytics-shapes`) is a one-line behavioural win for retention analysis.

- **Realtime-propagate the reverse `friend_requests` INSERT to the sender** — when B accepts, A's client only learns it became "friends" on its next pull/remount because the realtime subscription filters INSERTs narrowly. Subscribing to `friend_requests` INSERTs where `to_user_id = me` (BE-19 UPDATE/DELETE work is adjacent) would flip A's relationship chip to "friend" live, matching the marketplace claim's realtime UX.

- **Make the `SyncStatusPill` tappable to force a flush (and show last-synced time)** — the BE-16 pill is read-only; a tap could bump the shared `refreshTick`/re-run the flush effects and, when the queue is empty, briefly show "Synced just now" before fading. Turns a passive indicator into an explicit "retry now" affordance for users on flaky connections who don't want to wait for the next remount.

- **Animate the pill in/out instead of a hard mount/unmount** — `SyncStatusPill` returns `null` the instant `total` hits 0, so a fast flush makes it flicker. A short `LayoutAnimation`/`Animated` fade (or a 1.5s "Synced" grace state) would make the transition feel intentional rather than glitchy, matching how mainstream apps surface sync state.

- **Surface a per-context breakdown on long-press** — the pill collapses collections + social + chat into one number. A long-press tooltip ("2 items, 1 friend request, 3 messages") reusing the same three `pendingSyncCount` values would help a power user understand *what* is stuck without opening devtools, and is a pure-render addition (no new context plumbing).

- **Add a `pendingSyncCount`-aware "don't quit yet" guard on web** — when `total > 0`, register a `beforeunload` handler warning the user that unsynced edits will be retried next launch but aren't in the cloud yet. The count already lives in context; this closes the data-loss-perception gap when someone closes the tab mid-flush.

- **Unit-test the chat `pendingSyncCount` memo against a real `pendingByChatId` snapshot** — the BE-16 structural test pins the wiring regex, but nothing asserts that two messages across two chats yield `2`. A small React-Testing-Library (or reducer-extraction) test mirroring the other context suites would lock the multi-group sum behaviourally, not just structurally.

- **Network-reconnect trigger for the social/collections flush effects** — both flush effects currently re-fire only on `ready`/`user` (and `refreshTick` for collections), so a queue parked while offline isn't retried until the next remount or manual refresh. Subscribing to an online/offline signal (NetInfo on native, `window.online` on web) and bumping a shared `refreshTick` would make reconnect re-deliver parked writes automatically, closing the main gap left by BE-13c/13d.

- **Unit-test the `SocialProvider` queue wiring, not just the pure module** — `__tests__/pending-social.test.ts` covers `lib/pending-social.ts` exhaustively, but nothing asserts the *context* enqueues on a failed `sendFriendRequest`, dequeues on success, or flushes on reconnect. A React-Testing-Library suite with mocked cloud fns (mirroring the existing context tests) would pin the integration the structural tests can't.

- **Generalise the social-graph "deleted profile IDs" into a real tombstone + soft-delete sweep (BE-15)** — `deletedProfileIds` is an ad-hoc client-only tombstone list; the BE-13 engine and a `deleted_at` column could turn it into a uniform LWW soft-delete applied across collections/items/profiles, so an admin delete propagates through the same queue instead of a bespoke filtered-out set.

- **Cap and age out persisted pending queues** — `pendingSocial`/`pendingCollections`/`pendingItems` grow unbounded while offline and are serialised to AsyncStorage on every change. A max-size guard (drop oldest, surface a toast) plus a per-entry `queuedAt` so a write that never succeeds is eventually abandoned would prevent a corrupt/permanently-rejected mutation from wedging the flush loop forever.

- **pgTAP cases asserting the BE-7 CHECKs actually reject bad values** — `20260619_integrity_checks.sql` adds the visibility/condition/no-self CHECKs, but no test inserts `visibility='draft'` / `condition='mint'` / a self-request and expects `23514` (check_violation). A `03_integrity_checks.sql` pgTAP (or cases in `01_core_tables_rls.sql`) would lock in the enforcement, mirroring how `02_fk_cascade.sql` proves the FK behaviour.

- **Promote `visibility` and `condition` to native PostgreSQL ENUM types** — the values are validated by string CHECKs in two places (base schema inline + the BE-7 backfill). A `CREATE TYPE collection_visibility AS ENUM (...)` / `item_condition AS ENUM (...)` would centralise the allowed set, give the REST API typed columns, and remove the risk of the CHECK list drifting from `lib/types.ts`. Pair with a `pg_enum`-vs-TS-union parity test.

- **Parity test: DB CHECK enums vs the TS literal unions** — `lib/types.ts` defines `visibility`/`condition` as TS unions and the migrations repeat the same literals in SQL. A test that extracts both and asserts they match would catch a future value added to one side but not the other (e.g. a new `condition: "poor"`), the same drift risk the SEC-ADMIN-1 column list calls out.

- **Generalise the BE-6/BE-7 "ensure constraint exists" guards into one helper migration** — both migrations hand-roll `DO $$ … IF NOT EXISTS (pg_constraint …) $$` blocks. A small reusable `pg_temp.ensure_check(table, name, expr)` / `ensure_unique(...)` (like BE-6's `ensure_cascade_fk`) would shrink future integrity migrations and keep the guard logic consistent.

- **Audit chat/marketplace tables for the same missing-constraint gap** — BE-6/BE-7 only normalised the four core tables. `chat_messages`, `chat_reads`, `marketplace_listings`, `marketplace_transfers` were also created from their own migrations; a one-pass review confirming their CHECKs/uniques/FKs are present on the live (hand-created) project — and a backfill migration if any are missing — would close the gap class entirely rather than table-by-table.

- **pgTAP assertion that every core FK actually carries `confdeltype = 'c'`** — BE-6's migration sets ON DELETE CASCADE, and `02_fk_cascade.sql` proves the *behaviour*, but nothing asserts the catalog metadata directly. A small pgTAP case querying `pg_constraint.confdeltype` for the six core FKs (expecting `'c'`) would catch a future migration that re-adds a FK with the wrong action even before a delete is attempted, mirroring the chat `chat_id_integrity` CHECK test.

- **Generalise the `ensure_cascade_fk` pg_temp helper into a reusable migration snippet** — chat/marketplace tables have their own FKs to `auth.users`/`collections` defined in their migrations; if any were hand-created without CASCADE they'd share BE-6's problem. Extracting the drop-then-readd helper into a documented snippet (or a shared `_helpers.sql` applied first) would let a single follow-up normalise `chat_messages`, `chat_reads`, `marketplace_listings`, and `marketplace_transfers` FKs the same way.

- **CI lint that flags a new `REFERENCES` in a migration without an explicit `ON DELETE` action** — the default `NO ACTION` is exactly the foot-gun BE-6 had to repair. A matcher (mirroring `check-migration-docs`) scanning `supabase/migrations/*.sql` for `REFERENCES … ` clauses lacking an `ON DELETE` would force every future FK to declare its delete behaviour at authoring time.

- **`delete-account` Edge Function integration test against the from-empty CI DB** — BE-25 verifies the cascade in pgTAP, but the actual Edge Function (which calls `auth.admin.deleteUser`) isn't exercised end-to-end. A test invoking the function with a service-role client against the `supabase db start` database would prove the function + FK cascade work together, not just the SQL layer.

- **Schema-drift guard comparing base-schema FKs to the live catalog** — BE-6 exists because the hand-created live tables drifted from the committed schema. A periodic (or CI) check dumping `pg_constraint` delete actions and diffing them against the canonical list in the base-schema migration would surface any future manual dashboard edit that re-introduces drift before it bites a delete path.

- **Lint that fails when any `id:` mint in `lib`/`app` is a non-uuid template literal** — BE-5 fixed items by hand, but the next entity (a new `addX`) could re-introduce a `${slug}-${Date.now()}` server key. A CI matcher (mirroring `check-migration-docs`) flagging `id:` assignments built from template literals on persisted entities — with an allowlist for client-only optimistic ids like the reaction `tmp-` placeholder — would stop the legacy-id class from recurring.

- **Proactively re-upsert ALL legacy-rewritten items, not just on hydrate** — `normalizeOwnItemIds` runs at sign-in hydrate, but an item created offline mid-session before this build won't pass through hydrate. A one-time storage sweep on app launch (migrate-then-persist, gated by a `collectables-item-uuid-migrated-v1` flag) would catch every cached legacy id exactly once and remove the per-hydrate scan cost.

- **Add a pgTAP / structural check that `items.id` rejects a non-uuid insert** — the migration declares `items.id uuid` but nothing asserts a `slug-123` POST is refused (22P02). A pgTAP case in `01_core_tables_rls.sql` (or a new `03_id_integrity.sql`) inserting a bad id and expecting the cast error would lock in BE-5's server-side guarantee, matching the chat `chat_id_integrity` CHECK test.

- **Generalise `normalizeOwnItemIds` into a shared `normalizeOwnEntityIds`** — collections, wishlist items, and future synced entities share the "rewrite my own legacy non-uuid ids" need. Promoting the helper to take an `ownerKey` accessor (so it works for `Collection.ownerUserId` too) and reusing it in the collections hydrate path would pre-empt the same bug class for any entity that predates its uuid fix.

- **Dedupe items by `(collectionId, title, createdAt)` when a legacy and a uuid row both exist** — if a pre-fix item was partially synced under one id and rewritten under a new uuid, a cross-device merge could surface both. A content-hash dedupe in `mergeItemsFromCloud` (keep the uuid-keyed row, drop the legacy-id duplicate) would clean up any straddled rows from the transition.

- **Promote the BE-23 self-check to module load so it fails at cold-start, not first-request** — `assertServiceRoleKey` currently runs inside the request handler, so a misconfigured secret only surfaces when the first call arrives. Hoisting it to top-level module scope (a thrown error there crashes the function boot) makes a swapped key fail the moment the function deploys/cold-starts, which is closer to the task's "fails loudly at deploy" intent and matches how Supabase surfaces boot errors in the dashboard.

- **Add a `lib/check-edge-fn-self-check.ts` lint asserting every `supabase/functions/*/index.ts` imports the claim guard** — the three current functions are wired by hand; a CI lint (mirroring `check-migration-docs`) that fails when a new Edge Function constructs an admin client with `SUPABASE_SERVICE_ROLE_KEY` without first calling `assertServiceRoleKey` would stop a future function shipping the foot-gun. Pair it with the structural test already in `service-role-claim.test.ts`.

- **Verify the JWT `iss`/`ref` claims match the project, not just `role`** — a service-role key from a *different* Supabase project still carries `role: service_role` and would pass today's check while pointing the admin client at the wrong DB. Decoding `ref` (project ref) and comparing it to the host in `SUPABASE_URL` would catch a cross-project key paste — a subtler misconfiguration than the anon/service swap.

- **Optionally signature-verify the service-role JWT against `SUPABASE_JWT_SECRET`** — the current decode is claim-only (we own the secret, so tampering isn't the threat). But if `SUPABASE_JWT_SECRET` is also a function secret, an HMAC check would additionally catch a *truncated/corrupted* key that happens to still parse as JSON, turning a silent runtime failure into a named config error.

- **Surface a deploy-time smoke test in the functions CI** — add a workflow step (or a `supabase functions serve` + `curl` in the BE-31 Docker harness) that boots each function with a deliberately-wrong `SUPABASE_SERVICE_ROLE_KEY` (the anon key) and asserts it returns HTTP 500 `misconfigured`, proving the guard fails closed end-to-end rather than only in the structural source-scan test.

- **Land the BE-12b executable pgTAP (`supabase/tests/01_core_tables_rls.sql`) to back the BE-12a structural pins** — the verification test proves the policy *text* is shaped right, but only role-switched pgTAP in the BE-31 Docker harness proves a logged-in stranger actually gets zero rows from a private collection. Seed two synthetic `auth.users`, set `request.jwt.claims`, and assert cross-tenant SELECT/INSERT/UPDATE/DELETE deny.

- **Generalise the "RLS-enabled-implies-policy-present" check across ALL migrations, not just the core four** — BE-12a hardcodes the four core tables. A repo-wide lint that pairs every `ENABLE ROW LEVEL SECURITY` with either a `CREATE POLICY` on the same table or an allowlisted default-deny comment (like analytics) would catch a future table that enables RLS and forgets its policy, locking the app out.

- **Add a `USING (true)` / `WITH CHECK (true)` leak lint over `supabase/migrations/*`** — BE-12a only forbids bare-true SELECT policies on the core tables. A blanket check across every migration would flag any accidentally over-broad policy (e.g. a copy-pasted `USING (true)` on a write path) before it ships.

- **Unit-test the SECURITY DEFINER helpers in isolation via pgTAP** — `is_friend`/`is_visible_to`/`can_view_collection`/`is_admin` encode the entire visibility model; seeding friend_requests/collections and asserting their boolean output (no role switching needed) is a cheap, deterministic pgTAP layer that localises a logic bug to the helper rather than a policy.

- **Cross-check the MANUAL-TASKS leak-check SQL against the actual gated tables** — BE-12a asserts the leak check names collections/items/friend_requests, but if a future migration gates a fifth table the doc check won't notice it's missing from the leak-check block. Derive the expected table list from the `ENABLE ROW LEVEL SECURITY` statements and assert each appears in the documented leak check.

- **Seed `profiles.is_admin = true` for the founder row and retire the client allowlist (BE-11b follow-up)** — now that `social-context` reads `selfProfile.isAdmin`, the `1337antoxa`/email literals in `lib/social-context.tsx` are only an offline fallback. A one-time `UPDATE profiles SET is_admin = true WHERE username = '1337antoxa'` migration (documented in MANUAL-TASKS.md) plus removing the hardcoded literal would stop shipping the admin identity in plaintext to every client.

- **Hydrate `selfProfile` from the cloud on sign-in so the server `is_admin` flag actually wins** — `social-context` derives admin from `profiles`, but if the self row is sourced from the seed/offline cache the new flag is never present and the gate silently falls back to the allowlist. A fetch of the authed user's own profile row (mirroring the bug-2c displayCurrency hydration in `collections-context`) would make the server flag authoritative in practice, not just in code.

- **Add a CI guard that no `*-shapes.ts` body builder writes a REVOKE-protected column** — `upsertProfileBody` correctly omits `is_admin`, but the invariant is only enforced by one structural test. A lint that cross-references columns with `REVOKE UPDATE(...)`/`REVOKE INSERT(...)` in `supabase/migrations/*` against the keys returned by every body builder would catch a future builder that re-adds a server-authoritative column.

- **Extend the schema-parity test (BE-3/BE-37) to assert read-only columns appear in `select`/`DbProfile` but never in a body builder** — parity today only checks columns *exist*; pairing it with the REVOKE list would prove the read path covers `is_admin`/future server columns while the write path excludes them.

- **Surface an admin badge/affordance gated on the synced `isAdmin` flag** — with `UserProfile.isAdmin` now threaded through, the profile/settings UI could show a small "admin" indicator and unlock the delete-profile control consistently, instead of the gate living only inside the `deleteProfile` callback. Keep it behind the same memo so offline fallback still works.

- **Generalise the BE-11a anon/leak RLS check into executable pgTAP (BE-12)** — `__tests__/core-tables-rls-migration.test.ts` only asserts the policy *text* exists; it can't prove a logged-in user actually can't read a stranger's private collection. A `supabase/tests/rls.sql` that `SET ROLE`s to two synthetic users and asserts cross-tenant SELECT/INSERT/UPDATE all deny would run inside the BE-31 scratch-DB harness and catch a policy that's present but wrong.

- **Add a CI lint that every RLS-enabled table has at least one policy (or a documented default-deny)** — `analytics_events` is intentionally policy-less (default-deny), but a future migration that runs `ENABLE ROW LEVEL SECURITY` and forgets the `CREATE POLICY` would silently lock the app out of a table. A grep-style check pairing each `ENABLE ROW LEVEL SECURITY` with either a policy or an allowlisted "default-deny" comment would catch that class of outage before merge.

- **Move the `1337antoxa`/email admin allowlist fully server-side once BE-11b lands** — BE-11a added `profiles.is_admin` + `is_admin()` but the client still hardcodes the allowlist in `lib/social-context.tsx`. After BE-11b reads the flag, a one-time seed/migration that sets `is_admin = true` for the founder row (and removing the literal from the bundle) closes the gap where the admin identity ships in plaintext to every client.

- **Add a `search_path`-injection guard test for SECURITY DEFINER functions** — BE-11a pins `SET search_path = public` on all four helpers and the test counts them, but a future definer function added without the pin is a privilege-escalation vector. A structural check that every `SECURITY DEFINER` in `supabase/migrations/*` is accompanied by a `SET search_path` would make the invariant repo-wide, not just for this migration.

- **Wire `is_visible_to`/`can_view_collection` into the app's cloud read path** — the helpers now enforce friend/shared visibility server-side, but the app still sources social collections from `seedSocialCollections` (per CLAUDE.md). A follow-up that fetches friends' collections via the REST API (now safely RLS-gated) would let real shared collections replace seed data without leaking private rows.

- **Have the migration-docs check (BE-32) also flag stale/orphaned MANUAL-TASKS.md sections** — today it only fails when a migration lacks a section. The reverse (a `## <timestamp>_*.sql` H2 whose file was renamed/deleted) leaves dead manual-apply SQL that drifts from reality. Restrict the reverse check to H2s matching the migration-filename pattern so prose sections (Edge Functions, the free-plan setup block) aren't touched.

- **Assert each migration's MANUAL-TASKS section actually contains the applied SQL, not just a heading** — BE-32 only checks a heading exists, so an empty `## 20260601_foo.sql` with no body would pass. A follow-up could require either a fenced ```sql block or a recognised "no manual step" sentinel under each migration heading.

- **Lint that every migration heading's documented SQL stays in sync with the file** — diff the `CREATE TABLE`/`ALTER` statements named in the MANUAL-TASKS section against the actual `.sql` so a migration edited after its doc was written can't silently diverge (e.g. a new column added to the file but never to the manual-apply block).

- **Surface the migration-docs and hex checks as a single `npm run lint:all` aggregate** — `lint:ci` chains tsc + hex + migration-docs + test, but there's no quick "just the static lints" entry point for local pre-commit. A `lint:all` that runs only the fast file-scanning checks would speed the inner loop.

- **Add a git pre-commit hook (husky or a committed `.githooks/`) running `lint:migration-docs` + `lint:hex`** — catching an undocumented migration at commit time is far cheaper than a red CI run minutes later; the checks are sub-second and filesystem-only.

- **Extend the BE-3 parity test to the row-reader shapes (`toUserProfile`/`toCollection`/`toItem` in `lib/supabase-profiles.ts`), not just the writer builders** — BE-3 derives expected columns from the URL/body builders, which covers writes + filters but not every column the app *reads back*. A column that's selected via `select=*` and consumed by a `toX` mapper (e.g. a future `profiles.is_admin`) wouldn't be in the builders, so a missing migration column would slip through. Parsing the `row.<col>` accesses in the `toX` functions and folding them into the parity set would close the read side.

- **Make BE-3 fail on *extra* migration columns too (bidirectional parity)** — the test asserts builder columns ⊆ migration columns, but not the reverse, so a column the migration declares that no builder/reader uses (dead schema) goes unnoticed. An allowlist of server-only columns (`id`, `created_at`, …) plus an assertion that every other migration column is referenced somewhere would catch schema that drifts ahead of the client.

- **Add a migration-ordering lint covering *all* cross-table references, generalising BE-1's ordering guard** — BE-1's test only checks the base schema sorts first. A stronger script would scan every `*.sql` for `REFERENCES public.<t>` / `FROM public.<t>` / `public.<t>` policy refs and assert the migration that `CREATE TABLE`s `<t>` sorts earlier — catching the next 20260424-style "relation does not exist" before it hits the Supabase Preview.

- **Move the Supabase Preview branch off the from-empty replay or document why it's expected** — the Preview check replays *all* migrations from empty on first PR creation, which is what surfaced the ordering bug (good) but also means any historical migration that assumed dashboard-created tables will keep failing until backfilled into a base schema. A short `docs/` note (or a `supabase/seed.sql` guard) recording that the base schema must precede every dashboard-era migration would save the next contributor the same debugging loop.

- **Add a `supabase db reset` smoke job to CI (BE-31 precursor)** — the parity + ordering tests are static-text assertions; they can't catch a genuine SQL syntax error or a type mismatch (e.g. `uuid[]` vs `text[]` on `shared_with_user_ids` if prod differs). A lightweight GitHub Actions job running `supabase db reset` against a throwaway local Postgres would execute the migrations for real and turn "applies cleanly from empty" into a fast pre-merge signal, independent of the paid Supabase Preview.

- **Generate the base-schema migration from the `*-shapes.ts` builders instead of hand-maintaining both** — BE-1's `20260423_base_schema.sql` and the REST URL/body builders in `lib/supabase-profiles-shapes.ts` encode the same column set twice; they can drift. A small `scripts/check-schema-parity.ts` (the BE-3 direction) — or better, a codegen step that emits the column list from the body builders — would make the migration provably in sync with what the client sends, killing a whole class of "column does not exist" prod errors.

- **Add `created_at`/`updated_at timestamptz` + a `moddatetime` trigger to all four base tables (BE-9)** — the base schema gives `profiles`/`collections`/`items`/`friend_requests` a `created_at`, but no `updated_at`.
